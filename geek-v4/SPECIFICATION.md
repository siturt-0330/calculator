# 第1章: プロジェクト概要

## 1-1. GEEK とは何か

**GEEK（ギーク）** は、日本の「オタク文化」「推し活」「趣味コミュニティ」を中心とした **匿名型ソーシャルネットワーキングアプリ** である。

スマートフォン（iPhone・Android）で動作するモバイルアプリとして開発されており、同時にウェブブラウザからもアクセスできる。

### GEEKが解決しようとしている問題

現在の日本のSNS（X/Twitter、Instagram、TikTok など）には、以下のような問題がある：

1. **実名・アカウント紐付けのプレッシャー**  
   既存のSNSでは、フォロワーや知人に自分のアカウントが知られているため、「本当に言いたいこと」「マニアックな趣味の話」「恥ずかしい推し活の告白」ができない。匿名で発言できるスペースがない。

2. **炎上・誹謗中傷のリスク**  
   意見を発信すると、何万人もの見知らぬ人から攻撃される「炎上」が起きやすい。これが原因で、多くの人が発言を自己検閲してしまっている。

3. **タイムラインがノイズだらけ**  
   好きな話題だけ見たいのに、政治・芸能スキャンダル・ゴシップなど、見たくない情報が混ざってくる。興味のない話題をフィルタリングするのが難しい。

4. **公式情報との信頼性の混在**  
   アイドルのスケジュール、アニメの聖地、同人イベントの情報など、「確かな情報」と「噂話」が混在していて、どれを信じればいいか分からない。

5. **コミュニティが分散している**  
   同じ趣味を持つ人たちが、Discord・LINEオープンチャット・Twitterのスペース・pixivファンボックスなど、バラバラのプラットフォームに散らばっており、一か所にまとまっていない。

### GEEKが提供する解決策

| 問題 | GEEKの解決策 |
|------|-------------|
| 実名プレッシャー | すべての投稿が匿名表示。アカウント名はニックネームのみ |
| 炎上リスク | 「信頼スコア」システムで荒らし・誹謗中傷を自動抑制。投稿前にAI検査 |
| タイムラインノイズ | タグシステムで「好きな話題だけ」表示。不要な話題を完全ブロック |
| 情報信頼性 | 公式コミュニティ制度でアーティスト・企業が認証済み情報を発信 |
| コミュニティ分散 | タグ・コミュニティ機能で同好の士を一か所に集める |

## 1-2. バージョン履歴と v4 の位置づけ

GEEKはバージョンを重ねながら開発されてきた：

- **v1（初期プロトタイプ）**: 基本的な投稿・タグ機能
- **v2（コミュニティ機能追加）**: グループ・コミュニティの概念を導入
- **v3（UI刷新）**: デザイン全面リニューアル
- **v4（現行バージョン）**: 公式コミュニティ制度・信頼スコアシステム・AI機能・パフォーマンス大幅改善

v4 は単なるアップデートではなく、**プロダクトの方向性を根本から見直した** バージョンである。特に以下の3点が v4 の核心：

1. **「信頼の可視化」** — 信頼スコアシステムにより、発言の質と信頼性を数値で表現
2. **「公式の場」** — アーティスト・企業が公式アカウントとしてコミュニティを運営できる
3. **「ゼロからの最適化」** — 読み込み速度・バッテリー消費・データ通信量をすべて見直し

# 第2章: プロダクトビジョンと設計思想

## 2-1. コアバリュー（大切にしていること）

GEEKの開発において、すべての機能・設計判断は以下の4つの価値観を軸に行われている：

### 価値観1: 完全匿名性の保証

GEEKでは、投稿者の実名・本名・アカウント情報は**絶対に表示されない**。投稿カードに表示されるのは「匿名ユーザー」というラベルのみ。コメント欄では「1番目の人」「2番目の人」のように番号で区別される。

これにより：
- 人気者もそうでない人も、まったく対等に発言できる
- フォロワー数による「格差」が生まれない
- 本当に思っていることを遠慮なく書ける
- 推しへの愛を恥ずかしがらずに語れる

### 価値観2: 構造的安全性

「荒らし」「誹謗中傷」「デマ情報」などの有害コンテンツを、**ルール禁止だけでなく仕組みとして起きにくくする**。

具体的には：
- **信頼スコア**：スコアが低い人は投稿制限がかかる（低品質ユーザーを自然に抑制）
- **タグフィルター**：ゴシップ・スキャンダル系のタグは、ユーザー設定に関わらずトレンドに表示しない
- **気になるボタン**：いいねの逆として機能する「気になる（懸念）」機能で、問題のある投稿を可視化
- **AI検査**：投稿前に自動でコンテンツをスキャン（個人情報・誹謗中傷を検出）

### 価値観3: タグ中心の情報設計

GEEKでは「フォロー」ではなく「タグ」を中心に人とコンテンツが繋がる。

- 「このアカウントをフォロー」ではなく「このタグをフォロー」
- タグを好きに設定することで、自分だけのパーソナライズされたタイムラインができる
- ブロックしたいタグを設定することで、見たくないコンテンツを完全に除外できる

これにより、インフルエンサーによる「情報の独占」が起きない。コンテンツの質と関連性だけで注目が集まる、公平な情報空間を実現する。

### 価値観4: プライバシー尊重

- 広告表示はユーザーがオプトアウト可能
- 広告ターゲティングは「タグ情報のみ」を使い、個人IDは広告主に送らない
- ローカルでの学習（クリック履歴など）はデバイス内のみで処理し、サーバーに送信しない
- 設定で「懸念・気になる」をプライベートモードにできる

# 第3章: ユーザーアカウントと認証

## 3-1. アカウント作成（サインアップ）の全フロー

### ステップ1: メールアドレスとパスワードの入力

ユーザーがアプリを初めて起動すると、ログイン画面が表示される。「新規登録」をタップすると、アカウント作成画面に移動する。

**入力項目：**
- **メールアドレス**: `someone@example.com` の形式で入力
  - 形式チェック（`@` が含まれているか、ドメイン部分が正しいか）を即座に確認
  - 既に登録済みのメールの場合はログイン画面に自動移動（プリフィル済み）
- **パスワード**: 8文字以上、72文字以内
  - 目玉アイコンをタップすることで表示・非表示を切り替えられる
  - 72文字上限はセキュリティ上の理由（パスワードの暗号化アルゴリズムの仕様）

**裏側での処理：**
入力されたメールアドレスとパスワードは、Supabase（バックエンドサービス）の認証システムに送られる。最大12秒のタイムアウトが設定されており、それ以内に応答がなければエラーメッセージを表示（低速ネットワーク対応）。

### ステップ2: 電話番号の入力（任意）

メール・パスワード登録が完了した後、電話番号の入力画面が表示される。これは**任意入力**であり、スキップ可能。

**電話番号の仕様：**
- 10〜15桁の数字
- 使用可能な文字: 数字、`+`、`-`、`(`、`)`、スペース
- 日本の電話番号: `090-XXXX-XXXX` → 内部的に `+8190XXXXXXXX` 形式に変換

**なぜ電話番号を集めるのか：**
- アカウント回復のため（メールが使えなくなった場合）
- 将来的な多要素認証（2段階認証）への対応準備

### ステップ3: メール確認

登録したメールアドレスに確認メールが届く。メール内のリンクをクリックすると、アカウントが有効化される。

**メール確認に関する特記事項：**
- 確認メールが届かない場合、「再送信」ボタンから再送できる
- 迷惑メールフォルダに入っている可能性についてアプリ内で案内する
- メール確認前でもアプリは使用できる場合がある（設定による）

### ステップ4: 自動ログイン

アカウント作成が完了すると、多くの場合は自動的にログイン状態になり、オンボーディング（初期設定）画面に移動する。

自動ログインに失敗した場合は、3回まで自動再試行（間隔: 350ms → 700ms → 1050ms）を行う。それでも失敗した場合は、確認メール送信の旨を表示してログイン画面に誘導する。

## 3-2. ログインフロー

### 通常ログイン

1. メールアドレスを入力
2. パスワードを入力（目玉アイコンで表示切り替え可能）
3. 「ログイン」ボタンをタップ
4. 10秒以内に応答がなければタイムアウトエラー
5. 成功時: ホーム（フィード）画面に移動
6. オンボーディング未完了の場合: オンボーディング画面に移動

### エラーケースと対応

| エラーの種類 | ユーザーへの表示 | 対処法 |
|-------------|----------------|--------|
| メール・パスワードの不一致 | 琥珀色のバナー表示 | パスワードリセットリンクを提示 |
| メール未確認 | 確認メール送信を促すバナー | 「再送信」ボタンを表示 |
| 短時間での連続試行 | 「しばらく待ってから試してください」 | 1〜2分後に再試行 |
| ネットワークエラー | 「通信状況を確認してください」 | Wi-Fi/モバイルデータ確認を促す |
| アカウント停止（suspended） | 専用の停止メッセージ | カスタマーサポートへ誘導 |

**重要:** アカウント状態が「注意（caution）」「制限（restricted）」「警告（warned）」の場合でも、ログイン自体は可能。停止（suspended）の場合のみログインがブロックされる。

### アカウント状態システムの詳細

GEEKでは、ユーザーの行動に応じてアカウントに状態が付与される：

| 状態 | 説明 | ログイン可否 |
|------|------|------------|
| healthy（健全） | 通常状態 | ○ |
| caution（注意） | 懸念通報が若干多い | ○ |
| restricted（制限中） | 懸念通報/投稿比率が高い | ○（一部機能制限） |
| warned（警告） | 重大な違反の可能性 | ○（UI上に警告表示） |
| suspended（停止） | 規約違反が確定 | ✕ |

この状態は「懸念（気になる）」通報数と投稿数の比率によって自動計算される：
- 比率が0.5以上 → caution
- 比率が1.0以上 → restricted  
- 比率が1.5以上 → warned

（例: 10件投稿して7件懸念通報を受けた場合、比率0.7 → cautionに昇格）

## 3-3. パスワードリセットフロー

ユーザーがパスワードを忘れた場合のフロー：

1. ログイン画面の「パスワードを忘れた方」リンクをタップ
2. 登録済みのメールアドレスを入力
3. 「送信」ボタンをタップ
4. 登録済みでも未登録でも同じメッセージを表示（セキュリティ上、登録有無を教えない）
5. 登録済みの場合、リセット用リンクが記載されたメールが届く
6. リンクの有効期限は**1時間**
7. リンクをタップすると新しいパスワード入力画面に移動
8. 新パスワードを設定（8〜72文字）
9. 設定完了後、自動的にログアウト状態になり、再ログインを促す

**対応するリンク形式（技術的詳細）:**
- PKCE フロー: URL に `?code=xxxxx` が含まれる
- Implicit フロー: URLハッシュに `#access_token=...` が含まれる
- Legacy フロー: URL に `?token=...&type=recovery` が含まれる

これら3つの方式すべてに対応しており、どの環境（Web・iOS・Android）から送られたリンクでも正しく動作する。

---

## 3-4. アカウント削除

設定 → データとアカウント → アカウントを削除 から実行可能。

**削除前の確認事項：**
- 削除は取り消しができない（原則）
- 投稿・コメント・保存したコンテンツはすべて削除される
- ニックネーム・プロフィール情報も削除される

**猶予期間（グレースピリオド）：**
アカウント削除には「猶予期間」が設けられており、即座に消えるわけではない。一定期間内（詳細は運用設定による）であれば復元できる可能性がある。これは誤操作・後悔による削除リクエストへの対応。

## 3-5. セッション管理

### セッションの永続化

GEEKにログインした後、アプリを閉じても次回起動時には自動的にログイン状態が維持される。これは「セッション情報」をデバイスに安全に保存しているため。

保存場所：
- **スマートフォン（iOS/Android）**: SecureStore（暗号化されたローカルストレージ）
- **ウェブブラウザ**: localStorage（ブラウザのローカルストレージ）

### 複数タブでの同期（Webブラウザの場合）

ウェブブラウザでGEEKを複数タブで開いている場合、あるタブでログアウトすると他のタブも自動的にログアウト状態になる（storage イベントを監視）。

### ログアウト処理

ログアウトすると以下が行われる：
1. サーバー側のセッションを無効化
2. デバイスに保存されているセッション情報を削除（`geek-v4-auth` キーを削除）
3. アプリ内のすべてのリアルタイム接続（チャンネル）を強制切断
4. ログイン画面に移動

「二重ログアウト」（同時に2回ログアウト処理が走る）を防ぐロック機構が実装されている。

## 4-4. 画面3: ニックネーム設定

アプリ内で表示されるニックネームを設定する。

**ルール：**
- 2〜20文字
- 後から変更可能（設定 → プロフィール編集）

**注意：** ニックネームは他のユーザーに表示されるが、投稿自体は匿名なので、ニックネームが投稿に紐付いて表示されることはない。マイページとプロフィールページでのみ表示される。

## 4-5. 画面4: 好きなタグの選択

自分の興味・趣味に関するタグを選択する。これにより、フィード（タイムライン）が好みに合わせてカスタマイズされる。

**タグの例（カテゴリ別）：**
- アイドル: 坂道シリーズ、ハロプロ、48グループ
- VTuber: ホロライブ、にじさんじ、ぶいすぽ、個人勢
- アニメ: 少年漫画原作、少女漫画原作、ラノベ原作、オリジナル
- ゲーム: RPG、FPS、パズル、シミュレーション、ソシャゲ

**操作方法：**
- タグをタップ: 好きなタグとして追加（ハイライト表示）
- もう一度タップ: 選択解除
- 検索ボックス: タグを名前で検索
- スキップ可能（後から設定できる）

## 4-6. 画面5: ブロックタグの設定

見たくないコンテンツのタグを設定する。設定したタグが付いた投稿は、フィードに表示されなくなる。

**デフォルトでブロックされるタグ（安全のために最初から設定済み）：**
- 詐欺、暴力、虐待、わいせつ、ハラスメント、デマ情報、グロテスク

これらは削除することも可能。また、追加でブロックしたいタグがあれば検索して追加できる。

**ゴシップ・スキャンダル系タグについて：**
不倫、スキャンダル、事件報道などのゴシップ系タグは、たとえユーザーが設定していなくても「トレンド」には表示されない。これはGEEKのプロダクトポリシーによる措置（「人が不幸になる話題」は積極的に広めない）。

## 4-7. 画面6: 通知設定

プッシュ通知の設定を行う。

**通知の種類：**

| 通知の種類 | 説明 | デフォルト |
|-----------|------|-----------|
| いいね通知 | 投稿にいいねがついた時 | オン |
| コメント通知 | 投稿にコメントがついた時 | オン |
| フォロー通知 | 新しいフォロワーができた時 | オン |
| 返信通知 | コメントに返信がついた時 | オン |
| イベント通知 | 登録したイベントのリマインダー | オン |
| タグ新着通知 | 好きなタグの新着投稿 | オフ |
| おしらせ | GEEKからの公式通知 | オン |

**おやすみ時間設定：**
「おやすみ時間」を設定すると、その時間帯はプッシュ通知が届かなくなる。

例: 22:00〜07:00 をおやすみ時間に設定 → 夜10時〜翌朝7時の通知をすべてミュート

設定方法:
- 開始時刻（例: 22）と終了時刻（例: 7）を数値で入力
- 日をまたぐ設定も対応（22:00〜07:00 など）

## 4-8. オンボーディング完了

すべての設定が完了すると、`onboarded = true` フラグがサーバーに保存される。以後、アプリを起動するたびにオンボーディング画面が表示されることはなくなる。

設定はすべて後から変更可能：
- ニックネーム → 設定 → プロフィール編集
- 言語 → 設定 → 言語
- タグ → 設定 → フィルター
- 通知 → 設定 → 通知

# 第6章: フィード（ホーム画面）

## 6-1. フィードとは

フィードはGEEKのメイン画面であり、投稿（テキスト・画像）が時系列または関連度順に表示される「タイムライン」のこと。Twitterのホームタイムラインに相当するが、GEEKでは「フォロー」ではなく「タグ」をベースにパーソナライズされる点が大きく異なる。

## 6-2. フィードヘッダー

画面上部に以下の要素が並ぶ：

- **投稿作成ボタン（+アイコン）**: タップで投稿作成画面へ。アクセントカラーのグロー（発光）エフェクト付き
- **GEEKロゴ**: Orbitron Blackフォントで「Geek」と表示。グラデーションテキストとグロー効果
- **検索ボタン**: タップで検索画面へ
- **通知ベル**: 未読通知がある場合は赤いバッジ（件数）が表示される

## 6-3. フィードの表示モード（スコープ）

フィードには2つの表示モードがある：

### All（全体）モード
すべてのタグの投稿を表示する。ただしブロックしたタグの投稿は除外される。デフォルトはこのモード。

### For Me（自分向け）モード
好きなタグとして登録したタグの投稿のみを表示する。「好きなタグ」を1つも設定していない場合は選択できない（グレーアウト表示）。

切り替えは画面上部のトグルボタンで行う。設定は端末に記憶され、次回起動時も維持される。

## 6-4. フィードのソート順

フィードの並び順を選択できる：

| ソート名 | 説明 | 特徴 |
|---------|------|------|
| おすすめ（For You） | AIによるパーソナライズ | クリック履歴・タグ親和性・信頼スコアを総合判断 |
| 急上昇（Hot） | 短時間でいいねが多い投稿 | バズっている投稿を発見したい時 |
| 新着（New） | 投稿時刻の新しい順 | リアルタイムの動きを追いたい時 |
| 評価順（Top） | 総いいね数の多い順 | 長期的に人気の高い投稿 |

## 6-5. トレンドタグ行

フィードの一番上（スクロール前の位置）に「今トレンドのタグ」が横スクロールで表示される。これらのタグをタップすると、そのタグの投稿一覧に移動する。

**トレンドタグの選定ルール：**
- 直近の投稿数が多いタグが上位に表示される
- ゴシップ・スキャンダル・事件報道系のタグは、いかに投稿が多くても**絶対にトレンドに表示しない**（プロダクトポリシー）
- ユーザーがブロック設定しているタグも表示しない

## 6-6. ブロックタグバナー

ブロックしているタグの数が1件以上ある場合、フィードの上部に「○個のタグを非表示にしています」というバナーが表示される。タップするとフィルター設定画面に移動し、ブロックタグの管理ができる。

## 6-7. 広告の挿入

フィードには8件の投稿ごとに1件の広告が挿入される。広告には「広告」という明示的なラベルが表示される。

**プライバシー保護の広告設計：**
- 広告ターゲティングは「ユーザーのタグ」情報のみを使用
- 個人IDは広告主に送信しない
- 設定でパーソナライズ広告をオフにできる（オフにすると広告自体が表示されなくなる）

## 6-8. 投稿カード（AnonPostCard）の詳細

フィードに表示される1件の投稿は「投稿カード」として表示される。

**投稿カードに表示される情報：**

1. **投稿本文**: テキスト（最大2000文字）
2. **画像**: 最大4枚（スワイプで閲覧）
3. **タグ**: 投稿者が付けたタグ（最大5個）。オレンジ色のタグは他のユーザーが後から追加したタグ
4. **投稿種別バッジ**: fact（事実）/ opinion（意見）/ joke（ジョーク）/ wip（制作中）
5. **投稿時刻**: 「3分前」「昨日」などの相対表示
6. **コンテンツ警告**: spoiler（ネタバレ）/ nsfw / violence / sensitive の場合はラベル表示

**インタラクションボタン：**

| ボタン | アイコン | 機能 |
|--------|---------|------|
| いいね | ハートアイコン | 投稿に「いいね」を付ける。ハートアニメーション付き |
| 気になる | 特殊アイコン | 投稿に懸念を示す（信頼スコアに影響） |
| コメント | 吹き出しアイコン | コメント欄へ移動 |
| 保存 | ブックマークアイコン | 後で見返せるよう保存 |
| シェア | 共有アイコン | 投稿のリンクをシェア |
| もっと | 三点メニュー | 通報・ブロック・タグ追加などのメニュー |

**リアクション機能：**
投稿に対してメメ絵文字（スタンプ）でリアクションできる。ハートやいいねとは別に、絵文字スタンプを使った感情表現が可能。

**タグ追加機能：**
投稿に付いていないタグを、他のユーザーが追加できる「タグ追加」機能がある。追加されたタグはオレンジ色で表示される。これにより、投稿の分類・発見可能性が向上する。

## 6-9. フィードのパーソナライズ仕組み（For Youモード）

「おすすめ（For You）」ソートでは、以下の複数の要因を組み合わせて投稿の表示順を決定している：

1. **タグ親和性スコア**: ユーザーが過去にいいね・保存・閲覧した投稿のタグから、タグへの関心度を計算
2. **信頼スコアフィルター**: 信頼スコアの低いユーザーの投稿は優先度を下げる
3. **トレンドボーナス**: 急上昇中の投稿に加点
4. **CTRブースト**: 過去に同様のタグの投稿をクリックした履歴があれば加点
5. **時間減衰**: 古い投稿ほど優先度が下がる（指数的減衰）
6. **ブロックタグ除外**: ブロックタグの付いた投稿は完全に除外

これらの計算はデバイス上で行われ、サーバーに個人の行動履歴を送信しない（プライバシー保護）。

## 6-10. リアルタイム更新

フィードはリアルタイムで更新される。バックグラウンドで新しい投稿が追加された場合、フィードの先頭にバナー「新しい投稿があります」が表示され、タップすると最新の状態に更新される。

また、表示中の投稿のいいね数なども定期的に（最大30件まで）バックグラウンドで更新される。

## 6-11. 無限スクロール（ページネーション）

フィードは一度に全件を取得するのではなく、スクロールして下部に近づくたびに追加の投稿を取得する「無限スクロール」方式を採用している。

- 1回の取得: 最大40件
- カーソルベースのページネーション（いいね数・投稿時刻の複合カーソル）
- 取得失敗時は「もっと読み込む」ボタンを表示
- 最後まで読み込んだ場合は「すべて表示しました」を表示

## 6-12. オフライン対応

ネットワーク接続がない場合でも、キャッシュされた投稿を表示できる（読み取り専用）。画面上部に「オフライン」バナーが表示される。

オフライン中にいいね・コメント・保存などのアクションを行うと、「オフラインキュー」に保存され、ネットワーク回復時に自動的に送信される。

# 第7章: 匿名掲示板（BBS）

## 7-1. BBSとは

BBS（Bulletin Board System）は、テーマ別のスレッドに返信を投稿できる「匿名掲示板」機能。2ちゃんねる（現5ちゃんねる）のような「スレッド＋返信」形式で、時系列に流れてしまうフィードと違い、テーマが固定されたディスカッションができる。

**フィードとBBSの違い：**

| 特徴 | フィード | BBS |
|------|---------|-----|
| 投稿形式 | 独立した投稿 | スレッド＋返信 |
| 時系列 | 新着が上に流れる | スレッド内で積み上がる |
| 用途 | 近況・感想を発信 | テーマを決めたディスカッション |
| 永続性 | 流れやすい | テーマが保たれる |

## 7-2. BBS画面の構成

**ヘッダー：**
- 「掲示板」タイトル
- スレッド作成ボタン（+アイコン、アクセントグロー付き）

**検索バー：**
- キーワード検索（リアルタイム、入力停止後100〜150ms後に反映）
- クリアボタン（入力がある場合のみ表示）

**カテゴリタブ（横スクロール）：**
12のカテゴリから選択してフィルタリングできる：
```
すべて / 雑談 / アニメ / ゲーム / マンガ / 音楽 / アイドル / 
Vtuber / 推し活 / グルメ / コスプレ / ニュース
```
各カテゴリに固有の色が設定されており、スレッドカードの左側にカラーバーで色分け表示される。

**ソートモード：**
- 🕐 新着: 最新返信のあったスレッドが上位
- 🔥 人気: 返信数の多いスレッドが上位
- 🎯 関連度: 検索キーワードとの関連度が高いスレッドが上位（検索時のみ有効）

**検索結果件数表示：**
「1,234件のスレッド」のように件数を表示（日本語の桁区切り形式）。

## 7-3. スレッドカードの構成

1件のスレッドは以下の情報で表示される：

- **左カラーバー**: カテゴリの色
- **コミュニティバッジ**: コミュニティに紐付いたスレッドの場合、アイコン絵文字とコミュニティ名を表示
- **カテゴリバッジ**: カテゴリ名（背景色付き）
- **タイムスタンプ**: 「3分前」「2日前」などの相対表示
- **タイトル**: スレッドのタイトル（検索中はマッチ部分がハイライト表示）
- **返信数**: 💬 と数字で表示
- **賑わい中バッジ**: 返信数が10件を超えたスレッドに 🔥 マーク（橙色）を表示

## 7-4. 検索の仕組み（詳細）

BBS検索はクライアント側（端末内）で行われる高速なリアルタイム検索。

**検索処理の流れ：**

1. **クエリ解析（parseQuery）**
   入力されたクエリを分解する：
   - `keywords`: 通常のキーワード（「アニメ おすすめ」→ `['アニメ', 'おすすめ']`）
   - `phrases`: ダブルクォートで囲んだフレーズ（`"魔法少女まどか"` → フレーズとして一致）
   - `excludes`: マイナス付きのキーワード（`-スポイラー` → スポイラーを除外）

2. **表記揺れ対応（generateVariants）**
   各キーワードについて複数の表記バリエーションを生成：
   - 半角カタカナ ↔ 全角カタカナ
   - ひらがな ↔ カタカナ
   - 旧字体・新字体
   例：「アイドル」→「アイドル・IDOL・あいどる」なども検索対象に含める

3. **正規化（deepNormalize）**
   スレッドタイトルとカテゴリを正規化してから比較。これにより大文字小文字、全角半角の違いを無視した検索が可能。

4. **スコアリング（関連度ソート時）**
   各スレッドに関連度スコアを計算：
   - テキスト一致度（BM25ライクなアルゴリズム）
   - カテゴリ一致: +30点ボーナス
   - 返信数の対数 × 3
   - 時間減衰（新しいほど高スコア）
   - CTRブースト（過去に同じ検索クエリでこのスレッドを開いた場合: 最大+100点）

5. **タイポ補正（findClosestK）**
   0件ヒット時に「もしかして：○○」を最大3件提示。レーベンシュタイン距離（文字の編集距離）で近いキーワードを提案。

## 7-5. スレッド作成

「スレッドを作成」画面の入力項目：

- **タイトル**: 必須、最大50文字
- **カテゴリ**: 任意、プリセット12個のチップから選択 or テキスト入力
- **公開設定**: 2択
  - 🌐 一般公開: 全ユーザーが閲覧可能、掲示板リストに表示される
  - 🔒 コミュニティ限定: 指定コミュニティのメンバーのみ閲覧可能
- **コミュニティ紐付け**: 任意（一般公開の場合）/ 必須（コミュニティ限定の場合）
  - 検索で自分が参加しているコミュニティを選択できる

**投稿制限：**
BBSへの投稿には信頼スコア10以上が必要。スコアが不足している場合はエラーメッセージが表示される。

## 7-6. スレッド詳細・返信

スレッドをタップすると詳細画面に移動し、すべての返信が表示される。

**返信カードの要素：**
- **左アバター**: ランダムな色（投稿者ごとに一意の色）
- **番号バッジ**: #1, #2, #3... （スレッド内での返信番号）
- **信頼バッジ**: 投稿者の信頼スコア段階を示すバッジ
- **タイムスタンプ**: 投稿時刻
- **本文**: 返信テキスト
- **引用ボタン（>>N）**: その返信への引用が可能
- **リアクション**: メメ絵文字でのリアクション

**引用返信（>>N）：**
返信の「>>N」ボタンをタップすると、入力欄の先頭に `>>3` のような引用テキストが自動挿入される。これにより特定の返信へのレスポンスであることが明確になる。

**@メンション：**
`@` を入力すると、スレッド内の返信番号（#1, #2...）のオートコンプリートが表示される。

**リアルタイム更新：**
スレッドが表示されている間、新しい返信が投稿されると自動的に画面に追加される。

**返信の投稿：**
- 最大500文字
- 信頼スコア10以上が必要
- 投稿ボタンはテキスト入力時のみ有効（空欄は送信不可）

# 第8章: コミュニティ機能

## 8-1. コミュニティとは

コミュニティは、特定のテーマ・趣味・推しに関するユーザーが集まる「グループ」機能。Discordのサーバーに近い概念で、以下の機能を内包する：

- メンバー間の投稿フィード
- 専用の掲示板スレッド
- 聖地（スポット）マップ
- イベントカレンダー
- Q&A（公式コミュニティのみ）

## 8-2. コミュニティの種類

### 一般コミュニティ
ユーザーが自由に作成できるコミュニティ。タグベースで分類される。

**公開設定の3種類：**

| 種類 | アイコン | 説明 |
|------|---------|------|
| オープン | 🌐 | 誰でも参加可能。検索結果に表示される |
| 許可制（クローズ） | 🔒 | 参加申請を送り、管理者が承認した場合のみ参加できる |
| 完全招待制 | 🛡️ | 検索結果にも表示されない。招待リンク経由のみ参加可能 |

### 公式コミュニティ
GEEKが審査・認証したコミュニティ。アーティスト事務所・企業・公共機関などが運営する。公式バッジ（青いチェックマーク）が表示される。

**公式コミュニティの追加機能：**
- Q&A機能（AIが管理者が登録したナレッジを参照して回答）
- 管理者が特定のユーザーに匿名を剥奪して返信できる機能
- 詳細な分析・インサイト（メンバー数推移、投稿エンゲージメントなど）

## 8-3. コミュニティ探索画面

「探す」画面では全コミュニティを検索・閲覧できる。

**UI要素：**
- 検索入力フィールド（入力後debounce 100〜150ms）
- 「公式のみ」フィルタチップ
- 公式コミュニティセクション（横スクロール、クエリなし時のみ表示）
- コミュニティ一覧（スクロールリスト）

**コミュニティカードの表示項目：**
- 52×52pxのアイコン（画像 or 絵文字）
- コミュニティ名
- 説明文（2行まで表示、それ以降は省略）
- メンバー数・投稿数（数値ピル）
- 公式バッジ（認証済みの場合）
- 鍵アイコン（許可制の場合）

## 8-4. コミュニティ作成

**入力項目：**
- **アイコン画像**: 任意（96×96px、円形クロップ、最大5MB）
  - EXIF情報を自動削除（プライバシー保護）
  - magic byte検証（実際の画像ファイルかチェック）
  - JPEG形式にリエンコード
- **名前**: 必須、最大40文字
  - 入力中にリアルタイムで類似コミュニティを検索・警告表示
  - 表記揺れ（ひらがな/カタカナ）を正規化して比較
- **説明**: 任意、最大500文字
- **タグ**: 最大10個（コミュニティの分類に使用）
  - タグ入力時に既存タグのオートコンプリートが表示される
  - 「新しいタグを作る」オプションもあり
- **公開設定**: 3択（オープン / 許可制 / 完全招待制）

**作成処理の流れ（2ステップ）：**
1. コミュニティの基本情報を保存（データベースに行を作成）
2. アイコン画像をアップロード（ストレージに保存 → URLを取得）
3. コミュニティ情報にアイコンURLを紐付けて更新

## 8-5. コミュニティ詳細画面

コミュニティをタップすると詳細画面（ダッシュボード）に移動する。

**ヘッダー部分：**
- 96×96px アバター（枠線: 2px アクセントカラー、グロー効果）
- コミュニティ名 + @handle（スラッグ）
- タグのピル表示
- 統計（メンバー数・投稿数）
- 説明文（80文字以上は「もっと見る」ボタンで展開）
- 参加ボタン / 参加申請ボタン / 参加中ボタン

**参加ボタンの状態：**
| 状態 | 表示 | 操作 |
|------|------|------|
| 未参加・オープン | 「コミュニティに参加する」（アクセント色） | タップで即時参加 |
| 未参加・許可制 | 「参加を申請する」（アクセント色） | タップで申請送信 |
| 参加中 | 「参加中」（枠線のみ、ドロップダウン） | タップで脱退メニュー |
| オーナー | 管理メニュー表示 | コミュニティ設定 |

**参加時の動作（楽観的更新）：**
参加ボタンをタップすると、サーバーの応答を待たずに即座に「参加中」表示に切り替わる（楽観的更新）。もし後からエラーが発生した場合は元の状態に戻す。

## 8-6. コミュニティのタブ構成

コミュニティ詳細画面には複数のタブがあり、横スワイプまたはタブタップで切り替える。

### 一般コミュニティのタブ：
1. **ホーム**: コミュニティの最新投稿フィード
2. **掲示板**: コミュニティ専用のBBSスレッド
3. **聖地**: コミュニティに関連する場所のマップ
4. **カレンダー**: コミュニティのイベント一覧
5. **投稿**: （このタブをタップすると投稿作成画面に移動）

### 公式コミュニティのタブ：
1. **ホーム**: 最新情報フィード
2. **Q&A**: AIが答えるQ&A機能
3. **聖地**: 関連スポットマップ
4. **カレンダー**: 公式イベント一覧
5. **コメント**: ファンからのコメント

**タブの遅延読み込み（Lazy Loading）：**
各タブは初めて開いた時にのみデータを取得する。その後は`display: none`でDOMに保持（Keep-Alive）。これにより、タブを再度開いた時に再読み込みが不要で、高速な切り替えが可能。

## 8-7. 聖地（スポット）機能

コミュニティに関連する実際の場所（聖地・撮影スポット・ライブ会場など）を地図上に登録・表示する機能。

**表示情報：**
- 地図上のピン（マーカー）
- スポット名
- 住所・説明
- 認証バッジ（管理者が認証した正確な情報）

**スポットの追加：**
一般メンバーも候補地を投稿でき、管理者が審査・認証できる。

## 8-8. カレンダー・イベント機能

コミュニティ内のイベントを管理する機能。

**イベントの表示：**
- 月別グループ化（「2026年5月」「2026年6月」のように区切り）
- 各イベントカードの表示項目：
  - 日付（左側に大きく表示: 日・曜日）
  - 時刻バッジ
  - タイトル
  - 場所（アイコン付き）
  - 説明文（タップで展開）

**イベント追加（公式コミュニティ管理者のみ）：**
右下のFABボタンからイベントを追加。
- タイトル（必須）
- 開始日時（必須、`YYYY-MM-DDTHH:MM`形式）
- 終了日時（開始より後である必要あり）
- 場所
- URL
- 説明

## 8-9. Q&A機能（公式コミュニティ限定）

公式コミュニティでは、AI（人工知能）を活用したQ&A機能が使用できる。

**仕組み：**
1. 管理者が「ナレッジ」（知識ベース）を登録する（例: よくある質問と回答、ルール説明、アーティストのプロフィールなど）
2. ユーザーが質問を入力する
3. AIがナレッジを参照して回答を生成する（RAG: Retrieval-Augmented Generation）
4. 回答と共に参照したナレッジのソース（出典）が表示される

**ナレッジ管理（管理者のみ）：**
- タイトル（最大200文字）
- 内容（最大50,000文字）
- 一覧表示、編集、削除が可能
- ナレッジがない場合は「⚠️ 管理者にナレッジ追加を依頼してください」と表示

**Q&A表示：**
- 最新の回答: 大きく表示（アニメーション入場）
- 過去の質問履歴: 質問テキスト + 回答プレビュー + タイムスタンプ

# 第9章: 投稿作成機能

## 9-1. 投稿作成画面の概要

投稿作成はGEEKの中核機能。フィードのヘッダーにある「+」ボタン、またはタブバー中央の投稿ボタンからアクセスできる。

## 9-2. 投稿の基本要素

### テキスト（本文）
- 最大2,000文字
- 画像がある場合はテキストなしでも投稿可能
- 下書きが自動保存される（入力停止後500ms）
- 画面を離れて戻ると下書きが復元される（「下書きを復元しますか？」トースト表示）

### 画像
- 最大4枚
- 対応形式: JPEG, PNG等の主要フォーマット
- 最大サイズ: 5MB（超過時は自動圧縮）
- EXIF情報（撮影場所・撮影日時など）は自動削除（プライバシー保護）
- 円形クロップツール（アバター用）も内蔵

### タグ（1〜5個）
- 1投稿に最大5個のタグを設定可能
- タグは投稿の分類・検索・フィードパーソナライズに使用される
- 入力中にオートコンプリートが表示される（既存タグから提案）
- AIによる自動タグ提案機能（入力した本文・画像から自動的にタグを提案）

### 投稿種別（任意）
投稿の性質を示すラベルを設定できる：

| 種別 | アイコン | 意味 |
|------|---------|------|
| fact（事実） | - | 出典のある客観的事実。出典URLの記載が推奨（必須ではない） |
| opinion（意見） | - | 個人の意見・感想 |
| joke（ジョーク） | - | ジョーク・ユーモア系の投稿 |
| wip（制作中） | - | 制作途中の作品の共有（Work In Progress） |

### 出典URL（任意、factの場合は推奨）
- 参考にしたページのURLを記載
- linkPreview機能により、URLのOGP情報（タイトル・サムネイル）を自動取得して表示

### コンテンツ警告（CW）
特定のコンテンツを含む場合に設定できる警告。設定するとカードに警告ラベルが表示され、ユーザーが意図的にタップしないと内容が表示されない。

| 種別 | 説明 |
|------|------|
| spoiler | ネタバレを含む |
| nsfw | 成人向けコンテンツ |
| violence | 暴力的表現を含む |
| sensitive | センシティブな話題 |

CWラベルに加え、詳細説明テキスト（最大120文字）も入力できる。

## 9-3. 公開範囲の設定

投稿を誰が見られるかを設定する。4種類から選択：

| 設定 | アイコン | 誰が見られるか |
|------|---------|--------------|
| 自分だけ | 🔒 | 自分のみ（下書き的な使い方） |
| 一般公開 | 🌐 | GEEKの全ユーザー |
| コミュニティメンバーのみ | 👥 | 指定したコミュニティのメンバーのみ |
| コミュニティ公開 | 📣 | 全員が見られるが、コミュニティにも掲載される |

「コミュニティメンバーのみ」または「コミュニティ公開」を選択した場合、コミュニティ選択画面が表示される。自分が参加しているコミュニティから1つ以上を選択する必要がある。

## 9-4. 投票（ポール）機能

投稿に投票（アンケート）を追加できる。

**設定項目：**
- 質問文
- 選択肢: 最大6個（追加・削除可能）
- 複数選択: 許可するかどうかをトグルで設定
- 期間: 6時間 / 24時間 / 72時間 / 168時間（1週間）から選択

投票の結果は期間終了後に確定。投票中でも現在の集計状況が確認できる。

## 9-5. 匿名設定

デフォルトでは投稿は匿名。トグルで「実名（ニックネーム）表示」に切り替えることも可能。

## 9-6. 投稿前のAI検査

「投稿」ボタンをタップすると、送信前に以下の検査が行われる：

1. **文字数チェック**: 本文が2,000字以内か
2. **タグ数チェック**: 1〜5個のタグが設定されているか
3. **信頼スコアチェック**: 投稿に必要な最低スコア（20以上）を満たしているか
4. **AIコンテンツ検査（check-content）**: 個人情報（住所・電話番号・本名など）が含まれていないかをチェック

すべてのチェックを通過した場合のみ投稿が実行される。

## 9-7. 下書き自動保存

投稿作成中の内容は自動的にデバイスに保存される：
- 入力停止後500ms（0.5秒）で自動保存
- アプリを閉じたり別の画面に移動しても保存される
- 次回投稿作成画面を開くと、保存された下書きを復元するかどうかのトースト通知が表示される
- 「元に戻す」ボタンで復元をキャンセルできる（5秒以内）

# 第11章: マイページ（プロフィール画面）

## 11-1. マイページの概要

マイページは「自分のプロフィール」と「各種設定へのハブ」を兼ねた画面。

## 11-2. ヘッダーエリア

- **プロフィール編集ボタン**: 設定 → プロフィール編集 へのショートカット

## 11-4. KPIカード（統計情報）

| 項目 | 説明 |

参加しているコミュニティを横スクロールで表示。

- 各コミュニティの56×56pxアイコン（画像 or 絵文字）
- コミュニティ名
- 末尾に「+ 探す」チップ

**参加していない場合：**
「まだ参加していません 🎯」と「コミュニティを探す」ボタンを表示

## 11-7. アクティビティセクション

- **自分の投稿**: 自分が投稿した投稿一覧
- **いいねした投稿**: いいねした投稿一覧
- **保存した投稿**: 保存（ブックマーク）した投稿一覧
- **運営からのメッセージ**: GEEK運営からの管理メッセージ（未読がある場合は赤いバッジ）
- **Geek Official**: 管理している公式コミュニティへのリンク（ある場合のみ表示）

## 11-8. アカウントセクション

- **信用スコア**: 現在の信頼スコアとティア（絵文字・色付き）
- **プラン**: Free / Pro の現在のプラン
- **通知**: 通知設定へのリンク（未読バッジ付き）

- **ブロックしたタグ**: タグブロック設定
- **ブロックしたユーザー**: ユーザーブロック設定  
- **すべての設定**: 設定画面へ

## 11-10. 信頼スコアとティア

信頼スコアは0〜100の数値で、以下のティア（段階）に分類される：

| ティア | スコア範囲 | 絵文字 | 色 |
|--------|----------|--------|-----|
| Newcomer（新規） | 0〜19 | 🌱 | グレー |
| Trusted（信頼） | 20〜49 | ✅ | 緑 |
| Verified（検証済み） | 50〜79 | 🏅 | 琥珀 |
| Elite（エリート） | 80〜100 | 🏆 | 金 |

**スコアが低いと何が制限されるか：**
- スコア10未満: BBSへの投稿不可
- スコア20未満: フィードへの投稿不可

これにより、アカウントを作りたてのユーザーが大量スパムを行うことを防ぐ。

# 第12章: 通知システム

## 12-1. 通知の種類

GEEKでは以下7種類の通知が発生する：

| 種別 | 発生タイミング | 表示例 |
|------|--------------|--------|
| like | 自分の投稿にいいねが付いた | 「あなたの投稿にいいねが付きました」 |
| comment | 自分の投稿にコメントが付いた | 「あなたの投稿にコメントが付きました」 |
| reply | 自分のコメント・スレッドに返信が付いた | 「あなたへの返信があります」 |
| follow | 新しいフォロワーができた | 「新しいフォロワーがいます」 |
| event | 登録イベントのリマインダー | 「イベントが30分後に始まります」 |
| official_post | 参加中の公式コミュニティの新規投稿 | 「○○公式から新着投稿」 |
| announcement | GEEK運営からのお知らせ | 「【お知らせ】メンテナンスのご案内」 |

## 12-2. 通知一覧画面

マイページの通知ベル、または設定の通知から開く。

**表示形式：**
- 最新50件を表示
- 未読は背景がハイライト
- タップで詳細（該当投稿・スレッド・コミュニティ）へジャンプ
- 既読化は自動（画面を開いた時点）

## 12-3. プッシュ通知（Web Push）

ブラウザ版GEEKではWeb Push通知に対応。設定からプッシュ通知を有効化できる。

**技術的な仕組み：**
- VAPID鍵による安全なPush配信
- Supabase Edge Function（send-push）が配信処理を実行
- 失効した端末は自動的に登録解除（410/404レスポンス時）
- 通知タイプごとに絵文字付きタイトル（💛いいね、💬コメント、📣公式など）

## 12-4. おやすみ時間

設定で「おやすみ時間」を指定すると、その時間帯はプッシュ通知が届かない。日をまたぐ設定（22:00〜07:00）にも対応。

## 14-1. 検索の対象

- スレッド（BBS）
- タグ

## 14-2. 検索の特徴

### 表記揺れ対応
- 半角カタカナ ↔ 全角カタカナ
- ひらがな ↔ カタカナ
- 大文字 ↔ 小文字
- 旧字体 ↔ 新字体

### タイポ（誤字）補正
0件ヒット時に「もしかして：○○」を最大3件提示。レーベンシュタイン距離アルゴリズムで類似ワードを提案。

### 検索履歴
過去の検索クエリを最大50件まで保存。最近1時間以内の検索は優先表示。

- `スペース` で複数キーワード（AND検索）
- `"ダブルクォート"` でフレーズ一致
- `-キーワード` で除外検索

例: `アニメ おすすめ -スポイラー` → 「アニメ」「おすすめ」を含み「スポイラー」を含まない結果

## 15-1. タグの役割

| 関係 | 意味 |

| related（関連） | 関連するが別概念 |
| group_member（同グループ） | 同じカテゴリ |

## 15-5. ロジックツリー

- TTL（有効期限）: 1時間
- 計算対象: 最新500投稿

# 第16章: 信頼スコアシステム

## 16-1. 信頼スコアの計算式

信頼スコアは以下の要素で計算される（最大100点）：

| 要素 | 配点 | 計算式 |
|------|------|--------|
| アカウント年齢 | 20点 | 経過日数 ÷ 30（最大20） |
| 投稿数 | 30点 | 投稿数 ÷ 10（最大30） |
| もらったいいね | 30点 | いいね数 ÷ 20（最大30） |
| コメント数 | 20点 | コメント数 ÷ 20（最大20） |
| 合計 | 100点 | - |

例: アカウント90日、投稿20件、もらったいいね100件、コメント30件
- アカウント年齢: 90/30 = 3点
- 投稿数: 20/10 = 2点
- もらったいいね: 100/20 = 5点
- コメント数: 30/20 = 1.5点
- 合計: 11.5点 → Newcomerティア

## 16-2. 信頼スコアの影響

| スコア | 制限事項 |
|--------|---------|
| 10未満 | BBS投稿不可 |
| 20未満 | フィード投稿不可 |
| 50以上 | 一部の高度な機能解放 |
| 80以上 | エリート機能解放（将来予定） |

## 16-3. 「気になる」機能と懸念カウント

通常のいいねの逆として「気になる（懸念）」ボタンがある。懸念の理由は以下から選択：

| 理由 | 意味 |
|------|------|
| misinfo | デマ・誤情報 |
| unverified | 未検証・要確認 |
| spam | スパム |
| rude | 無礼・攻撃的 |
| scam | 詐欺の疑い |
| other | その他 |

懸念カウントが多くなると、投稿者のアカウント状態が自動的に悪化（caution → restricted → warned）する。これにより、低品質な投稿者を自然に抑制する。

# 第18章: セキュリティとプライバシー

## 18-1. 認証セキュリティ

- パスワードは8〜72文字（bcrypt上限を尊重）
- PKCEフロー対応（モバイル向け推奨方式）
- Supabase Authによる業界標準の認証管理
- セッションはSecureStore（モバイル）/ localStorage（Web）に暗号化保存
- パスワードリセットリンクは1時間で失効

## 18-2. データ保護

- すべてのDB通信はHTTPS
- RLSによりユーザーは自分のデータ以外にアクセスできない
- 画像アップロード時にEXIF情報を自動削除
- 画像のmagic byte検証で不正ファイルを拒否
- アップロード上限: 5MB

## 18-3. コンテンツ検査

投稿前にAIで以下をチェック：
- 個人情報（電話番号・住所・メールアドレス）
- 名指しでの誹謗中傷
- 明らかなスパムパターン

検査はSupabase Edge Function（check-content）で実行。fail-openの方針（検査でエラーが起きた場合は投稿を許可、サーバー側の他のセーフガードに任せる）。

## 18-4. SSRF対策

公式コミュニティのURL検証時、サーバーが悪意あるURLにリクエストを送らないよう以下を実装：

- http(s)のみ許可（javascript: URL を拒否）
- プライベートIP範囲を拒否（10.0.0.0/8, 127.0.0.0/8, 192.168.0.0/16 等）
- 5秒タイムアウト
- 500KB上限（DoS対策）

## 18-5. レート制限

クライアント側に簡易レート制限を実装：

| アクション | 制限 |
|-----------|------|
| post | 5回/分 |
| comment | 10回/分 |
| reaction | 30回/分 |
| like | 30回/分 |

サーバー側RLSが真の防御を担い、クライアント側は誤操作防止が主目的。

## 18-6. プライバシー設計

- **ローカル学習**: クリック履歴・パーソナライゼーション計算はデバイス内のみ
- **広告ターゲティング**: 個人IDを広告主に送らず、タグ情報のみを使用
- **パーソナライズ広告のオプトアウト**: 完全に広告を非表示にできる
- **エラートラッキング**: Sentryに送るデータはJWT・メール・電話番号を自動削除（redact）
- **PII保護**: Sentry サンプル率5%

## 18-7. アカウント保護

- 短時間での大量試行はサーバー側でレート制限
- 不正アクセス検知時のアカウント自動保護
- 信頼スコアによる自然なスパム抑制
- 通報されたコンテンツの自動非表示（一定数を超えた場合）

# 第21章: Edge Functions（サーバー処理）

Supabase Edge Functionsで実行される5つのサーバー処理：

## 21-1. check-content
- **目的**: 投稿のコンテンツモデレーション
- **入力**: 投稿本文、タグ
- **処理**: 個人情報パターン（住所・電話番号）を正規表現でチェック
- **出力**: `{ ok: true }` または `{ ok: false, reason: "..." }`

## 21-2. send-push
- **目的**: Web Push通知配信
- **トリガー**: notificationsテーブルへのINSERT（Database Webhook）
- **処理**: 全ての登録Push Subscriptionに並列配信
- **特徴**: 失効endpoint（404/410）の自動削除

## 21-3. suggest-caption
- **目的**: 投稿キャプション自動提案
- **入力**: タグ配列
- **処理**: タグから提案文を生成（将来的にAI API統合予定）
- **出力**: `{ suggestion: "...", alternatives: ["...", "..."] }`

## 21-4. verify-official-url
- **目的**: 公式コミュニティ申請時のURL所有権検証
- **検証方式**: well-known / meta-tag の順に試行
- **セキュリティ**: SSRF対策（private IP拒否、5秒timeout、500KB上限）
- **出力**: `{ status: "verified", method: "..." }`

## 21-5. calculate-trust-score
- **目的**: 信頼スコアの計算と更新
- **入力**: user_id
- **処理**: アカウント年齢・投稿数・いいね数・コメント数から算出
- **出力**: profiles.trust_scoreの更新

# 第22章: 公式コミュニティ制度

## 22-1. 公式コミュニティとは

GEEKが審査・認証した特別なコミュニティ。アーティスト事務所、企業、公共機関などが運営し、公式バッジ（青いチェックマーク）が表示される。

## 22-2. 申請プロセス

1. 通常のコミュニティを作成（オーナーになる）
2. コミュニティ詳細画面の「公式コミュニティとして申請する」をタップ
3. 申請フォームに記入：
   - 申請者の組織情報
   - 公式サイトURL
   - 連絡先メールアドレス
   - 認証方法を選択（well-known / meta-tag）
4. 認証用トークンを公式サイトに設置
5. URL所有権の自動検証（verify-official-url Edge Function）
6. GEEK運営による最終審査
7. 承認後、公式バッジが付与される

## 22-3. URL所有権の検証方法

### well-known方式
公式サイトの `https://example.com/.well-known/geek-verify.txt` にトークンを設置。

### meta-tag方式
公式サイトのHTMLヘッダーに `<meta name="geek-verify" content="トークン">` を追加。

## 22-4. 公式コミュニティの追加機能

| 機能 | 説明 |
|------|------|
| Q&A | AIによる自動回答（管理者登録ナレッジを参照） |
| 匿名剥奪返信 | 管理者は特定のユーザー匿名性を解除して返信できる（official_author表示） |
| カレンダー管理 | イベントの追加・編集・削除 |
| 詳細分析 | メンバー数推移、投稿エンゲージメント、視聴データ |
| プッシュ通知 | 公式投稿時に参加メンバー全員にプッシュ通知 |

# 第28章: まとめ

## GEEK v4の本質

GEEK v4は単なる「もう一つのSNSアプリ」ではない。**「匿名でありながら、信頼できる場所」** という、SNSの歴史において新しい試みである。

実名SNS（X, Instagram）は「信頼できるが、本音を言えない」
完全匿名SNS（5ちゃんねる、Reddit匿名）は「本音を言えるが、信頼できない」

GEEKはこの2つの欠点を、**信頼スコアシステム × タグフィルター × 構造的安全設計** によって両立しようとしている。

## 技術的特徴のまとめ

- **モダンなスタック**: React Native + Expo + Supabase
- **クロスプラットフォーム**: iOS / Android / Web すべて単一コードベース
- **パフォーマンス重視**: コールドスタート1.2秒以内、FlashList仮想化、Worklets活用
- **プライバシーファースト**: ローカル学習、広告ターゲティング個人ID不使用、EXIF自動削除
- **拡張性**: Edge Functionsで機能追加が容易、マイグレーションで安全なDB更新

## ユーザー体験のまとめ

- **完全匿名**: 投稿者は表示されない
- **タグ中心**: 興味のあるコンテンツのみ表示
- **信頼の可視化**: スコアで品質を判断
- **公式情報**: 認証済みコミュニティから正確な情報
- **多機能**: フィード・BBS・コミュニティ・カレンダー・聖地マップ・Q&A
- **多言語**: 9言語対応で国境を越える

## 設計哲学

> 「自由と安全は両立する。匿名と信頼は両立する。
>  GEEKはその証明である。」

すべての機能・設計は「ユーザーが安心して本音を語れる場所」を作るために存在する。技術はその手段に過ぎない。
