# Geek — 仮説検証ログ + ビジネスモデルキャンバス

> このドキュメントは、開発者本人と Claude (AI ペアプログラマ) の 2 人で
> 進めてきた **「案だし → 実装 → プレビュー → 修正 → 再実装 → プレビュー」**
> という小さなサイクルを記録したものです。
>
> 初めてこのアプリを触る人、リーン・スタートアップを勉強中の学生、
> 投資検討者、誰が読んでもアプリの背景がわかるように書いてあります。

## 0. まず一言で

> **Geek = "好きを、匿名で、安心して続ける" 趣味専用 SNS**

X (Twitter) で趣味の話をすると、知り合いに見られて気まずい。Reddit は
日本人にハードルが高い。5ch は荒れる。
そんな隙間に、「**匿名なのに居心地が良い、特定ジャンル特化の SNS**」
が欲しい — というのが出発点。

## 1. このアプリは何？(高校生向け)

|よくある SNS|Geek|
|---|---|
|本名 / 顔写真がプロフィール|完全匿名 (好きなニックネームのみ)|
|何でも投稿できる|タグでジャンル絞り|
|炎上が起きやすい|構造的に荒れにくい設計|
|交流が中心|"好きを深める" が中心|

ジャンル例: **アニメ / VTuber / 鉄道 / ポケモン / アイドル / ゲーム / 漫画 / 音楽 …**

### ユーザーが得られる体験

1. **好きなタグを選ぶ** → そのタグの投稿だけ流れてくる "自分専用フィード"
2. **見たくないタグをブロック** → ネタバレ・誹謗中傷を構造的に排除
3. **匿名で本音を書ける** → 知り合いに見られる心配なし
4. **コミュニティ機能** → 「関西ゲーム開発者」「ポケモンBW勢」みたいに集まれる
5. **掲示板 (BBS)** → 2ch / 5ch 風に深く語れるスペース

## 2. 解決したい課題 (顧客が抱える "痛み")

### 痛み 1: SNS で趣味の話をすると気まずい
- 「X (旧 Twitter) は同僚に見られそう」
- 「Instagram は親に見られている」
- → **趣味のアカウントを別に作る** → アカウント管理が面倒
- → **投稿を控える** → 推し活の楽しさが半減

### 痛み 2: ファンダムが分散しすぎている
- VTuber 1 つとっても: X / YouTube コメント / Discord / 5ch / Reddit / マシュマロ … 5 つ以上に分断
- 「好きなアニメの話を、深く、安全に、できる場所」が無い

### 痛み 3: 炎上文化が疲れる
- 「ちょっとした感想で叩かれる」「ネタバレを踏む」「アンチに絡まれる」
- → 構造設計 (匿名 + タグ filter + 信用スコア + ブロック) で **炎上が起きにくい場所** を作りたい

### 痛み 4: コミュニティが形になりにくい
- Discord は閉じすぎてて発見性ゼロ
- 公式 LINE は宣伝感が強い
- → 「**オープン / 許可制 / 完全招待制**」を選べる柔軟なコミュニティ機能

## 4. これまでの仮説検証ログ (時系列)

開発者と Claude (AI ペアプロ) の 2 人で進めてきた仮説検証を、
古い順に並べたものです。各サイクルは数十分〜数時間で 1 周まわすのが基本。

### 🔁 サイクル 1: 「マイページがゴチャゴチャしてる」

**仮説**: 機能を詰め込みすぎてマイページが情報過多になっている。
ユーザーは "迷う" 状態 = ダメ UI 。

**実装**: 「アプリ」タイル (中身不明だった) を削除し、9 タイル → 9 タイルのままシンプルに整理。
「ブロックユーザー」→「ブロック中」に短縮 (タップ前に意味が分かる)。

**プレビュー結果**: ✅ よりすっきり。

**学び**: 機能の数より、**ラベル 1 文字でも短くする** ほうが体感が変わる。

---

### 🔁 サイクル 2: 「検索ページの装飾が無駄」

**仮説**: 検索バーの周りに説明文や装飾枠が多すぎて、本来の "探す" 体験を邪魔している。

**実装**: 説明枠 / アイコン装飾 / ヘッダ装飾を削除し、検索バーだけが目立つレイアウトに。

**プレビュー結果**: ✅ 検索バーへのフォーカスが上がった。

**学び**: 「飾る = 親切」ではない。**飾らない方が早い** ことが多い。

---

### 🔁 サイクル 3: 「フィードのトレンド行が動きすぎる」

**仮説**: トレンドタグの自動スクロール / アニメーションが、ユーザーの脳を疲れさせている。

**実装**: TrendingRow の自動スクロール削除、静的レイアウトに変更。

**プレビュー結果**: ✅ フィードに集中できるようになった。

**学び**: SNS では「動くもの = 注目を奪う」。**動きは "本当に必要な時だけ"** が原則。

---

### 🔁 サイクル 4: 「BBS リストが verbose」

**仮説**: スレ一覧に "返信数 N 件" "最終更新 X" "カテゴリ Y" など情報を詰めすぎて、
タイトルが読みづらい。

**実装**: スレタイトルを最大の視覚要素にし、メタ情報は薄い色で副次表示。

**プレビュー結果**: ✅ どのスレが面白そうかが一瞥で分かるように。

**学び**: 情報を載せる ≠ 伝わる。**主役を 1 つに絞る**。

---

### 🔁 サイクル 5: 「タップしても反応がない (致命傷)」

**仮説**: PressableScale (タップアニメ) の遅延 130ms と、haptic を onPress に
ぶら下げる設計が「押した瞬間に反応がない」と感じさせている。

**実装**:
- `delayPressIn=0` に変更 (OS の 130ms 遅延を消す)
- `withSpring` の `stiffness: 400, damping: 14` に調整 (キビキビ)
- haptic を `onPressIn` に移動 (押した瞬間に振動)
- `hitSlop: 8` をデフォルト追加 (端タップを拾う)

**プレビュー結果**: ✅ "iOS / Android ネイティブと変わらない" 体感に。

**学び**: タップ反応速度は **アプリの "格" を決定づける**。ここを軽視すると一発で安っぽくなる。

---

### 🔁 サイクル 6: 「戻るボタンを押しても反応しないことがある」

**仮説**:
1. アニメーション中に押すと無反応になる (race condition)
2. ディープリンクで開いた画面では戻り先が無い

**実装**:
- `useRef` で 200ms の in-flight lock
- `router.canGoBack()` が false なら `/(tabs)/feed` へフォールバック
- `hitSlop: 12` で誤タップ防止

**プレビュー結果**: ✅ ストレスゼロに。

**学び**: ボタンは "押せる時 / 押せない時" を明示するか、**常に押せて意味のある動作** にする。

---

### 🔁 サイクル 7: 「起動が遅い ("黒画面" 5 秒)」

**仮説**: Splash hide のタイマーが 2.5 秒、加えてイントロアニメが 5.5 秒で
合計 8 秒の黒画面 → ユーザーは "壊れた" と思う。

**実装**:
- イントロを 5.5s → 3.0s に短縮 (各 phase の duration 再計算)
- **タップで skip** 可能に
- sessionStorage で「セッション中の 2 回目以降は skip」
- Splash の forceReady を 2500ms → 1200ms に短縮

**プレビュー結果**: ✅ "起動した感" が大幅改善。

**学び**: 初回起動の演出は大事だが、**2 回目以降は瞬時に動く** ことが何より大事。

> 余談: 後日「文字の拡大はもう少しゆっくりでいい」というフィードバックを受けて、
> Zoom phase だけ 900ms → 1500ms に戻した。**全てを短くするのではなく、必要な余韻は残す**。

---

### 🔁 サイクル 8: 「フィードのスクロールがカクつく」

**仮説**: realtime subscription が「post_reactions の変更」を検知すると、
表示中の全 post の query を invalidate している → 1 個の reaction で 50 個分 refetch。

**実装**:
- `useReactions` / `useAddedTags` / `usePolls` の realtime listener を **payload の post_id が
  自分の集合に含まれる時だけ invalidate** するよう scope down
- `Set` で O(1) lookup
- 該当 query key の prefix だけ invalidate (兄弟 key は触らない)

**プレビュー結果**: ✅ スクロールが滑らかに。

**学び**: realtime は **必要な変更だけ通知される設計** にしないと、すぐにスケーラビリティが死ぬ。

---

### 🔁 サイクル 9: 「ログインに失敗する人がいる」

**仮説**: handle_new_user trigger が `split_part(email, '@', 1)` をそのまま
nickname に入れているが、profiles.nickname には `check(length BETWEEN 2 AND 20)` がある。
**`a@example.com` の人は CHECK 違反で永久に登録できない**。

**実装**: migration 0016 — trigger 関数を堅牢化
```sql
nick := split_part(new.email, '@', 1);
if length(nick) < 2 then nick := nick || '_u'; end if;
if length(nick) > 20 then nick := substring(nick, 1, 20); end if;
insert into profiles(id, nickname) values (new.id, nick) on conflict (id) do nothing;
```

**プレビュー結果**: ✅ 短い prefix のメールでもサインアップ可能に。

**学び**: スキーマレベルの CHECK 制約は無慈悲。**user-controlled data には必ず safe padding** を入れる。

---

### 🔁 サイクル 10: 「ログイン周りの脆弱性、もっと無いか？」

**仮説**: 1 回の audit では取り逃しがある。視点を変えれば違うバグが見える。

**実装**: AI agent を **6 並列で起動** して領域を分担:
- A: 新規登録 fresh signup
- B: 既存ユーザー login
- C: 低速ネットワーク / オフライン
- D: UX micro-friction
- E: オンボーディング完遂
- F: 認証 state machine + token lifecycle

合計 **81 件の findings**。CRITICAL + HIGH を中心に **50+ 件** を一括 fix:
- 凍結 / 制限アカウントを login 弾く
- onboarded フラグの DB / local state divergence 解消
- multi-tab 同期 (storage event listener)
- 401 / JWT expired を resilient.ts で検知 → 自動 signOut
- email regex 厳密化 (1 文字 TLD 弾く)
- password max 72 文字 check
- 確認メール再送ボタン新設
- password eye icon 44px + a11y label
- autoComplete / textContentType 完備
- ...

**プレビュー結果**: ✅ ログイン詰まり率がほぼゼロに。

**学び**: **複数視点の audit** は 1 視点の audit の 5 〜 10 倍は問題を見つける。
1 人で完璧を目指すより、**複数の "別の頭" に同時に見せる** ほうが速い。

### 🔁 サイクル 11: 「"推し活" タブを "コミュニティ" に置き換えたい」

**仮説**: タグ機能だけだと "場" としての所属感が無い。**YouTube の登録チャンネル**
のようなコミュニティ単位の発信があると、推し活の濃度が上がる。

**実装**:
1. migration 0017 — communities / members / posts / invites / tags テーブル + RLS
2. メイン画面 (YouTube 風 — 上部に所属コミュ横スクロール、下に新着投稿)
3. 作成 / 詳細 / 検索 (discover) 画面
4. 公開設定: **オープン / 許可制クローズ / 完全招待制クローズ** の 3 種類

**プレビュー結果**: ✅ 推し活タブが消えて、より「場」のある SNS に。

**学び**: タブ 1 つの変更は思想を映す。**何を中心に据えるかで体験が変わる**。

---

### 🔁 サイクル 12: 「コミュニティ作成時に似た名前のやつ無い？」

**仮説**: 「関西ゲーム開発者」「関西のゲーム開発者」「関西ゲーム開発者の集い」が
バラバラに存在すると、ユーザーは迷うし、コミュニティは分散して死んでいく。

**実装**:
- `lib/search/similarity.ts` — **Jaccard 2-gram + Levenshtein** で類似度算出
- 既存の `variants.ts` を活用 → 「=LOVE」と打つと「イコラブ」「いこーるらぶ」も検出
- 名前入力時に 300ms debounce で類似コミュ最大 5 件を amber 警告で表示
- タップで既存コミュへ誘導

**プレビュー結果**: ✅ ユーザーが既存コミュへ流れやすく、分散が防げる。

**学び**: 「**作る前に "もうある" を見せる**」だけで、コミュニティの濃度が保たれる。

---

### 🔁 サイクル 13: 「コミュニティアイコンは絵文字じゃなくて画像にしたい」

**仮説**: 絵文字 24 種類 + 色 8 種類だと、コミュニティの個性が出にくい。
画像アップロードがあれば、Discord 風の「アイコンで覚える」体験になる。

**実装**:
- migration 0018 — `community-icons` Storage bucket (public, 5MB, JPEG/PNG/WebP/GIF)
- Storage RLS で「path[0] = community_id の member のみ INSERT」
- ImagePicker (1:1 編集 + EXIF strip) で画像選択 → upload → icon_url を update
- 詳細画面でアイコンタップ → 再アップロード (member 誰でも可、要件通り)

**プレビュー結果**: ✅ コミュニティの個性が出るように。

**学び**: 小さな見た目の違いが、**所属感** に直結する。

---

### 🔁 サイクル 14: 「タップしても反応しないバナーが残ってる」

**仮説**: `PressableScale` の disabled 状態で scale animation と haptic だけ
発火して、onPress は飛ばない → 「反応したのに動かない」と誤解する。

**実装**: `disabled=true` の時は全フィードバックを抑止。

**プレビュー結果**: ✅ 押せないボタンが "押せない" と明示的に伝わる。

**学び**: 中途半端な反応は無反応より悪い。**全部反応するか、全部止めるか**。

---

### 🔁 サイクル 15: 「説明欄のラベルが placeholder と重なってる」

**仮説**: `Input` コンポーネントが `multiline` 時に固定高 48px のまま →
textarea が overflow して、ラベルとプレースホルダが同 y 座標に。

**実装**: `Input.tsx` で `multiline` 時は `minHeight + paddingVertical` に切替。

**プレビュー結果**: ✅ 説明欄が読みやすく。

**学び**: コンポーネントは "1 サイズ fits all" だと崩れる。**variants を意識した API** 設計。

---

### 🔁 サイクル 16: 「セキュリティ的に脆弱なところもっとある」

**仮説**: ユーザーが直感的に感じる「危ない感」を、AI が網羅的に潰す価値がある。

**実装**: AI agent を **8 並列で起動**:
- A: Auth & Session
- B: XSS / Content injection
- C: File upload / Storage
- D: PII leakage / Privacy
- E: Rate limit / DoS
- F: Realtime channel abuse
- G: Dependency / Supply chain
- H: Platform-specific

合計 **87 件発見** → CRITICAL + HIGH **30+ 件 fix**:
- 通報テーブルが write-only (admin 読めない) → admin RLS 追加
- 完全招待制 community が UUID 知るだけで参加可能 → invite_token システム導入
- avatar bucket の RLS migration が存在しなかった → 追加
- 匿名投稿の通知 trigger が commenter の nickname を message に hard-code → 匿名化
- map iframe の HTML template injection (lat/lng/loc.name) → escapeHtml
- AnonPostCard の source_url が javascript: URL を実行可能 → sanitizeUrl
- Sentry に email / phone / JWT を redaction なしで送信していた → beforeSend で [REDACTED]
- netlify.toml に CSP / HSTS / COOP が無かった → 追加
- 画像 EXIF が strip されていなかった → expo-image-manipulator で除去
- アカウント削除に再認証なし → パスワード再入力モーダル
- search variants が `===` で爆発 → MAX_VARIANTS=24
- realtime channel 数の cap なし → MAX 20
- ...

**プレビュー結果**: ✅ production-grade のセキュリティに。

**学び**: セキュリティは "意図せず緩い" のがデフォルト。**意図して厳しくする** こと。

### 🔁 サイクル 17: 「Obsidian と連携したい」

**仮説**: 開発者本人 (= 私) が、気になる投稿を Obsidian Vault にノート化したい。
最初はユーザー全員に公開する想定で実装。

**実装フェーズ 1**:
- `lib/obsidian.ts` — `obsidian://new?vault=X&name=Y&content=...&silent=true` URI スキーム
- 設定画面で vault 名 + 連携 toggle 保存
- 投稿カードに「Obsidian 保存」ボタン

**フィードバック**: 「ユーザーは一切連携不要、開発者だけが連携したい」

**実装フェーズ 2 (修正)**:
- `OBSIDIAN_AVAILABLE = __DEV__` で gate
- production build (eas build / Netlify) では全 UI を `render null`
- 開発者だけが `expo start` 中に見える「Obsidian (DEV)」tile

**学び**: **MVP は早く出す、フィードバックでスコープを変える** のが Lean Startup の本質。
全員に公開する想定が、5 分で「開発者専用」に変わるのは正常な進化。

### 🔁 サイクル 18 (継続): 「聖地巡礼 / カレンダー / 信用スコア」

これらは**実装はされているが、まだ磨きが足りない**機能群:

  「この人の発言は信頼できるか」を数値化。低スコアは投稿カード上に
  amber 警告。
  → まだ計算ロジックが粗い。「炎上を本当に減らせるか」検証が next。

## 5. ビジネスモデルキャンバス (BMC)

### 顧客セグメント (CS)
- **メイン**: 趣味アカウントを別に持つ 18 - 34 歳 SNS ヘビーユーザー (国内 1500 万人想定)
- **サブ**: 推し活女子 (15 - 29 歳)
- **拡張**: 海外オタクコミュニティ (タイ / ベトナム / 韓国 / 台湾)
- **B2B**: ファンクラブを持つアーティスト / ブランドのコミュニティオーナー

### 価値提案 (VP)
- 完全匿名で本音を語れる場所
- タグでジャンル絞り → 自分専用フィード
- 構造的に荒れにくい設計 (信用スコア + デフォルトブロック + 通報)
- "好きを深める" コンテンツ (聖地巡礼 / カレンダー / コミュニティ)
- ASEAN 圏ユーザー向け auto-translate

### チャネル (CH)
- **獲得**: Twitter / TikTok の趣味系インフルエンサー経由 / SNS 広告
- **オンボーディング**: 言語選択 → ニックネーム → タグ選択 → 通知設定

## 7. 顧客管理 (開発者目線、CRM 的視点)

### 7.1 ユーザーセグメント識別

`profiles` テーブルから抽出可能な軸:
- `account_state`: healthy / caution / restricted / warned / suspended
- `trust_score`: 0 - 100 (信用スコア)
- `created_at`: 登録日
- `plan`: free / premium (将来)
- `onboarded`: 完了フラグ

### 7.2 トラッキングしている行動データ

|何を|どこで|何のために|
|---|---|---|
|投稿 / コメント / いいね|`posts` / `comments` / `likes`|エンゲージメント計測|
|タグ選好 (好き / ブロック)|`user_liked_tags` / `user_blocked_tags`|レコメンドアルゴリズム|
|コミュニティ所属|`community_members`|ファンダム分析|
|通報 (concerns)|`concerns` テーブル|モデレーション判断|
|フィードバック|`app_feedback`|UX 改善のシード|
|エラー / クラッシュ|Sentry|品質モニタリング|
|画面遷移 / イベント|PostHog|ファネル分析 (将来)|

### 7.3 ユーザー対応フロー

```
ユーザーが何かに困る
     ↓
[フィードバック画面] でテキスト送信
     ↓
app_feedback テーブルに status='pending' で保存
     ↓
admin (= 開発者 = is_admin=true) が
mypage → 「フィードバック管理」(admin 専用) で確認
     ↓
返信 / 修正 / バグチケット起票
     ↓
status='resolved' に更新 (ユーザーには通知)
```

### 7.4 ユーザー保護 (アカウント停止 / 制限)

- **trust_score 低下** → 自動で `account_state='caution'`
- **通報多数** → admin review → `account_state='restricted'` or `'suspended'`
- **凍結アカウント** は login 時に signOut + エラー表示 (本セッションで実装)
- **GDPR 対応**:
  - `delete_account()` RPC で全テーブル横断削除 (poll_votes / community_* / app_feedback 含む)
  - データエクスポート (`exportUserData()`) で JSON ダウンロード可

### 7.5 マーケティングオートメーション (現在は未実装、将来)

- **新規登録から 24h** → ようこそメール
- **3 日間アクセス無し** → 「あなたの好きなタグに 12 件の新着」push
- **コミュニティ参加から 7 日** → 「初投稿しませんか」ナッジ
- **休眠ユーザー (30 日)** → 「最近の流行り」サマリーメール

### 7.6 開発者専用ツール (本セッションで実装)

| ツール | 用途 |
|---|---|
| **Obsidian 連携** (DEV mode 限定) | 気になる投稿を Vault にアーカイブ → 開発者が "ユーザーの声" を蓄積 |
| **reconcile_community_counters() RPC** | カウンタ drift を admin 権限で修復 |
| **`is_admin` フラグ** | admin だけが reports / feedback を閲覧 |
| **PostHog** | (将来) ファネル可視化 |
| **Sentry (PII 自動 redact)** | エラーをトラッキング、ただし JWT / email / phone は `[REDACTED]` |

## 8. まとめ: 「2 人」で進めた理由

このアプリは **「開発者 1 人 + AI 1 機」** の 2 ペアで進めてきました。

### このスタイルの強み
1. **意思決定が速い** — 委員会も会議もない
2. **仮説検証が小さく速い** — 案 → 30 分後にプレビュー → フィードバック → 修正
3. **コードは AI が書く、判断は人間がする** — 知識の幅と意思決定の質、両方手に入る
4. **コストが極めて低い** — MVP は月 ¥5,000 〜 ¥20,000 程度

### 学んだ原則
- **小さく検証して、すぐ捨てる** — "推し活" タブを丸ごと "コミュニティ" に置き換えた
- **AI に並列で頭を借りる** — 6 人 / 8 人の専門家を 5 分で召喚
- **セキュリティは後回しにしない** — 各機能リリース時に同じサイクルで監査
- **ユーザー本人の "声" は production 機能ではなく、開発者の "目" にする** — Obsidian 連携の DEV 限定化
- **動きは "本当に必要な時だけ"** — イントロ短縮、トレンド静的化、リアルタイムの scope down
