# フィード・ランキング・レコメンド

Geek (geek-v4) のホームフィードがどう「投稿を集めて・並べて・1ページ表示する」かの実態記録。
クライアント側の素朴な個人化(`smartRank`/`score.ts`)から、サーバー側 Value Model 個人化フィード
(`get_for_you_feed` / 0139-0141)まで、複数世代の仕組みが **fallback チェーン** で重なっている。
本ノートは実コード(2026-06-10 時点)と memory を突合し、**どれが本番で効いていて・何が未配線(★)か** まで書く。

---

## 概要

- **ホームフィードの sort 軸**: `for-you` / `hot` / `new` / `top` / `rising`(`SortMode`、`lib/api/posts.ts`)。
  既定は **`for-you` + scope `open`**。この組合せだけが「Value Model 個人化 RPC」の対象。
- **3 世代のランキングが共存**している(新しいほど優先・失敗で下へ落ちる):
  1. **サーバー Value Model 個人化** = `get_for_you_feed` RPC(migration `0141`)。タグ親和性・既読除外・コールドスタート・多様性を **DB 側** で適用。**本番で有効化済**(`EXPO_PUBLIC_FOR_YOU_FEED_RPC=1`)。
  2. **サーバー hot プール集約** = `get_home_feed` RPC(migration `0114`)。個人化なしの hot 順 1RTT 集約。`get_for_you_feed` が無い/失敗時の中間 fallback。**既定 OFF**(`EXPO_PUBLIC_HOME_FEED_RPC=1` のときだけ)。
  3. **素の `fetchPosts`(cursor)** = `lib/api/posts.ts`。RPC 全滅・2ページ目以降・別 sort/closed scope はこれ。最終 fallback で **絶対に壊れない経路**。
- **設計思想**: RPC は全部「1ページ目専用・成功時 1RTT 集約・失敗/0件は `null` を返して下位へ完全 fallback」。
  migration 未適用でもフラグ OFF でも UI は `fetchPosts` で動く(= **silent degrade の罠** に注意、後述)。
- **個人化の素性**: Instagram Explore の多段ファネル + Value Model の簡略移植。deep-research(YouTube/Instagram Two-Tower + cold-start)で検証した知見ベース。pgvector(post_embeddings)は infra だけ存在し **未使用**。

---

## 仕組み・設計(具体ファイルパス付き)

### A. ランキング fallback チェーン(単一ソース)

`lib/feed/feedQuery.ts` の `fetchFeedFirstPage()` が「ホームフィード1ページ目」をどう取るかを集約。
`hooks/useFeed.ts`(画面 mount)と `app/_layout.tsx`(起動時 prefetch)が **同一 queryKey + 同一 queryFn** を共有するための単一ソース。

```ts
// lib/feed/feedQuery.ts (要点)
if (sort === 'for-you' && scope === 'open') {
  if (FOR_YOU_FEED_RPC_ENABLED) {            // ① 0141 Value Model
    const forYou = await fetchForYouFeedFirstPage(userId);
    if (forYou) { seedHomeFeedSurroundingCaches(qc, userId, forYou.posts); return ...; }
  }
  if (HOME_FEED_RPC_ENABLED) {               // ② 0114 hot 集約
    const home = await fetchHomeFeedFirstPage(userId);
    if (home) { seedHomeFeedSurroundingCaches(qc, userId, home.posts); return ...; }
  }
}
return fetchPosts({ sort, likedTags, blockedTags, filterTags, cursor: undefined, home: true }); // ③ 最終
```

- queryKey は `feedQueryKey(sort, scope, likedTags, blockedTags)` = `['feed', sort, scope, likedTags, blockedTags]`。
  **`useFeed.ts` の `useInfiniteQuery` と完全一致必須**(ズレると起動 prefetch が当たらない=無害だが効果なし)。
- フラグ定義は `lib/api/homeFeed.ts`:
  - `FOR_YOU_FEED_RPC_ENABLED = process.env.EXPO_PUBLIC_FOR_YOU_FEED_RPC === '1'`
  - `HOME_FEED_RPC_ENABLED = process.env.EXPO_PUBLIC_HOME_FEED_RPC === '1'`(★ **コードベース初の「既定 OFF」フラグ**。コア feed・cursor/cache 契約に触る高リスクなので `!== '0'`(既定 ON)パターンを反転)。

### B. サーバー Value Model 個人化 — `get_for_you_feed`(migration `0141_get_for_you_feed.sql`)

Instagram Explore の多段ファネルを PostgreSQL で実装。`security definer` + `stable`。1ページ目専用(既定 limit 12、内部 clamp 1..50)。

**多段ファネル(CTE で実装):**

| Stage | 内容 | 実装 |
|---|---|---|
| 0. cold start 判定 | `user_tag_affinity` が 1 件でもあれば personalized、無ければ cold | `v_has_affinity` |
| 1. 候補プール 150 件 | 過去 **7日** ・公開・可視性チェック・**既読除外** → `hot_score` 降順で 150 件に絞る | `base_candidates` CTE |
| 2. Value Model スコアリング | 150 件を `for_you_score` で再ランク | `scored` CTE |
| 3. 多様性 | dominant tag(`tag_names[1]`)ごと **最大3件** | `diverse`(`row_number() over partition by tag_names[1]`) |
| 4. 最終 | `tag_rank <= 3` を `for_you_score` 降順で `v_limit` 件 | `foryou_pool` |
| 5. 周辺データ JOIN | communities / official_author / my_like|concern|save / reactions / added_tags / poll を 1 json に | 0114 と同一 |

**Value Model スコア式(personalized 時、`scored` CTE):**

```
for_you_score = tag_affinity_bonus × 4.0                                  -- マッチタグの平均 affinity_score
              + ln(likes_count + 1) × 2.0                                  -- エンゲージ(対数)
              + ln(comments_count + 1) × 1.5                               -- コメント(対数)
              - least(concern_count × 0.3, 3.0)                            -- 低品質ペナルティ(max 3.0 cap)
              + (1 / (1 + epoch_since_created / 28800)) × 2.0              -- 鮮度(8h half-life)
              + cold_boost                                                 -- 新規露出保証: 2h以内=5pt, 6h以内=2pt
```

- `tag_affinity_bonus` = `user_affinities`(上位 30 タグ)と投稿 `tag_names` の **マッチタグの平均 affinity_score**。
- **cold start ユーザー**(affinity 無し)は `for_you_score = coalesce(hot_score, 0)` のみ = `get_home_feed` と同等体験。
- **既読除外** = `seen_posts` CTE: `post_impressions` で **3日以内に `seen_count > 1`(=2回以上見た)** 投稿を `not in` で除外。
- **cold_boost** は popularity bias 対策(新規投稿が hot 上位に勝てない問題への保証露出)。

**匿名性・セキュリティ(0107/0114 と同一述語を必ずコピー):**
- 可視性述語: `(public.can_view_post(p.id) or p.author_id = auth.uid()) and public.author_visible(p.author_id)`。`security definer` で RLS を bypass する以上、この述語を省くと情報漏えい。
- IDOR gate: `p_user_id` が `auth.uid()` と不一致なら `42501` で raise。
- **匿名 author_id マスク**: `is_anonymous` かつ viewer 本人以外には `author_id` を NULL で返す。`is_own` boolean を別途供給。詳細は [[匿名性設計と de-anon ホール]]。
- `nextCursor` は `hot_score|ISO_ts` 形式(`fetchPosts` cursor と互換)。2ページ目以降は `fetchPosts` 経路へ。

### C. タグ親和性テーブル — `user_tag_affinity`(migration `0139`)

Two-Tower の「ユーザーベクトル」を Supabase で持つ基盤。

- スキーマ: `(user_id, tag_name)` PK / `affinity_score real 0..100` / `event_count` / `last_event_at`。RLS は `auth.uid() = user_id` の自己アクセスのみ。index `(user_id, affinity_score desc)`。
- RPC:
  - `upsert_tag_affinity(text[], real)` — タグ親和性を加算/作成。`affinity_score` を `0..100` で clamp。タグ上限 50。
  - `get_user_top_tags(int)` — 上位タグ降順(clamp 1..50)。
  - `run_decay_tag_affinity()` — **週次減衰**: 7日以上更新なしを ×0.9、0.5 未満を削除。**pg_cron 登録必須**。
- シグナル重み(client `AFFINITY_DELTA` ↔ server `p_delta` が対応):
  いいね **+3.0** / 保存 **+2.0** / 長閲覧 **+1.0** / タグクリック **+1.5** / unlike **-2.0** / 懸念 **-3.0** / 非表示 **-4.0**。

### D. 閲覧履歴テーブル — `post_impressions`(migration `0140`)

既読除外 + コールドスタート追跡の基盤。

- スキーマ: `(user_id, post_id)` PK / `first_seen_at` / `last_seen_at` / `seen_count`。RLS は自己のみ。index `(post_id)`。
- RPC:
  - `record_impression_batch(uuid[])` — バッチ閲覧記録(`seen_count` インクリメント、上限 100)。
  - `cleanup_old_impressions()` — TTL 30日削除。**pg_cron 登録必須**。

### E. クライアント側の同期/記録ライブラリ(`lib/personalize/`)

| ファイル | 役割 | 配線状況 |
|---|---|---|
| `events.ts` | 端末ローカル行動ログ(FIFO 1000件・debounce 500ms・schema 検証)。`logEvent` 内で **いいね/保存/unlike/懸念/非表示/タグクリック時に `pushAffinityDelta` を fire-and-forget** でサーバー同期 | ✅ 統合済 |
| `syncAffinity.ts` | `pushAffinityDelta(tags, delta)`(→ `upsert_tag_affinity` RPC)/ `syncAffinityOnStartup(tagAffinity)`(MMKV→server、12h に1回) / `AFFINITY_DELTA` 定数 | `pushAffinityDelta` ✅ / **`syncAffinityOnStartup` ★未配線**(後述) |
| `impressions.ts` | `recordImpression(postId)`(Set バッファ・20件/30s で自動 flush → `record_impression_batch`)/ `flushImpressions()` | `recordImpression` ✅ / **`flushImpressions` ★未配線**(後述) |
| `score.ts` | クライアント側マルチ目的ランキング(`rankFeed`/`diversifyFeed`/`scoreCandidate`)。interest × quality × freshness × novelty × session × trending + exploration budget + diversity injection | feed.tsx 本線では未使用(下記参照) |
| `profile.ts` | `computeProfile` — events → `UserInterestProfile`(tagAffinity 等) | score 系の入力 |

**`recordImpression` の配線(memory の TODO は解消済)**: `app/(tabs)/feed.tsx:607` の `handleViewableItemsChanged` 内で、
`viewabilityConfig = { viewAreaCoveragePercentThreshold: 30 }`(30% 以上表示)の各 post に対して呼ぶ(広告アイテムは `isAdItem` でスキップ)。
→ `project_recommendation_system.md` の「`recordImpression` を onViewableItemsChanged に追加(未実装)」は **現コードでは実装済**。

### F. クライアント側ランキング(`lib/feed/smartRank.ts` / `lib/personalize/score.ts`)

サーバー RPC 登場前の世代。**現在ホームフィード本線では主役ではない**が残存:

- `smartRank.ts` の `smartSort` は **タグ詳細画面**(`app/tag/[name].tsx:95`)で使用。シグナル = likedタグ affinity / recentTags / trendingTags / CTR boost / trust / recency(24h half-life)/ engagement。mode 別重み(`hot`/`new`/`top`)。
- `score.ts` の `rankFeed`/`diversifyFeed` は barrel(`lib/personalize/index.ts`)で export されるが feed.tsx 本線の呼び出しは確認できず(score 系は profile ベースのより重いランカー)。
- `homeFeed.ts` のコメントには「client `rankFeed` の1ページ目 再ランク母集合」への言及があり、初回 limit を 30→**12**(`HOME_FEED_FIRST_PAGE_LIMIT`)に絞ったことで再ランク母集合が縮むトレードオフが記録されている(初回 payload を ~⅓ に圧縮するため)。

### G. 周辺データ取得と cache seed(`feedPage.ts` / `homeFeed.ts`)

- `get_for_you_feed` / `get_home_feed` は base posts + **周辺データ**(communities / official_author / my_like|concern|save / reactions / added_tags / poll)を **同一 JSON shape** で返す。`lib/api/homeFeed.ts` は `normalizeFeedPageRow`(`feedPage.ts`)を再利用して `['feed-page']` cache と shape を厳密一致させる。
- `seedHomeFeedSurroundingCaches(qc, userId, posts)`(`homeFeed.ts`)が **非破壊 seed**(`(prev) => prev ?? seeded`)で 2 つの cache を埋め、下流 RPC の二重 fetch を抑止:
  - `['feed-page', userId??'anon', stableKeyFor(ids.sort())]` ← `useFeedPage` と同形
  - `['feed-post-communities', ids.join('|')]` ← `useFeed` の communitiesQ と同形
- 2ページ目以降の周辺データは `fetchFeedPage`(`feedPage.ts`)= `get_feed_page` RPC(migration `0041`)が担う(postIds を最大 100 で cap、8s timeout、失敗で空配列 → 旧 hook 群へ fallback)。

## 注意点・地雷

### ★ 有効化は「手動 3 点セット」— 1 つ欠けると silent degrade

`get_for_you_feed` を本番で効かせるには **全部** 必要。どれか欠けると例外を出さず古い経路に静かに落ちる(= 直ったつもりで効いていない罠):

1. **migration 手動適用**: `0139` + `0140` + `0141`(+前提 `0138`)を **Supabase SQL エディタで順に実行**。Netlify/Supabase CLI は本番に migration を流さない(0118-0123 の教訓)。
   - 未適用だと RPC が `PGRST202` を返す → `fetchForYouFeedFirstPage` が `null` → fallback。
2. **pg_cron 登録**(Supabase Dashboard → Database → pg_cron):
   ```sql
   select cron.schedule('decay-tag-affinity', '0 3 * * 1', $$select public.run_decay_tag_affinity()$$);
   select cron.schedule('cleanup-impressions', '0 4 * * *', $$select public.cleanup_old_impressions()$$);
   ```
   未登録でもフィードは動くが、affinity が減衰せず・impressions が無限に貯まる。
3. **フラグ設定**: Netlify 環境変数 `EXPO_PUBLIC_FOR_YOU_FEED_RPC=1`(置き場は **`geek-v4/netlify.toml` の `[build.environment]`**、repo root のは無視)。**`=== '1'` 判定**(`!== '0'` をコピペ流用すると意図せず既定 ON 事故)。`EXPO_PUBLIC_*` は static inline なので `process.env[key]` 動的アクセス禁止・変更後は再 build。

→ memory `project_recommendation_system.md`(2026-06-09)では **上記 1-3 すべて適用済・Value Model フィード本番稼働中** と記録。
ただし memory は point-in-time。実際に効いているかは、本番で `get_for_you_feed` が呼ばれているか(ネットワーク)で確認すること。

### ★ 未配線で残っている起動/終了フック(memory と現コードの差分)

- **`syncAffinityOnStartup(profile.tagAffinity)`**: 定義はあるが呼び出し元なし(grep で `syncAffinity.ts` のコメントのみ)。
  → MMKV のローカルタグ履歴がサーバー `user_tag_affinity` に **初回バルク同期されない**。`pushAffinityDelta`(いいね等の都度同期)は効いているので、行動を重ねれば affinity は育つが、既存ユーザーの過去履歴は反映されない。
- **`flushImpressions()`**: 定義はあるが呼び出し元なし。アプリ終了/BG 時の未送信 impression(< 20件 かつ 30s 未経過)が取りこぼされる。スクロール継続中は閾値 flush で送られるので影響は軽微。
- **`post_embeddings` Edge Function**(0096 infra)は未デプロイ・pgvector HNSW は未使用(deep-research では verified だが実装は親和性タグベースに留まる)。

### 0114 → 0141 のスコア境界・cap の注意

- 候補プールは `hot_score desc` で **150 件に固定**してから Value Model 再ランク。`hot_score` 上位 150 に入らない投稿は personalized でも浮上しない(深い掘り起こしはしない設計)。
- 多様性は **dominant tag(`tag_names[1]`)ごと最大3件**のみ。`tag_names` 先頭タグに依存するので、タグ順が安定していないと多様性の効きが変わる。
- 鮮度は **8h half-life**(`28800` 秒)。client `smartRank` の 24h half-life とは別物。世代で減衰時定数が違う点に注意。
- いいね/懸念の数の効き方は **対数 + cap**。「いいねが 1 増えても並びがほぼ動かない」のは仕様(線形ではない)。なお過去に別レイヤで「いいね反映バグ(vote fuzz の自票飲み込み、tier 境界 2↔3/10↔11)」があった → [[フィード・ランキング・レコメンド]] 外、cache 層は正常だった(修正済)。

### 0件返却を fallback トリガにする設計の含意

- `fetchForYouFeedFirstPage` / `fetchHomeFeedFirstPage` は **0件でも `null`** を返す(= fallback)。「RPC 成功だが候補なし」を空 feed として見せず `fetchPosts` に拾わせる安全策。
  真に空の feed(新規ユーザー)は fallback も空を返すので結果は同じ(+1 RTT のみ)。
  → 「為念 RPC が過少返却してる?」を疑うときは、0件 fallback で `fetchPosts` 結果に化けている可能性を念頭に。

### パフォーマンス連動

- 初回 limit を 30→**12**(`HOME_FEED_FIRST_PAGE_LIMIT`)に絞り初回 payload を ~⅓ に圧縮。トレードオフ = for-you 初回並びの再ランク母集合が縮む。
- feed cold open の 3段ウォーターフォール(posts → get_feed_page + communities)を RPC で 1RTT 集約するのがそもそもの 0114/0141 の動機。詳細は [[パフォーマンス最適化]]。
