# パフォーマンス最適化

Geek (geek-v4) の体感速度・起動速度・スクロール 60FPS を支える最適化の実態記録。
2026-06-03 の「ロードが遅い」4 並列監査 + フィード周りの継続改善が起点。
**「解消済み」と「残課題 (★未着手/要検証)」を明確に分けて記録する** — 古い情報を「直っている」と誤認しないため。

> 出典: `memory/project_geek_v4_perf_hotspots.md` (2026-06-03 監査) / `geek-v4/CLAUDE.md` §2 (Metro) §4 §11 /
> `components/feed/AnonPostCard.tsx` / `app/(tabs)/feed.tsx` / `babel.config.js` の実コード照合 (2026-06-10)。

---

## 概要

最適化は大きく 4 レイヤに分かれる。どこがボトルネックかは層ごとに切り分けて測る。

| レイヤ | 主なコスト | 主な対策 | 状態 |
|---|---|---|---|
| 起動 (cold start) | intro 再生 / hydrate / prewarm 競合 | 永続フラグで intro 1 回化・prewarm 遅延 | ✅解消 (一部 native は確定演出に固定) |
| バンドルサイズ (web) | lucide 全 icon / フォント TTF | babel deep import / フォントは要検証 | ⚠️一部 (lucide 解消, フォント★残) |
| データ取得 | feed 3 段ウォーターフォール / 多往復 RPC | RPC 集約・focus invalidate・押下 prefetch | ⚠️一部 (討議集約は ★未deploy/要DB) |
| レンダリング (60FPS) | カード再 render / FlashList overscan / style 再生成 | memo + selector + useMemo + Reanimated worklet | ✅解消 (継続中) |

**測定の鉄則 (CLAUDE.md 由来):** 推測でなく実測する。バンドルは build 後の `dist/` を実測、
画像潰れは `image-size` で本番 URL 実測、cache の中身は React Query を fiber 経由で覗く。
「green build (type-check/lint 0) = 速い・安全」ではない。

関連の全体像は [[アーキテクチャ概要]]、フィード取得経路は [[フィード・ランキング・レコメンド]]、
データ往復の RPC は [[データ層・Supabase・RLS・マイグレーション運用]] を参照。

## 仕組み・設計 (具体ファイルパス付き)

### 1. 起動 (cold start) の最適化 — ✅解消済

- **IntroAnimation の毎回再生を撲滅 (native の最大体感遅延だった)**
  - 旧: `IntroAnimation` が **毎 cold start で ~3s 再生**。`sessionStorage` ガードは web 専用で、
    native は `shouldShowIntro` が常時 true だった。
  - 対策: 永続フラグ `geek:intro_seen_v4` (`lib/storage.ts` の `getBool/setBool`) で「一度だけ」に。`app/_layout.tsx`。
  - ⚠️ **2026-06-06 に起動演出 (splash + intro) は「確定版・変更禁止」になった** (CLAUDE.md §0)。
    現行は splash → intro → 本体が seam なく繋がるよう寸法/色/タイミングを固定し、
    バー sweep を必ず 2 周完走 (`SWEEPS_BEFORE_EXIT=2 × SWEEP_MS=1150` = ~2.3s 表示 + fade ~2.6s 体感)。
    **「速くする」名目で勝手に短縮しない**。回帰テスト `tests/unit/introSplashLock.test.ts` が A↔B 固定値一致を assert。
  - 古い情報: memory の「intro 5.5s→3.0s + sessionStorage で 2 回目 skip」は **当時の話**。
    今は永続フラグ + 確定演出に置き換わっている。

- **起動時 prewarm を first paint と競合させない**
  - コミュ pre-warm prefetch 2 本が first paint と競合 → `setTimeout(300ms)` に遅延 (`app/_layout.tsx`)。
  - idle prewarm: コミュ feed 先頭 8 件のメディアを
    `ExpoImage.prefetch(thumbedUrl(u, 480), 'memory-disk')` で先読み。
    **`ProgressiveImage` の `thumbWidth` (=480) と URL を一致させて cache hit させるのが要点** (ズレると prefetch が無駄になる)。

- **hydrate 並列化**: `app/_layout.tsx` の `Promise.allSettled([...])` で起動 hydrate を並列化。
- **`forceReady` floor**: 旧 500ms safety。hydration 同期化済の今は ~300ms に短縮余地あり (★要 throttled 回線で login flash 確認)。
- 認証セッションの「急に login に戻る」系は別ノート [[認証・セッション]]。

### 2. バンドルサイズ (web) の最適化

- **lucide barrel の全 icon 同梱を撲滅 — ✅解消済 (最大の web 改善)**
  - 旧: web entry JS に lucide-react-native の **全 ~1540 icon** が同梱されていた。
  - 対策: `babel.config.js` の自作 plugin **`lucideDeepImportsPlugin`** が
    `import { Foo } from 'lucide-react-native'` を
    `import Foo from 'lucide-react-native/dist/esm/icons/<kebab>'` の **per-icon deep import** へ一括書換。
  - **production のみ適用** (`...(isProduction ? [lucideDeepImportsPlugin] : [])`)。dev/test は無影響 (Metro cold start を遅くしない)。
  - 旧名エイリアスは `LUCIDE_ALIAS` map で吸収 (例: `Home→house` / `AlertTriangle→triangle-alert` /
    `CheckCircle2→circle-check-big` / `MoreHorizontal→ellipsis` / `Edit3→pen-line` / `BarChart3→chart-column` など 11 個)。
  - `type LucideIcon` の type-only import は触らず TS が elide。ソース (`constants/icons.ts` 等) は無変更 (tsc は barrel 型を見るので OK)。
  - 実測: entry **7.2MB→5.64MB (-22%, gzip 1.46→1.31MB)**。
  - 🔴 **Metro 罠 (重要):** `babel.config.js` を変えても `.metro-cache/` は**自動で無効化されない**。
    変更後は `geek-v4/.metro-cache/` を削除 (or `expo start --clear`) してから build しないと
    旧変換が残り「効いてない」ように見える。**実際に 1 回目の build は古い cache で 7.2MB のままだった**。

- **フォント — ⚠️★残課題 (要検証)**
  - Noto Sans JP フル TTF 6 weight = **26MB**、Inter 9 weight ~2.8MB が `dist/` に emit される。
  - **ただし first paint はブロックしない**: `useAppFonts` に 100ms `forceFallback` +
    `app/+html.tsx` に system-font cascade があるため。改善の主目的は帯域削減と JP 表示の安定化。
  - 改善案 (★未着手): web は Google Fonts woff2 (`css2` + `unicode-range` subset)、
    または使用 weight だけ deep import。**要検証 = `font-family` 名整合** (壊すと表示崩れ)。
  - テーマ/フォント周りの設計は [[i18n・テーマ・デザインシステム]]。

- **console strip**: production build は `babel-plugin-transform-remove-console` で `console.*` を除去。
  **ただし `exclude: ['error', 'warn']`** = realtime 等の `console.warn` は本番でも残る (debug 用、PII は別途 Sentry redact)。
  CI 環境で plugin が install されない事故に備え `require.resolve` で囲み、無ければ silently skip。

### 3. データ取得 (往復削減) の最適化

- **リロード鮮度 (新着がリロードで出ない) — ✅解消済**
  - 真因: `app/(tabs)/feed.tsx` に focus 時の `invalidateQueries(['feed'])` が**実在しなかった**
    (`useFeed.ts` のコメントは "ある前提" だが無かった) + `staleTime 30s` + `refetchOnWindowFocus:false`。
  - 対策: `feed.tsx` に `useFocusEffect` → `invalidate(['feed'])` を追加。feed/コミュ feedQuery を `staleTime:0`、検索 discovery も focus-invalidate。
  - 実コード照合: `feed.tsx:395` 付近に `useFocusEffect` + `lastFeedFocusRefreshRef` で
    「短時間の連続 focus は skip」する throttle 付き。**live push ではなく「戻る/リロードで page1 取り直し」** (ユーザー要望どおり)。

- **検索 discovery を 7 往復 → 1 RTT に集約 — ⚠️★未deploy / 要手動適用**
  - migration `0113_get_discovery_payload.sql` (SECURITY DEFINER STABLE) が
    `{hot, recommended, rising, official, my_community_ids}` を 1 json で返す。
  - 安全策: 本番 `get_feed_page (0107)` を厳密コピー (可視性述語 + **匿名 author_id マスク** + IDOR gate + limit clamp)。詳細は [[匿名性設計と de-anon ホール]]。
  - client: `lib/api/discovery.ts` (RPC→per-shelf 自動 fallback) / `hooks/useDiscovery.ts`。
    **kill-switch `EXPO_PUBLIC_DISCOVERY_RPC=0`** で fallback 固定。
  - ⚠️ **0113 は本文を返す SECURITY DEFINER → 適用前レビュー必須 + Supabase SQL editor で手動適用** (Netlify は migration を流さない)。未適用でも client は fallback で完全動作。
  - 同様に discovery 各 fetch を `withApiTimeout(8s)+retry:1` で必ず data-or-empty に解決 (ソケット停止時の無限スケルトン化を防止)。

- **ホーム feed の往復削減 — ★未着手 (flag-gated OFF で用意)**
  - `get_home_feed (0114)` で posts + いいね/反応/コミュを 3→1 往復にする RPC。
    **既定 OFF**、`EXPO_PUBLIC_HOME_FEED_RPC='1'` で有効化 (★`=== '1'` 判定。`!== '0'` でコピペすると意図せず ON になる事故)。
  - Value Model 個人化フィードも `EXPO_PUBLIC_FOR_YOU_FEED_RPC='1'` で有効化 (0139-0141 適用 + pg_cron 後)。→ [[フィード・ランキング・レコメンド]]。

- **押下 prefetch + cache seed (spinner 撲滅) — ✅実装済**
  - `AnonPostCard.tsx` の `handleOpenDetail` がカードタップ瞬間に詳細画面の 2 cache
    (`['post', id]` と `['feed-page', userId, stableKeyFor([id])]`) を**カードが既に持つデータでシード**。
  - `setQueryData((prev) => prev ?? seed)` で**既存 (より新しい) cache を上書きしない**。詳細画面は mount 時 cache hit で即描画、background revalidate で鮮度維持。
  - feed-page key は `useFeedPage` と完全一致させる (`[prefix, userId ?? 'anon', stableKeyFor([id])]`)。コミュ一覧→詳細・BBS スレも同型 prefetch。

- **`['post-communities-of']` の遅延 fallback 化 — ✅**
  - 通常は `fullPost.communities` (get_feed_page RPC 由来) を再利用し HTTP を撃たない。
    `enabled` は RPC kill-switch/error/RLS deny で undefined のときだけ true。単純削除は回帰があるため不採用。
- React Query 全体設定 (staleTime/gcTime/persist) は [[State管理 (Zustand・React Query)]] と [[Zustand・React Query ベストプラクティス]] 参照。

### 4. レンダリング (60FPS) の最適化 — ✅継続中

フィードカード (`AnonPostCard.tsx`) は **feed の全 post で大量 mount** されるため、ここが 60FPS の主戦場。

- **`memo` + カスタム `arePropsEqual` (`AnonPostCard.tsx:1231`)**
  - post / liked / reactions / communities / 各 handler ref を 1 つずつ比較し、
    関係ない変化では再 render を skip。**handler ref は親 (`feed.tsx` の `handlersByPostId`) が安定化している前提**。

- **親の handler 安定化 (`feed.tsx`)**
  - `handlersByPostId` を `useMemo` で `postIds` (= 安定 ID セット) を deps に構築。
  - **`postsRef` パターン**: `const postsRef = useRef(posts); postsRef.current = posts;`。
    handler のクロージャは `postsRef.current.find(...)` で最新 post を逆引きするので、
    background refetch で `posts` 参照が変わっても (中身が同じなら) **全 handler が再生成されない** (stale closure にもならない)。
  - `postIdsHash = useMemo(posts.map(p=>p.id).join('|'))` で ID 集合の同一性を安定化 → 下流 `useQuery/useMemo` を無駄に再評価させない。

- **Zustand は必ず selector で subscribe (destructure 禁止)**
  - `AnonPostCard` 例: `useLanguageStore((s) => s.lang)` / `useAuthStore((s) => s.user?.id)` のように **必要 field だけ購読**。
    丸呼びすると他フィールド変更で全カードが再 render。未読バッジは `useUnreadCount` の `select` で件数(number)まで絞る narrow selector 化済。
  - 設計原則は [[State管理 (Zustand・React Query)]] / [[Zustand・React Query ベストプラクティス]]。

- **style の参照安定化 (reconciliation コスト削減)**
  - 静的 inline を避け module-level `StyleSheet.create` (factory `makeStyles(C)` を `useMemo([C])` でテーマ毎再生成、同テーマでは同一参照)。
  - props/state 依存の style だけ `useMemo` 化 (`containerStyle`, `bodyTextStyle`, `listContentStyle` 等)。
    JSX inline object/array は毎 render 新参照になるので memoize する。

- **sub-component の memo 化**: `SingleMediaItem` / `ReactionPill` / `QuotePostMiniLoader` を `memo` 化し、
  `.map()` 内の `onPress={() => ...}` anonymous arrow を `useCallback` + props bind で排除。
  `ThreadRow` / `NotificationRow` も memo 化済。

- **Reanimated worklet で UI スレッド実行 (React state を介さない)**
  - カードの press feedback (`pressLift` shared value + `interpolateColor`) は **shared value 駆動**なので
    100 件 mount でも cheap (React の再 render を起こさない)。
  - `useNativeDriver` 相当: アニメは Reanimated 3 worklet で UI スレッド実行し JS スレッドを塞がない (60FPS 死守)。
  - reduce-motion 時は worklet 側で `transparent` 固定 (アニメ停止)。詳細は [[React Native・Expo パフォーマンス最適化]]。

- **FlashList (`@shopify/flash-list`) のチューニング (`feed.tsx`)**
  - `estimatedItemSize`: 旧 300 → 520 → **現 640** (実測 P50/P75 寄り)。**過小だと overscan バッファが小さく fast scroll で blank が出る**。
    text-only post の下限が ~520、メディア付きは 650-700px。
  - `drawDistance`: 旧 250 → **600** (メディアカード実寸 500-700px に対し過小で空白セル/位置ズレが出ていた)。
  - `getItemType` で `'ad'`/`'post'` を分けて recycle pool を分離。`keyExtractor` は一意 id (ad は `__ad` prefix + index)。**`key={i}` は禁止**。
  - `AnonPostCard.tsx` の `mediaItemBase` に `minHeight` 相当 + `alignSelf:'center'` で web recycled cell の 0-height を回避。

- **アスペクト測定パイプライン (`AnonPostCard.measureAspect`)**
  - 画像の自然比は `RNImage.getSize(thumbedUrl(url, 240))` で測る。**240px サムネで測るので帯域 1/9 + decode コスト削減** (旧 720px)。
  - モジュールレベル `_aspectCache` (Map) に **TTL 1h + size cap 500 件の LRU 風 trim**。同 URL は 2 回目以降ゼロコスト。詳細画面の初期 state にも seed してレイアウトシフトを消す。
  - 同時実行は `_MAX_CONCURRENT=3` でキュー制御 (一斉 getSize で詰まらせない)。
  - 🔴 測定 URL は **必ず `resize=contain` 既定** (height 無し)。`resize=cover` を渡すと縦横が化けて「横長が縦長セルで出る」。詳細は [[画像・メディアパイプライン]]。

- **`useFeatureFlag` の useMemo 化**: 旧は毎 render `flags.find` + FNV ハッシュ (カード 1 枚で 3 回) → useMemo 化。
- **タブ切替の即時化**: `app/(tabs)/_layout.tsx` の `lazyPreloadDistance 1→2` で
  feed から search/community を first paint **後**に背景 mount (IG/YouTube 風)。

## 注意点・地雷

- **★ 未 deploy / 未適用が多い (silent degrade の罠)**
  - discovery 集約 (0113) / home feed RPC (0114) / for-you (0139-0141) は **migration 手動適用が必須**。
    Netlify は migration を流さない。**未適用でも client は fallback で動く = 「効いている」と誤認しやすい**。
  - 適用前に必ず実機確認 (特に like→count の反映、匿名マスクの維持)。SECURITY DEFINER RPC はレビュー必須。
  - 反映確認の方法は [[運用 — デプロイ・プレビュー・本番反映確認]]。

- **🔴 Metro cache は babel/config 変更で自動無効化されない** (再掲)。`babel.config.js` を変えたら
  `.metro-cache/` 削除 or `--clear`。これを忘れると「最適化が効いていない」と誤診断する。
  通常 dev は `--clear` 無し (2nd start 30-60s)、cache 汚染疑い時のみ `--clear` (cold 185s)。CLAUDE.md §2 参照。

- **i18n unit test の RN 非依存性を壊さない**
  - `lib/i18n.ts` → `languageStore` を import。`stores/languageStore.ts` (や `adPreferencesStore`) の**トップで
    `import { Platform } from 'react-native'` や `lib/storage` (MMKV) を import すると**、純ユニットテスト
    (`i18nBrand`/`toastDuration` 等) の import 連鎖に react-native が混入し
    「Cannot use import statement outside a module」で suite が落ちる。
  - → これら store は **AsyncStorage のまま** (パフォ目的の同期 MMKV 化は見送り)。「速くしよう」で踏む地雷。

- **native の offline/reconnect 検知は no-op (★未解決)**
  - `@react-native-community/netinfo`/`expo-network` 未導入のため、RQ `onlineManager` は native で恒久 online 固定。
    `refetchOnReconnect:'always'` は native で不発。`networkMode` を変えても直らない (真因は dep 不在)。
    有効化は EAS native rebuild が必須 (OTA 不可) → **要ユーザー判断**。web は既定 window listener で機能。

- **「効いていない最適化」と「偽の前提」だったもの**
  - QueryClient `networkMode`/`structuralSharing` 明示 = 偽の前提 (v5 既定で十分、native は online 固定)。⏭️skip。
  - Expo Router web `asyncRoutes` = route group で export 失敗の既知バグ + 白画面リスク。真の hotspot を縮めない → ⏭️skip。
  - `scripts/fix-html.mjs` は依然 build 未配線で OG/PWA/preload が本番未適用。**繋ぐ場合は cache-first SW を絶対に外す** (古い shell 残存事故) → [[運用 — デプロイ・プレビュー・本番反映確認]]。

- **「画像がフル解像度で重い」は誤り**: フィード画像は既に `ProgressiveImage thumbWidth=480` でサムネ化済。
  「フル解像度を出している」という監査指摘は誤診だった。重さの真因は別 (waterfall / 測定 URL)。

- **緑ビルド ≠ 速い・安全**: type-check/lint 0 + 精読でも「安全」と断言しない。
  実測 (build サイズ / `image-size` / preview の `naturalWidth`) と code-review を通す。→ [[地雷・落とし穴 総覧]]。

- **★残る最大の feed 体感初速問題 (未着手):** feed cold open が
  **posts → get_feed_page RPC + fetchCommunitiesForPosts の 3 段ウォーターフォール** (postIds 依存で直列)。
  landing なのに boot prefetch されていない (コミュ tab だけ prewarm)。
  → boot で default feed prefetch + 1 ページ目の like/reaction を posts query に埋める案 (要 migration か onSuccess kick)。
