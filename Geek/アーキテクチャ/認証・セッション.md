# 認証・セッション

GEEK (geek-v4) の認証は **Supabase Auth (PKCE flow)** をベースに、クライアント側は `stores/authStore.ts` (Zustand) と `app/_layout.tsx` の redirect ガードで構成される。このノートは「**投稿しようとした瞬間に急に login / onboarding に戻る**」という最重要バグの真因と対策を中心に、session のライフサイクル全体を記録する。

> 関連: [[アーキテクチャ概要]] / [[State管理 (Zustand・React Query)]] / [[データ層・Supabase・RLS・マイグレーション運用]] / [[匿名性設計と de-anon ホール]] / [[地雷・落とし穴 総覧]]

---

## 概要

- **認証方式**: メール + パスワード。Supabase Auth (GoTrue)、`flowType: 'pkce'`。
- **session 永続化キー**: `geek-v4-auth` (`AUTH_STORAGE_KEY`、`lib/supabase.ts`)。
  - Web: `localStorage` を Promise ラップ (`webStorage`)。SSR / 例外時は in-memory map にフォールバック。
  - **Native: SecureStore (iOS Keychain / Android Keystore) で暗号化保存** (`nativeSecureStorage`)。
    - ★ **CLAUDE.md §4 の「SecureStore は使わず localStorage/AsyncStorage」という記述は古い**。実コードは security 監査 (平文 JWT の物理盗難対策) で SecureStore へ移行済み。chunk 分割 (1800 byte/chunk + `__meta` にチャンク数) で 2KB 上限を回避している。
- **トークン**: access_token は約 1 時間で失効。refresh_token は長期間有効。`autoRefreshToken: true` で自動更新。
- **アクター**:
  - `stores/authStore.ts` — session lifecycle / signIn / signUp / signOut / hydrate / profile merge を一手に持つ唯一の store。
  - `app/_layout.tsx` の `RootLayout` — auth state を見て login / onboarding / feed へ redirect する。
  - `lib/supabase.ts` — client 生成 + storage adapter + `readPersistedSession()` fallback。
  - `lib/resilient.ts` — 401 検知時に `unauthorizedHandler` を発火 (authStore が登録)。
- **画面**: `app/(auth)/` (login / signup)、`app/onboarding/` (初回 6 画面ウィザード)。
- **核心メッセージ**: **「token 失効 = 即 signOut」ではない。** access token が切れても refresh token が生きていれば継続できる。あらゆる「session 喪失シグナル」を受けたら、まず控えた token で復活を試し、本当に失効しているときだけ `user: null` にする。これが「急に login に戻る」を撲滅する全対策の共通設計。

## 仕組み・設計

### 1. 起動時の hydrate (`authStore.hydrate`)

`app/_layout.tsx` の `useEffect` が `Promise.allSettled([hydrateAuth(), ...])` で起動時 1 回だけ呼ぶ。流れ:

1. **listener 登録** (`registerAuthListener` / `registerStorageListener`) — 何があっても先に登録する。
2. **`setUnauthorizedHandler` 登録** — 401 を検知したら即 signOut せず refresh を試す (後述)。
3. **safety timer = 13 秒** — 返らなければ `{ user: null, hydrated: true }` で起動を継続。
   - ★ なぜ 13s か: 最悪逐次コスト `getSession(3s) + refreshSession(6s) + buildUser(3s) = 12s` を上回る値。
     **7s では「refresh は成功したのにタイマーが先に user:null を確定 → false logout」が起きていた** (各 await は個別 timeout 済なので実際に 13s 超ハングはしない)。UI は別途 `forceReady` 500ms で先に出る。
4. **`getSession()` を 3 秒で race** — web で稀に内部 lock/refresh が stall して**永遠に返らない**ことがある (backend 健全・token 有効でも発生)。stall 時は `readPersistedSession()` で localStorage/SecureStore から **session を直接読んで** 起動継続 (login 画面への張り付き防止)。
5. session があれば **token expiry を check**:
   - `expiresAt * 1000 < Date.now() + 60_000` (期限切れ or 60s 以内) なら **`refreshSession()` を試す** (F7)。成功すれば継続、失敗時のみ `signOut()` + `detachAllChannels()` (F9)。
   - 期限内なら `buildUser(authUser)` で profile を merge。

```ts
// authStore.ts hydrate — token 期限切れ/間近でも即 signOut しない
if (expiresAt && expiresAt * 1000 < Date.now() + 60_000) {
  const res = await withTimeout(supabase.auth.refreshSession(), 6000);
  if (res && !res.error && res.data.session?.user) refreshedUser = res.data.session.user;
  // refreshedUser が取れなければ初めて signOut + detachAllChannels (F9)
}
```

### 2. `onAuthStateChange` listener (`registerAuthListener`) — F1 の本丸

Supabase auth-js は内部イベントを発火する。全イベントを明示処理し、**有効 session が来るたび `lastKnownTokens` に access/refresh token を控える**。

| event | 処理 |
|---|---|
| `INITIAL_SESSION` | 何もしない (return) |
| `TOKEN_REFRESHED` | `supabase.realtime.setAuth(新 access_token)` で **realtime に新トークンを再注入** (Audit E#4: 旧 token のまま RLS-gated postgres_changes が silent 配信停止する事故を防ぐ) |
| `PASSWORD_RECOVERY` | ログのみ (reset flow 中) |
| `SIGNED_IN` / `USER_UPDATED` | `buildUser` して `set({ user })` |
| **`SIGNED_OUT`** | **`signOutInFlight`(=明示 signOut) なら素直に `user:null`。それ以外は `tryRecoverSession()` で復活を試す (F1)** |

#### F1 (最重要): auth-js の自律 SIGNED_OUT

> auth-js は `getUser` / `refreshSession` / visibilitychange 復帰の**失敗時に、内部で自律的に `SIGNED_OUT` を発火し、その前に storage を消す**。

旧コードは SIGNED_OUT 分岐で無条件に `set({ user: null })` していた → refresh token がまだ生きていても login に飛んでいた。しかもこの経路は `resilient` の `unauthorizedHandler` も hydrate の expiry check も通らない **別経路** (自前 refresh コードを素通りする)。

対策 = `tryRecoverSession()`:

```ts
// storage は SIGNED_OUT より前に消されるため、引数なし refreshSession では復活できない。
// 控えた refresh_token を setSession に「明示的に」渡すのが要点。
const res = await withTimeout(supabase.auth.setSession(lastKnownTokens), 6000);
if (res && !res.error && res.data.session?.user) { /* 復活成功 → set({ user }) */ }
else { lastKnownTokens = null; set({ user: null }); } // 本当に失効
```

- ★ **`refreshSession()` (引数なし) は storage 削除済みで効かない。`setSession({access_token, refresh_token})` に控えた token を渡すのが鍵。**
- ★ **二重復活ガード必須** (`recoverInFlight`): setSession 失敗が再び SIGNED_OUT を呼ぶため、再入を防ぐ。
- 復活した user が凍結/制限なら復活させない (`checkAccountState` で signIn と一貫)。

### 3. cross-tab storage listener (`registerStorageListener`) — F5 (Web 限定)

`window.addEventListener('storage', ...)` で他タブの `geek-v4-auth` 消失を検知。**即 logout せず `tryRecoverSession()` を呼ぶ**。token rotation の一瞬の remove や別タブの一過性 `_removeSession` の巻き添えで誤発火するため、控えた token で復活を試してから落とす。

### 4. 401 ハンドラ (`setUnauthorizedHandler`)

`lib/resilient.ts` が API エラーを `looksLikeUnauthorized` で判定し `unauthorizedHandler` を発火。authStore はここで「**即 signOut せず `refreshSession()` で回復を試し、refresh も失敗したときだけ `signOut()`**」を登録する。

```ts
setUnauthorizedHandler(async () => {
  if (!get().user) return;
  const res = await withTimeout(supabase.auth.refreshSession(), 6000);
  if (res && !res.error && res.data.session?.user) return; // user 維持
  await get().signOut();
});
```

- ★ **実態の注意**: memory によれば offline queue (`useOfflineQueueProcessor`) は `executeAction` が `.error` を捨てて throw しないため 401 → signout に到達しない。つまり **resilient の unauthorizedHandler は本番では実質ほぼ未到達** (唯一の自動 signout 経路だが空振り気味)。実際の login 戻りは大半が F1 経由だった。

### 5. `buildUser` と onboarded 判定 — F3

`buildUser(authUser)` は profile (`nickname, onboarded, phone, account_state`) を **3 秒 timeout** で取得し merge。**ログインを絶対に詰まらせない**設計。

- profile fetch が timeout / miss なら `onboarded` を **MMKV/localStorage の同期キャッシュ** (`geek-v4-onboarded:<userId>`) から復元 (`getCachedOnboardedSync`)。これが無いと cold start で profile timeout 時に古い `false` を返し **再オンボループ** になる。
- ★ **F3 の核心**: profile timeout で `onboarded` が **`undefined`** になり得る。`_layout` で `!user.onboarded` と書くと undefined を falsy 扱いして `/onboarding` へ誤バウンスしていた → **`user.onboarded === false` の厳密比較** に修正。真の新規ユーザー誘導は `signIn` の `next` が担保。

### 6. redirect ガード (`app/_layout.tsx`) — F4

`RootLayout` の `useEffect([user, authHydrated, segments, router])` が redirect を担う。

```ts
// F4: redirect 判定は authHydrated を直接見る。ready (=forceReady 500ms 含む) で判定すると
//     hydrate 完了前 (user=null) に !user 分岐で login へ誤バウンスしていた。
if (!authHydrated || segments.length < 1) return;
const inAuth = segments[0] === '(auth)';
const inOnboarding = segments[0] === 'onboarding';
const inResetPassword = (segments as readonly string[])[1] === 'reset-password';

if (!user) { if (!inAuth) router.replace('/(auth)/login'); }
else if (inResetPassword) return;                 // reset 中は留まる
else if (user.onboarded === false) { if (!inOnboarding) router.replace('/onboarding'); } // F3
else if (inAuth || inOnboarding) router.replace('/(tabs)/feed');
```

- ★ **F4**: redirect 判定の基準を `ready` (描画ガード用、forceReady 500ms を含む) から **`authHydrated`** に変更。`ready` 基準だと hydrate 完了前 (有効 session 復元前 = user=null) に login へ誤バウンスしていた。**描画ガード `if (!ready) return null` は ready のまま残す** (黒画面 safety を維持)。
- `reset-password` 中は session が確立していても画面に留める (recovery トークンの exchange → updateUser → signOut → login フローを完走させるため)。
- `segments` の `useSegments()` を依存に入れることで遷移ごとに再評価される。

### 7. signIn / signUp / signOut

- **signIn** (`authStore.signIn`): client 側 rate limit (`checkRate('login')`) → `signInWithPassword` (10s timeout) → エラーは **`error.code` と message の両方で分類** (GoTrue のバージョン/ロケール差で message が変わっても死導線にしない)。成功時 `next: user.onboarded === true ? 'feed' : 'onboarding'` を返す (undefined は安全側で onboarding)。
- **signUp**: rate limit → `signUp` (12s) → email confirmation 設定に応じ 3 分岐 (auto-logged-in / confirm メール送信 / error)。session が返らなければ短い backoff で 3 回 `signInWithPassword` を再試行し伝播遅延を吸収。
- **signOut** (`authStore.signOut`): `signOutInFlight` の single-flight lock。先に `set({ user: null })` + `lastKnownTokens = null` (明示 logout なので復活させない) → `supabase.auth.signOut()` → `detachAllChannels()` → offline queue / draft / block / onboarded cache を全クリア (`clearAllUserData`)。
  - ★ `signOutInFlight` を立てることで、直後に飛んでくる SIGNED_OUT が `tryRecoverSession` に渡って**ログアウトしたはずの user を復活させる race を防ぐ** (security)。
- **setUser**: onboarding 完了時など。`onboarded === true` なら同期キャッシュにも焼く (再オンボループ防止)。
- **refreshProfile**: session 中の account_state 変化を検知 (現状 `checkAccountState` は常に null = ハードブロック廃止、suspended でもログイン許可し in-app banner で通知)。

### 8. AppState による auto-refresh 制御 (`app/_layout.tsx`)

- **Native**: `AppState` change で background/inactive 時に `supabase.auth.stopAutoRefresh()` + `realtime.disconnect()`、active 復帰時に **`startAutoRefresh()` を `realtime.connect()` より先に**呼ぶ (即時 tick で期限間近 token を catch-up → stale JWT での 401 を防ぐ)。OS が JS スレッドを凍結し `setInterval` が止まるための明示制御。
- **Web**: `visibilitychange` で realtime を切断/再接続 (auth は library 任せ)。

## 注意点・地雷

### 確定済みの真因 (F1–F9) — すべて master 反映済 (PR #139)

| ID | 真因 | 対策 |
|---|---|---|
| **F1** ★最重要 | auth-js が getUser/refresh/visibility 失敗時に**自律 SIGNED_OUT** を発火し storage を消す。authStore が無条件 `user:null` 化していた | TOKEN_REFRESHED/SIGNED_IN で token を控え、SIGNED_OUT (非明示) では `setSession(控えた token)` で復活を試す。二重復活ガード必須 |
| F3 | profile timeout で `onboarded=undefined` → `!user.onboarded` が undefined を falsy 扱いし onboarding へ誤バウンス | `user.onboarded === false` の厳密比較 |
| F4 | redirect が `ready` (forceReady 500ms 含む) 基準で hydrate 完了前に login へ誤バウンス | redirect 判定を `authHydrated` 基準に。描画ガードは `ready` のまま |
| F5 | cross-tab storage 消失 (rotation の一瞬 remove 等) を即 logout 扱い | F1 と同じ復活経路へ (Web 限定) |
| F6 | Native SecureStore の setItem が delete-before-write で非アトミック → rotation 書込中断で session 永久消失 | 新 chunk 書込 → meta (コミット点) → 旧掃除 の**非破壊順** + `keychainAccessible: AFTER_FIRST_UNLOCK` + 失敗 warn |
| F7 | 起動直後の最初の API が期限切れ目前 token で 401 | hydrate で期限切れ + 60s 以内も refresh 対象に |
| F9 | hydrate の refresh 失敗枝で bare signOut が channel detach を通らず残留 channel が後続 401 を二次誘発 | `detachAllChannels()` を明示 |

### 反証 (原因ではなかった)

- **F2 無実**: `createPost` の `supabase.auth.getUser()` は /user 送信前に auto-refresh するので access token 期限切れ単独では落ちない。
- offline queue は `.error` を捨てて throw しないので 401→signout に到達しない (= **unauthorizedHandler は実質未到達**)。
- Native 起動時の `AsyncStorage.removeItem('geek-v4-auth')` は SecureStore のキー空間 (`geekv4_..._meta/__c0`) と別なので無罪 (= 平文残骸の cleanup であって session を消していない)。

### 症状の出やすい条件

- **タブ非アクティブ / アプリ復帰の直後の最初の操作**で発生しやすい (auto-refresh ticker は可視タブ/foreground のみ稼働するため)。
- サーバ時刻ズレで `EXPIRY_MARGIN_MS = 90s` 相当を割る場合も。

### 実装上の落とし穴

- **module-scope の `let` lock** (`listenerRegistered` / `storageListenerRegistered` / `signOutInFlight` / `recoverInFlight` / `lastKnownTokens`) は **全 store インスタンスで共有**される。テストは `afterEach` で `_resetAuthLocks()` を呼ぶこと。**hot-reload 後に `listenerRegistered=true` のまま残ると新 store が listener を再登録しなくなる** (開発中の既知 tradeoff)。
- `withTimeout` は **rejection と timeout を区別せず両方 null を返す**。呼び出し元は「null = 喪失とみなしてリカバリ」の安全側に倒す設計。詳細が必要なら p 側で catch して Sentry へ。
- **Sentry に PII を渡さない** (`app/_layout.tsx`)。`setSentryUser(user.id)` は **id のみ**。logout 時 `clearSentryUser()` で id leak を防ぐ。
- **getSession() の web stall** は backend 健全でも起きる。`readPersistedSession()` fallback (`lib/supabase.ts`) を絶対に外さない。

### ドキュメント差分 (要注意)

- ★ **CLAUDE.md §4 の Auth 行「SecureStore は使わず localStorage/AsyncStorage」は実コードと不一致**。Native は SecureStore (chunked, AFTER_FIRST_UNLOCK) を使用している。CLAUDE.md の方が古い。
- `checkAccountState` は現在常に `null` を返す **dead-code 状態** (ハードブロック廃止、suspended は in-app banner)。コールサイトの `if (stateError)` 分岐は将来「法的 ban」復活用に意図的に残してある (除去せず本体差し替えで機能する構造)。
