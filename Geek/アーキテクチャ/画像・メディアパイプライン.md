# 画像・メディアパイプライン

GEEK(geek-v4)の画像・動画の **アップロード前処理** と **表示時サムネ変換** の実態。
この領域は 2026-05〜06 に「真っ黒アイコン」「横長写真が縦長に化ける」「web で無音アップロード失敗」など
事故が **連続多発** したので、特に厚く・正確に記録する。CLAUDE.md §5.9 / §5.10 と §11 地雷表が一次ソース。

二大コンポーネント:
- **アップロード**: `lib/image.ts`(前処理) → `lib/media.ts`(Storage upload)。EXIF strip + magic-byte 検証 + JPEG/向き正規化。
- **表示**: `lib/utils/imageUrl.ts` の `thumbedUrl()`(Supabase render endpoint へURL変換)→ `ProgressiveImage` / `FeedMediaGrid` が描画。

---

## 概要

| 段階 | 入口関数 | ファイル | 何をするか |
|---|---|---|---|
| ピック後の前処理 | `prepareImageUpload(uri, opts)` | `lib/image.ts` | EXIF strip / magic-byte 検証 / リサイズ+JPEG再エンコード / 5MB上限 / web⇄native分岐 |
| EXIF除去+リサイズ単体 | `stripExifAndResize(uri, opts)` | `lib/image.ts` | `[{rotate:0}]` で EXIF向きを物理適用しタグを1にリセット。長辺のみ縮小で比率維持 |
| web 表示用ダウンサンプル | `makeWebPreviewDataUrl(uri)` | `lib/image.ts` | webのピッカー直後に `data:` URL化(blob: revoke回避 + EXIF焼き込み) |
| web Canvas crop | `cropImageOnWebCanvas(input)` | `lib/image.ts` | ImageManipulatorを bypass し純Canvasで crop/resize/rotate。HEIC黒画像をthrowで検出 |
| 画像 Storage upload | `uploadPostImage(uri, userId)` | `lib/media.ts` | `prepareImageUpload`→ `posts-media` bucketへ→ public URL返す |
| 動画 validate | `validateVideoSource(asset)` | `lib/media.ts` | size(100MB)/MIME(mp4/mov/webm/m4v)チェックのみ(transcodeなし) |
| 動画 Storage upload | `uploadPostVideo(uri, userId, validated)` | `lib/media.ts` | web=Blob / native=FormData で `posts-media` へ |
| 表示用サムネURL変換 | `thumbedUrl(url, width, opts)` | `lib/utils/imageUrl.ts` | object URL → render endpoint。**height無しはcontain既定**(潰れ罠回避) |
| 正方形 crop サムネ | `squareThumbedUrl` / `iconThumbedUrl` | `lib/utils/imageUrl.ts` | width=height で投げる。アバター=cover / アイコン=contain |
| web srcset | `thumbedSrcSet(url, widths)` | `lib/utils/imageUrl.ts` | DPR/viewport別の responsive srcset |
| 画像描画(2層crossfade) | `<ProgressiveImage>` | `components/ui/ProgressiveImage.tsx` | blurhash→本画像 fade、ken-burns、render endpoint変換、web lazy |
| 複数画像カルーセル | `<FeedMediaGrid>` | `components/feed/FeedMediaGrid.tsx` | 固定高さ×アスペクトで横スクロール。aspect測定 |

bucket 一覧: `avatars` / `community-icons` / `posts-media`(投稿写真・動画)。
保存形式: 投稿画像は **最大 1600x1600 / JPEG 85%**(`stripExifAndResize` 既定)。`cacheControl: '31536000'`(1年・命名がimmutableなので長期cache可)。

## 仕組み・設計

### アップロード前処理 `prepareImageUpload` (`lib/image.ts`)

戻り値: `{ blob: Blob | FormData; mime; ext; size; uri }`。**web は `Blob`、native は `FormData`**(両方とも Supabase Storage の `upload()` に渡せる)。
処理の流れと分岐:

1. **`data:` URL ショートパス**(`uri.startsWith('data:')`) ─ cropper や `makeWebPreviewDataUrl` が返した `data:image/...;base64,...` は **既に処理済み**(512x512 or 1600px JPEG, EXIFなし)なので、manipulator を再度呼ばず直接 Blob 化する。
   - 理由: ① Safari は `blob:` URL を勝手に revoke するが `data:` は revoke されない。② manipulator の二重呼び出しは iPhone Safari Canvas で memory error / 空Blob を返す。
   - base64 を `Uint8Array` 化 → サイズ上限/**下限(200byte)**チェック → magic-byte 二重確認(`data:` の MIME ヘッダ偽装防止)→ Blob 化。
   - 下限チェックは「1x1透明/黒JPEG(~140byte)が通って真っ黒アイコンになる」事故対策。
2. **`stripExifAndResize(uri)`** で JPEG化(EXIF除去)+リサイズ。これで URI が `ph://`/`asset://` → `file://` に正規化される(FileSystem/FormDataが読める形式へ)。
3. **web 分岐: 多段フォールバック**(iPhone Safari の "Load failed" 根絶)
   - Path 1: 最適化済み `cleanUri` を `fetch` → `blob.size===0` なら "Safari blob URL may be revoked" として失敗 → Path 2 へ。magic-byte 判定必須。
   - Path 2: 元 `uri` を直接 `fetch`(HEIC/巨大画像で manipulator が落ちるケース)。magic失敗時は `blob.type` → 最終 `image/jpeg` に fail-soft。
   - 全滅 → "HEIC形式の写真を選んでいないか確認してください" と詳細pathErrors付きで throw。
4. **native 分岐: FormData 経路**
   - `getInfoAsync` でサイズ check(取れなくても続行)。
   - 先頭 16byte だけ `readAsStringAsync(Base64, position:0, length:16)` で読み、magic-byte で MIME 判定(大ファイルでもメモリを食わない)。
   - `FormData.append('', { uri, name, type })` で構築。**フィールド名は必ず空文字 `''`** ─ Supabase storage-js は FormData の最初のファイルエントリ(name=`''`)を upload 対象にする。`'file'` 等にすると取り出せず失敗。
   - 旧実装(`Uint8Array` body)は Android okhttp で確実に動かず実機アップロード失敗の原因だった → RN idiomatic な FormData multipart に変更。

### `stripExifAndResize` の EXIF/向き/アスペクト処理

```ts
// lib/image.ts: 空 actions だと iOS で EXIF回転タグが残る罠を回避
const first = await ImageManipulator.manipulateAsync(uri, [{ rotate: 0 }], {
  compress: quality, format: ImageManipulator.SaveFormat.JPEG,
});
// 長辺が上限超のときだけ「長辺のみ」を指定 (短辺は比率維持で自動計算)
const resizeAction = w >= h ? { resize: { width: maxWidth } } : { resize: { height: maxHeight } };
```

- **`[{rotate:0}]` を渡す理由**: `manipulateAsync` に **空の actions** を渡すと iOS では EXIF回転タグを保持したまま返す場合がある。保存後JPEGを Supabase transform でサムネ化すると transform が EXIF を無視して raw ピクセル向きで返し、フィードの aspect 測定が縦横逆になる。`{rotate:0}` で EXIF回転を **物理適用** しタグを1(回転不要)にリセット →「display向きのピクセル + EXIF=1」になる。
- **アスペクト維持(2026-06修正)**: 旧実装は `resize:{width:max, height:max}` と両軸を渡していた。expo-image-manipulator は **両軸指定だと比率を無視して exact resize** する → 16:9等の非正方写真が 1:1 に潰れる + 512pxアイコンが1600pxに upscale される無駄。修正後は「まず無resizeでJPEG化して寸法取得 → 長辺が上限超のときだけ長辺のみ指定」。長辺以下なら再エンコードのみ(歪みなし・upscaleなし)。

### web 専用ヘルパ

- **`makeWebPreviewDataUrl(sourceUri, maxEdge=1024, quality=0.8)`**: 高解像度画像を表示用にダウンサンプルした `data:` URL を返す。
  - **EXIF orientation 焼き込み**: `<img>` + `canvas.drawImage` は EXIF orientation を **適用しない**(生ピクセルのまま描く)→「横で撮った写真が縦で投稿される」。一方 `createImageBitmap(blob, { imageOrientation: 'from-image' })` は向き適用済み bitmap を返し width/height も適用後の値になる。これを drawImage すれば「正しい向きが焼き込まれEXIFは無い」JPEG になる。未対応エンジン(古いSafari/WebView)では従来 `<img>` 経路へフォールバック。
  - `ImageBitmap` は `close?.()` で GPU/メモリを早期解放。同期 canvas 処理の前に `await Promise.resolve()` で 1回 yield し複数画像処理中も React に再描画の隙を与える。
- **`cropImageOnWebCanvas`**: ImageManipulator を bypass し純Canvas で crop+resize+rotate。HEIC/巨大画像/Safari Canvas memory error を回避。結果は `data:` URL(revokeされない)。

### web 黒画像/無音失敗ガード(★事故対策の核心)

`makeWebPreviewDataUrl` / `cropImageOnWebCanvas` の両方に **2種類のガード** が入っている:

1. **HEIC silent-onload ガード**: iOS Safari / WKWebView(TikTok等 in-app browser)は HEIC や巨大画像で `<img>.onload` を発火させつつ `naturalWidth=0` を返すケースがある。この状態で drawImage しても透明canvasになり toDataURL が無音で `'data:,'` を返す → 0byte相当JPEGが Supabase に上がり「真っ黒アイコン」事故。`if (!natW || !natH || natW < 1)` で明示検出して throw。
2. **toDataURL 無音失敗ガード**: WebView Canvas メモリ不足や tainted canvas で `'data:,'` / 極端に短い data URL が返る。有効なJPEG data URLは base64でほぼ確実に200文字以上 → `if (!out || !out.startsWith('data:image/') || out.length < 200)` で throw。
   - throw すると caller(`image-cropper.tsx handleNext`)の catch で FileReader fallback → 最終的に sourceUri に巻き戻すロジックに繋がる。

### magic-byte 検証

`detectImageTypeFromBytes(buf)` が先頭12バイトから実MIMEを判定(拡張子を信用しない):
- JPEG `FF D8 FF` / PNG `89 50 4E 47 0D 0A 1A 0A` / GIF `GIF87a|GIF89a` / WebP `RIFF????WEBP`。
- `safeExtension(mime)` は許可リスト(`jpg/jpeg/png/webp/gif`)外なら `jpg` にフォールバック(パストラバーサル防止)。
- web は `Blob.slice(0,12).arrayBuffer()`、native は base64→`Uint8Array`(`atob` は Hermes RN0.76+ で利用可、無ければ `Buffer`)。

### `lib/media.ts` ─ Storage upload

- **存在理由**: 既存バグで `app/post/create.tsx` がピッカーのローカルURI(`file://`/`blob:`)を直接 `media_urls` 列に保存していた → **投稿者本人にしか画像が見えない**(他デバイスでURI解決不能)。これを必ず直す唯一の入口。
- `uploadPostImage`: `prepareImageUpload`(5MB/1600px/q0.85)→ path `${userId}/${uuid()}.${ext}` → `upload()` を `withApiTimeout(..., 30_000)` で包む → `getPublicUrl()` の HTTPS URL を返す。`upsert:false`。
- `uploadPostVideo`: validateで通った mime/ext を使う。**web の罠**: storage-js は Blob の `.type` を multipart content-type に採用し `options.contentType` を **無視** する。`blob:` 由来Blobは `.type` が空/`application/octet-stream` になりがちで bucket の video MIME 許可に弾かれる → `raw.type === mime ? raw : new Blob([raw], { type: mime })` で検証済みmimeに包み直す。timeout は 180秒(5分は放置しすぎ↔90秒は遅回線で大容量を誤失敗、の折衷)。
- `uuid()`: `crypto.randomUUID()` 優先、無ければ Math.random フォールバック(path に user_id 接頭辞があるので衝突しても被害最小)。
- **責務は1ファイル単位**。部分成功(1枚目OK/2枚目失敗)は呼び出し側(createPost flow)が「ここで止める」判断する。

### 表示サムネ `thumbedUrl` (`lib/utils/imageUrl.ts`) ★ 最重要・潰れ罠

Supabase Storage の画像変換 endpoint(`/storage/v1/render/image/public/`)に URL を書き換えて `width`/`quality`/`resize`/`format` で動的サムネ化する。`/object/public/` を含まない(外部CDN等)URLや、既に `width=` 付きURLはそのまま返す(二重変換防止)。

```ts
// imageUrl.ts: ★ この既定を絶対に壊さない
const format = opts.format ?? 'webp';   // JPEG比 25-30% 軽量
const resize = opts.resize ?? (opts.height ? 'cover' : 'contain');
//                              ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
//   height 無しで cover を渡すと Supabase が高さを元のまま残し画像が潰れる
```

- 既定: `width=720`, `quality=75`, `format=webp`。
- **🔴 鉄則: `resize=cover` は `height` とセットでしか使わない**(下記「注意点・地雷」で詳述)。
- `squareThumbedUrl(url, size)` = `thumbedUrl(url, size, { height: size })`(=cover で正方形 center-crop。アバター/集合写真コミュアイコン向け)。
- `iconThumbedUrl(url, size)` = `thumbedUrl(url, size, { height: size, resize: 'contain' })`(ロゴ全体を見せる。余白に地色 icon_color。`CommunityIcon.tsx` が contentFit="contain" で表示)。
- `thumbedSrcSet(url, [320,480,720,1080])` = web responsive。`sizes='(max-width: 720px) 100vw, 720px'`。

### `ProgressiveImage` (`components/ui/ProgressiveImage.tsx`)

- 二層crossfade: 下層 blurhash placeholder → 本画像 `onLoadEnd` で 480ms `easeOutQuart` fade-in。blurhash は 0.8 で 200ms hold → 240ms で 0 へ溶ける(sharpが完全に乗ってから抜く=pop-in/flicker回避)。
- ken-burns: 本画像 mount時 scale `1.04→1.0` を 600ms ease-out。
- **★ `contentFit='contain'` 時は `useKenBurns=false`**(scale 1.0固定)。理由: contain は画像がコンテナ端まで広がるため scale>1.0 + `overflow:hidden` で左右がクリップされる(=「フィード画像が一部しか映らない」事故の真因の片割れ)。cover はもともとクロップ前提なので問題ない。
- 内部で `thumbedUrl(uri, thumbWidth)` を呼ぶ(既定 `thumbWidth=480`。旧720はフィード1列~360-500pxには過剰で~120KB余分)。`cachePolicy="memory-disk"`、`recyclingKey={resolvedUri}`、`priority` ヒント、web のみ `IntersectionObserver`(rootMargin 200px)で lazy。
- error 時は blurhash を留めて中央に淡い `Icon.image`(透明にしない)。

### `FeedMediaGrid` (`components/feed/FeedMediaGrid.tsx`)

- Threads流の **横スクロールカルーセル**。各写真を固定高さ `H`(= `min(cap, max(180, min(winW,600)*0.58))`、cap=web320/native300)・幅 `H×アスペクト` にして「写真全体」を `contentFit="contain"` で等倍表示(cover でズーム/クロップしない)。セル枠=画像比なので灰色帯が出ない。
- aspect は呼び出し側が計測済みなら props で渡す(feed/詳細=ちらつきなし)。未指定なら `RNImage.getSize(thumbedUrl(it.uri, 240))` で自前計測し `measured` に保持(再計測しない)。サムネ失敗時はオリジナルURLで再試行。
- **測定URLも `thumbedUrl(240)`(height無し=contain既定)を使う**のが必須 ─ ここで cover を渡すと測定値が縦長に化ける(下記地雷参照)。

## 注意点・地雷

### 🔴 [事故多発] `resize=cover` を `height` 無しで渡すと画像が潰れる(2026-06)

**最も重要。CLAUDE.md §5.10 / §11 が繰り返し警告している領域。**

Supabase render endpoint に **`width` だけ + `resize=cover`** を渡すと、**幅だけ要求値に縮小して高さは元のまま** 返す。実測(元 1179x866 横長):

| URL | 結果 | aspect | 判定 |
|---|---|---|---|
| `width=480&resize=cover` | 480x866 | 0.55(水平つぶれ) | ❌ |
| `width=240&resize=cover` | 240x866 | 0.28(激細) | ❌ |
| `width=480&resize=contain` | 480x353 | 1.36 | ✅ 正 |

これが ①フィード写真が縦長/細いセルで出る(`getSize` が 0.28 を測る)②表示画像そのものが水平に潰れる(ProgressiveImage が潰れた bitmap を取得)の **両方の真因**。アバター/アイコン(正方形512)も width のみ cover では 120x512 に潰れていた。

- **対策(実装済み)**: `thumbedUrl` は `opts.resize ?? (opts.height ? 'cover' : 'contain')` で **height未指定なら contain を既定**。contain は width だけでも比率維持で比例縮小する。**この既定を壊すな。**
- 正方形 crop が欲しい時(avatar/icon)は `squareThumbedUrl`/`iconThumbedUrl`(=width=height)を使う。**生 `thumbedUrl(url, w, {resize:'cover'})` を height無しで書かない。**
- crop/fit は **expo-image の `contentFit`** に任せる。Supabaseからは比率を保ったcontainソースをもらう。Supabase側でcover cropしようとしない。

### 🚫 「写真が縦長/潰れる」を見ても EXIF を疑う前にこれ(↑)を疑え

「縦長/潰れる」を見ると EXIF orientation を疑いがちだが、**アップロードは `prepareImageUpload` で EXIF strip 済み**で原因ではないことがほとんど。実際の真因は上記 `resize=cover`(2026-06)。`[{rotate:0}]` や `format=origin` を足しても **直らない**(どちらも render endpoint を通る限り cover の潰れは残る)。検証は推測でなく **`image-size` で本番の実画像URLを実測**(object=正 / render+cover=潰れ なら原因確定)。preview なら `img.naturalWidth/Height` と `getBoundingClientRect()` を eval して `cellAspect ≈ natAspect` を確認。

### 🔴 [事故多発] web の `blob:` URL は Safari が revoke する → 無音アップロード失敗

webでは `ImagePicker.launchImageLibraryAsync` が返す `asset.uri` は `blob:` URL。**iPhone Safari はこれを勝手に revoke** するため、生のまま `prepareImageUpload` に渡すと web fetch が空Blob/"Load failed" になり **無音でアップロード失敗**(症状=「選んでも反映されない」)。

- **対策パターン**: ピック時に **web のみ** `makeWebPreviewDataUrl(asset.uri, maxEdge, 0.85)` で `data:` URL化してから渡す。`data:` は `prepareImageUpload` の revoke耐性ショートパス(`uri.startsWith('data:')`)を通る。
- `maxEdge` は uploader が `prepareImageUpload` に渡す `maxWidth/maxHeight` と一致させる(album/cover/post いずれも **1600**, q0.85, アスペクト比保持)。変換失敗時は生URIにフォールバックして退行なし。
- native(`Platform.OS !== 'web'`)は `asset.uri` のまま(挙動不変)。
- **適用済み**: `app/post/create.tsx`(~L261)/ `app/post/comment.tsx`(~L106)/ `app/settings/profile-edit.tsx` pickCover(58f2532)/ `app/mypage/photo/add.tsx`(PR #157)。
- **Why 重要**: 同じ blob: revoke バグが **カバー画像 → アルバム写真 と連続で再発** した。既存upload経路を踏襲せず新規に書くと必ず踏む。**web画像uploadを実装/レビューするときは、ピッカーの生 `asset.uri` を直接 `prepareImageUpload`/storage に渡していないか必ず確認** し、渡していたら web 分岐で `makeWebPreviewDataUrl` を挟む。
  > ※ この memory は 3 日前の point-in-time 観測。file:line(~L261 等)は現コードとズレている可能性あり ─ 適用箇所は実コードで再確認すること。

### 「真っ黒アイコンが upload される」事故(HEIC silent-onload)

iOS Safari/WKWebView は HEIC/巨大画像で `<img>.onload` を発火させつつ `naturalWidth=0`。透明canvasから `'data:,'` が生成され 0byte相当JPEGが上がる。`makeWebPreviewDataUrl`/`cropImageOnWebCanvas` の `naturalWidth/Height` ガードと、`prepareImageUpload` の data URL 下限200byteチェックの **両方** でブロックしている。どれかを外すと再発しうる。

### ken-burns × contain クリップ(フィード画像が一部しか映らない)

`ProgressiveImage` で `contentFit="contain"` のとき scale 1.04→1.0 アニメ + `overflow:hidden` が左右をクリップしていた。修正: contain 時は `useKenBurns=false` で scale 1.0固定。**`FeedMediaGrid` は contain で使うので、ProgressiveImage 側のこのガードに依存している**。ken-burns を contain でも有効化し直すと再発する。

### native のグラデ文字・動画 transcode は未対応(★)

- 動画は `validateVideoSource` で **size/MIME チェックのみ**。**transcoding はしない**(100MB上限はそのままアップロード)。再エンコードや解像度ダウンは未実装。
- `prepareImageUpload` の native 経路で magic 判定が落ちた場合は `image/jpeg` と仮定して **fail-soft で続行**(UX優先)。MIME が実際に異なると後段で弾かれうる。

### FormData フィールド名は空文字固定(画像・動画共通の規約)

`fd.append('', { uri, name, type })` の **キーは必ず `''`**。Supabase storage-js は FormData をそのまま内部fetchに渡し、サーバは name=`''` の最初のファイルエントリを取り出す。`'file'` 等にすると取り出せず upload 失敗。`lib/image.ts`(画像)と `lib/media.ts`(動画)で同じ規約。

### TS narrowing の小罠

`Uint8Array → BlobPart`: TS5.x の lib.dom.d.ts では `Uint8Array<ArrayBufferLike>` が `ArrayBufferView<ArrayBuffer>` と互換でないため `new Blob([bytes as BlobPart], ...)` と明示 narrowing が必要。
