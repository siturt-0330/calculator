## 概要

**Geek（ギーク）** = 「好きを、匿名で、安心して続ける」**完全匿名型の趣味/推し活専用 SNS**。

- **1 コードベースで iOS / Android / Web** を配信する React Native (Expo) アプリ。Web は <https://geekboard.netlify.app/>。
- バックエンドは **Supabase**（PostgreSQL + RLS + Realtime + Edge Functions + Storage）。サーバーコードはほぼ DB（RLS + SQL 関数/トリガ）+ Edge Function に寄せ、**専用 API サーバーを持たない**（BaaS 構成）。
- 状態は **2 系統**に明確分離: クライアント状態 = **Zustand**、サーバー状態 = **TanStack Query (React Query) v5**。
- ルーティングは **Expo Router 4**（file-based）。`app/` のファイル構造がそのまま URL/画面ツリーになる。
- 開発スタイルは **開発者 1 人 + Claude ペアプロ**。バージョン 4.0.0 / アクセント `#7C6AF7`（紫）。

### コアバリュー（設計判断の軸 — SPECIFICATION.md 第2章）

| 価値観 | 実装への波及 |
|---|---|
| 完全匿名性 | 投稿者名を出さない。コメントは「N番目の人」表記。→ [[匿名性設計と de-anon ホール]] |
| 構造的安全性 | 信頼スコア・AI コンテンツ検査・タグフィルタで荒らし抑制。→ [[Admin Console (運営管理)]] |
| タグ中心設計 | フォローでなく **タグ** でコンテンツを繋ぐ。広告も個人IDでなくタグでターゲティング |
| プライバシー尊重 | 個人ID を広告主に送らない。Sentry も PII redact |

### レイヤ俯瞰（データの流れ）

```
[ app/ 画面 (Expo Router) ]
        │  画面は supabase を直接叩かない
        ▼
[ hooks/ (useFeedPage / useLike ...) ]  ←─ React Query (server state / cache)
        │                                      ↑ Zustand (client state: auth/settings/lang/...)
        ▼
[ lib/api/*.ts (Supabase クエリ層) ]  ──→ withApiTimeout / resilient (timeout + retry + 401 signOut)
        │
        ▼
[ lib/supabase.ts (client) ]  ──REST/Realtime──→  Supabase (PostgreSQL + RLS + Edge Functions)
        ▲
        └─ lib/realtime.ts (attachChannel singleton) が Realtime購読をrefCount管理
```

## 仕組み・設計（具体ファイルパス付き）

### 1. 起動シーケンスと最上位 provider — `app/_layout.tsx`

このファイルが **アーキテクチャの中枢**。読むとほぼ全体像が掴める。やっていること:

- **Reanimated worklet polyfill**（ファイル冒頭）: Web build で `_WORKLET` / `_getAnimationTimestamp` / `__reanimatedLoggerConfig` 等が未定義 → white screen になる問題を、`import 'react-native-reanimated'` の **前** に globalThis へ shim 注入して回避。native では実 runtime が上書きするので無害。
- **Provider ツリー**: `ErrorBoundary > GestureHandlerRootView > SafeAreaProvider > PersistQueryClientProvider > BottomSheetModalProvider > View(key=theme) > Stack`。
- **QueryClient 設定**（`const qc = new QueryClient(...)`）: `staleTime 30s` / `gcTime 2h`（旧24h→監査で短縮、AsyncStorage使用量約75%削減）/ `refetchOnWindowFocus:false` / `refetchOnReconnect:'always'` / `refetchOnMount:true`（永続cache復元データは必ず stale 扱い → cold open で最新を取り直し「古い画面が残る」を解消）/ `retry:1` / mutations は `retry:0`。
- **永続化**: `PersistQueryClientProvider` + `createAsyncStoragePersister(AsyncStorage)`、`maxAge 2h`、`buster: PERSIST_BUSTER='geek-rqcache-v1'`（cache shape を変えたら bump して旧 dehydrated cache を破棄）。
- **hydrate 並列化**: `Promise.allSettled([hydrateAuth, hydrateSettings, hydrateLang, hydrateTagFilter, hydrateAdPrefs])`。theme は同期 (MMKV/localStorage) なので別途即時。
- **redirect 制御**: `useEffect([user, authHydrated, segments])` で `(auth)` / `onboarding` / `(tabs)` を出し分け。`authHydrated` を直接見る（`ready` は forceReady 500ms を含むため hydrate 完了前の誤バウンスを防ぐ）。reset-password 中は feed に攫われないようガード。詳細 → [[認証・セッション]]。
- **起動時 prefetch**: auth 確定直後にホームフィード（`feedQueryKey` で同一 key）とコミュニティタブを裏で先読み → 画面 mount 時に spinner を挟まず即表示。先頭メディアは `ExpoImage.prefetch(thumbedUrl(url,480))` で温める。
- **forceReady safety**: hydrate/font が詰まっても 500ms で強制描画（黒画面で固まるのを防ぐ）。`if (!ready) return null`。
- **テーマ remount**: `<View key={resolvedTheme}>` で theme 切替時に navigation tree 全体を remount（193+ ファイルが static な `import { C }` を直参照しているため、パレットを破壊的に書き換えて全再描画）。→ [[i18n・テーマ・デザインシステム]]。
- **Realtime ライフサイクル**: AppState/visibilitychange で background 時に `supabase.realtime.disconnect()` + `auth.stopAutoRefresh()`、復帰で再接続（バッテリー/通信削減）。user-scoped channel は `RealtimeRoot`（`useUserChannel`）に集約。→ [[Realtime]]。
- **再 render storm 対策**: store は **全 destructure せず selector で subscribe**（`useAuthStore((s) => s.user)`）。`syncStaticPaletteIfChanged` は theme が実際に変わった時だけ実行。

### 3. 技術選定の "なぜ"（CLAUDE.md §4）

| 領域 | 選定 | 理由 |
|---|---|---|
| UI | React Native 0.76 + Expo SDK 52 | 単一コードで iOS/Android/Web。New Architecture + Hermes |
| Router | Expo Router 4 (file-based) | typed routes、`(tabs)` で底辺タブ保持 |
| Styling | NativeWind + Tailwind | RN で Tailwind syntax、`design/tokens.ts` で C/SP/R/SIZE を中央管理 |
| State (client) | **Zustand 5** | 1 store 1 file。**selector で subscribe**（destructure 禁止 = re-render 連鎖の元）|
| State (server) | **TanStack Query v5** | staleTime 30s / gcTime 2h / focus refetch off / 永続化 |
| Storage (永続) | MMKV(native)/localStorage(web) | `lib/storage.ts` で同期 wrap。cold start から async を排除 |
| Auth | Supabase Auth (**PKCE**) | session key=`geek-v4-auth`。native は SecureStore（後述）、web は localStorage |
| Realtime | Supabase Realtime + **1 channel/1 table** | 未登録 table を chain bind すると channel 全死。`attachChannel` で refCount |
| HTTP | Supabase JS + `withApiTimeout` | PostgrestBuilder に AbortController 無 → race で timeout 必須 |
| List | @shopify/flash-list | recycler 方式で大量投稿でも軽い |
| Image | expo-image + ProgressiveImage | EXIF strip → magic-byte → 再エンコード |
| アニメ | Reanimated 3 + Moti | worklet で UI スレッド実行（60fps） |
| Monitor | Sentry + PostHog | Sentry は PII redact 必須 |

詳細な「良いコード観点」での技術論は → [[アーキテクチャパターン (FSD・Atomic・Clean)]] / [[State管理 (Zustand・React Query)]] / [[React Native・Expo パフォーマンス最適化]]。

### 4. 4 つの主要パターン（codebase に馴染む鍵 — CLAUDE.md §5）

1. **DB アクセスは `lib/api/*` に集約**。画面/component から `supabase.from(...)` を直接叩かない。GET でリトライしたいなら `resilient(fn, {name, timeoutMs, retries})`、それ以外（特に mutation = 副作用ありはリトライ禁止）は `withApiTimeout(promise, label, ms)`。401/JWT expired は `resilient` 内の `unauthorizedHandler` が自動 signOut を発火。
2. **React Query cache 書き戻し**は `lib/cacheUpdates/feedPagePatcher.ts`。v5 で `setQueriesData` の partial-match が散発的に伝播しない事象を観測済 → `getQueriesData` で exact key を列挙して `setQueryData` 逐次。optimistic は snapshot→apply→settledでinvalidate→error時revert。queryKey は配列・先頭要素を prefix、大量 ID は `stableKeyFor` で djb2 hash 化。
3. **Realtime は `attachChannel` 経由・1 channel 1 table**。→ [[Realtime]]。
4. **Zustand は selector 購読**。`try{}catch{}` は `swallow('scope', e)` に置換（breadcrumb 残す）。

### 5. バックエンド = Supabase（サーバーコードの置き場）

- **RLS が全 table に設定済**。クライアントには **anon key のみ**（`service_role` key は絶対に置かない）。認可ロジックは RLS ポリシー（`create policy`）+ SQL 関数 (SECURITY DEFINER) が担う。→ [[データ層・Supabase・RLS・マイグレーション運用]]。
- **マイグレーション**: `supabase/migrations/00XX_*.sql` を番号順に積む。**既存ファイル編集禁止**（idempotency 崩壊）、revert は新ファイル。実測 **0001〜0145（144 ファイル）**。
- **主要トリガ/RPC**: `handle_new_user` / `update_likes_count` / `refresh_account_state` / counter 再計算 `reconcile_community_counters()` / フィード 1 ページ集約 `get_feed_page`(0041) / `get_home_feed`(0114) / `get_for_you_feed`(0141)。
- **Edge Functions**（実測 `supabase/functions/`）: `check-content`（投稿前モデレーション・fail-secure）/ `send-push`（Web Push・失効endpoint自動削除）/ `calculate-trust-score` / `suggest-caption` / `automod-eval` / `og-fetch` / `og-image` / `quality-scorer` / `rank-blender` / `search-explainer` / `_shared`。

## 注意点・地雷

> ⚠️ **CLAUDE.md / memory が実コードと食い違う点**（このノート作成時に実コードで確認した差分。**コードを正とする**）:

- **🔴 Auth ストレージ: native は SecureStore（CLAUDE.md §3/§4 の「SecureStore は使わず localStorage/AsyncStorage」は stale）。** 実際の `lib/supabase.ts` は **native = SecureStore（Keychain/Keystore）に chunk 分割保存**、web = localStorage。セキュリティ監査で「平文 JWT が物理盗難で漏れる」を塞ぐため移行済。`keychainAccessible: AFTER_FIRST_UNLOCK`、書き込みは delete-before-write を避け meta をコミット点にする（途中失敗で session 全損を防ぐ）。起動時に旧 AsyncStorage の `geek-v4-auth` を cleanup（既存ユーザーは一度再ログイン要）。`getSession()` stall 対策に `readPersistedSession()` fallback あり。→ [[認証・セッション]]。
- **Realtime 同時 channel 上限は 12（CLAUDE.md §5.3 の「20」は stale）。** `lib/realtime.ts` の `MAX_CONCURRENT_CHANNELS = 12`（Free tier の per-connection 上限を 10 user で食い潰さないため 20→12 に縮小）。eventsPerSecond も 5（`lib/supabase.ts`、1000 並行ユーザーの fanout 過剰抑制）。
- **マイグレーション番号は 0145 まで（memory「0001-0035」/ CLAUDE.md「0001-0043」は古い）。** Admin Console / レコメンド / プラットフォーム機能（引用・クロスポスト・下書き・ブロック・spam）の追加分（0118-0145）を含む。これらの一部は **Supabase SQL エディタで手動適用が必要**で、未適用でも fallback で動く＝**silent degrade の罠**。→ [[データ層・Supabase・RLS・マイグレーション運用]] / [[地雷・落とし穴 総覧]]。
- **Edge Function の顔ぶれが変わった。** memory/旧 CLAUDE.md にある `verify-official-url` / `send-notification` は現在の `functions/` に無く、代わりに `automod-eval` / `og-fetch` / `og-image` / `quality-scorer` / `rank-blender` / `search-explainer` がある。

> ⚠️ **アーキテクチャ横断の地雷**（CLAUDE.md §11/§14・memory）:

- **store 全 destructure → 全画面 re-render**。必ず `(s) => s.field` の selector。`_layout.tsx` で settings 16 フィールド destructure が onboarding 中の「かくかく」を起こした実例あり。
- **1 channel に複数 table の `.on()` チェーン禁止**。publication 未登録 table の binding が CHANNEL_ERROR を起こすと channel 全死（event 一切来ない）。
- **EXPO_PUBLIC_* は static 参照のみ inline される**。`process.env[key]` の動的アクセスは Expo が inline しない。既定 OFF flag は `=== '1'` で判定（`!== '0'` をコピペ流用すると意図せず既定 ON 事故）。
- **Web の env は `geek-v4/netlify.toml` に置く**（repo root の netlify.toml は使われない）。env 欠落で「白画面」事故の常連。→ [[運用 — デプロイ・プレビュー・本番反映確認]]。
- **既存 migration ファイルを編集しない**（revert は新ファイル）。
- **画像表示は `thumbedUrl` の `resize=cover` を height 無しで使わない**（幅だけ縮小し高さ据え置きで潰れる）。→ [[画像・メディアパイプライン]]。
- **memory は point-in-time**。`★` 付き（未適用 migration・実機確認待ち等）は鵜呑みにせずコードと現状を確認。
