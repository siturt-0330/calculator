# 関数型プログラミングパターン

> JavaScript / TypeScript / React における関数型プログラミング (FP) の中核パターンを、一次情報 (React 公式 "Rules of React"、TypeScript Handbook、Immer、現役エンジニアの論考) に基づいてまとめ、**実際の geek-v4 コードベース**へどう落ちているかを具体ファイルで紐付けたノート。
>
> 関連: [[素晴らしいコードとは — 総論]] / [[Clean Code 原則]] / [[SOLID 原則と React 実践]] / [[テスタビリティとテスト戦略]] / [[State管理 (Zustand・React Query)]] / [[フィード・ランキング・レコメンド]]

FP は「全部 FP で書け」という宗教ではない。**純粋関数 (pure function)・不変性 (immutability)・合成 (composition)・副作用の分離 (effect isolation)** という 4 つの道具を、I/O や UI のような本質的に副作用的な世界の「内側のコア」に適用し、外殻 (副作用) と明確に線を引く——これが現代の実務的 FP。React/Redux/TanStack Query/Immer はすべてこの思想で設計されている。

---

## 定義・原則

### 1. 純粋関数 (Pure Function)
2 つの条件を**両方**満たす関数:

1. **同じ入力 → 同じ出力** (referential transparency / 参照透過性)。`new Date()`・`Math.random()`・`Date.now()` を内部で呼ぶと破れる。
2. **副作用を持たない** — 引数や外部変数・DOM・ネットワーク・ストレージを変更しない。

React 公式は「React はすべての component を pure function だと仮定する」と明言する。`y = 2x` のように、x=2 なら必ず y=4。

```js
// pure: 何度呼んでも同じ、外を触らない
function double(n) { return 2 * n; }

// impure: 既存変数を書き換える ← レンダー順で結果が変わる
let guest = 0;
function Cup() { guest = guest + 1; return <h2>guest #{guest}</h2>; }
```

**なぜ重要か** (React 公式が挙げる利点):
- **メモ化が安全** — 入力が変わらなければレンダーをスキップできる (`React.memo` / `useMemo`)。
- **中断・再開・並行レンダー** — React は深いツリーのレンダーを安全に止めて再開できる (Concurrent / `useTransition`)。
- **サーバーレンダー** — 同じ入力が複数リクエストに使い回せる。
- **テスト容易** — モック・セットアップ・後始末が要らない (後述 [[テスタビリティとテスト戦略]])。

### 2. 不変性 (Immutability)
> 「データは生成後に変更しない。変更が必要なら**新しいデータ構造**を返す」

- props / state / context は**読み取り専用**として扱う。直接 mutate しない (`item.url = ...` ✗)。
- 配列・オブジェクト更新は spread / `map` / `filter` / `reduce` で**新しい参照**を作る。
- React は参照同一性 (`Object.is`) で再レンダー要否を判定するため、**mutate すると再レンダーが起きない / 起きてほしくない所で起きる**バグになる。

```js
// ✗ mutate（再レンダーされない・他所に波及）
todos.push(newTodo); setTodos(todos);
// ✓ 新配列（更新関数形式で stale closure も回避）
setTodos(prev => [...prev, newTodo]);
```

TypeScript は `readonly` 修飾子・`Readonly<T>`・`readonly T[]`・`as const` で**コンパイル時に**不変性を強制できる:

```ts
interface Rx { readonly x: number }
const rx: Rx = { x: 1 };
rx.x = 12;                 // ✗ コンパイルエラー
const a: readonly number[] = [1, 2, 3];
a[0] = 101;                // ✗ コンパイルエラー
```

**ローカル mutation は OK** (React 公式が明言): レンダー中に**その関数内で生成した**配列/オブジェクトを mutate するのは安全。外に漏れていないから。

```js
function FriendList({ friends }) {
  const items = [];                 // ローカル生成 → mutate OK
  for (const f of friends) items.push(<Friend key={f.id} friend={f} />);
  return <section>{items}</section>;
}
```

> 注意: JSX に渡した**後**で値を mutate するのも NG (React がもう評価済みかもしれない)。"create separate values" が鉄則。

### 3. 合成 (Composition) と高階関数 (Higher-Order Functions, HOF)
- **HOF** = 関数を引数に取る/返す関数。`map`/`filter`/`reduce`/`sort` は標準 HOF。
- **合成** = 小さい純関数を繋いで複雑な処理を組む。`compose(f, g)(x) = f(g(x))`、`pipe` は逆順。TypeScript なら型安全に合成できる。

```ts
const pipe = <T,>(...fns: Array<(x: T) => T>) => (x: T) => fns.reduce((acc, f) => f(acc), x);
const clean = pipe(trim, toLowerCase, removeDiacritics);
```

- **カリー化 (currying)** = 多引数関数を「1 引数を受けて残りを待つ関数」の連鎖にする。部分適用で設定済みの専用関数を量産できる。
- React では **custom hook の合成** (`useX` が `useY` を呼ぶ)、**HOC** (`withLogger(Component)`)、`children`/render-props がこの思想の現れ。

### 4. 副作用の分離 (Effect Isolation) ★ 最重要の実務原則
純粋にできない処理 (DOM 変更・fetch・ストレージ・ログ・乱数・時刻) は、**レンダーの外**に追い出す:

| 置き場所 | 用途 | 純粋であるべきか |
|---|---|---|
| レンダー本体 / hook 本体 | 計算 (入力→JSX/値) | **必ず純粋** |
| イベントハンドラ | ユーザー操作起点の副作用 (最優先) | 不要 |
| `useEffect` | レンダー後に外界と同期 (**最後の手段**) | 不要 |

```js
// ✗ レンダー中に DOM を触る（副作用）
function Page({ product }) { document.title = product.title; }
// ✓ Effect に逃がす
function Page({ product }) {
  useEffect(() => { document.title = product.title; }, [product.title]);
}
```

非冪等な値もレンダーから締め出す:

```js
// ✗ レンダー毎に違う → 冪等性違反
function Clock() { return <span>{new Date().toLocaleString()}</span>; }
// ✓ Effect でサンプリングして state へ
function useTime() {
  const [t, setT] = useState(() => new Date());
  useEffect(() => { const id = setInterval(() => setT(new Date()), 1000); return () => clearInterval(id); }, []);
  return t;
}
```

## 具体例 (コードブロック)

### a) Reducer は純関数 (state, action) → 新 state
Redux / `useReducer` の reducer は「現在の state を mutate せず**新しい** state を返す」純関数。これが time-travel debug・テスト容易性・楽観更新の土台。

```ts
type State = { count: number };
type Action = { type: 'inc' } | { type: 'dec' } | { type: 'reset' };

const reducer = (s: State, a: Action): State => {
  switch (a.type) {
    case 'inc':   return { count: s.count + 1 };   // 新オブジェクト
    case 'dec':   return { count: s.count - 1 };
    case 'reset': return { count: 0 };
  }
};
```

### b) 判別可能な union (Discriminated Union) で「不正な状態を表現不能に」
TypeScript で FP の代数的データ型 (ADT) に最も近いもの。`kind` で網羅分岐 → switch の exhaustiveness をコンパイラが保証。`null` チェックの散乱を防ぐ。

```ts
type Result<T> =
  | { ok: true; value: T }
  | { ok: false; error: string };

function render<T>(r: Result<T>) {
  if (r.ok) return r.value;   // ここでは value が型として存在
  return r.error;             // ここでは error が型として存在
}
```

### c) Immer — 「mutate するように書いて不変を得る」
ネストが深い更新で spread 地獄になるとき、Immer は Proxy で draft への "mutation" を記録し、**structural sharing** (変更パスのノードだけ shallow copy、無変更ブランチは同一参照のまま) で新 state を生成する。Redux Toolkit の `createSlice`/`createReducer` は内部で Immer を使っており、**reducer 内で `state.x.push(...)` と書いても安全**。

```ts
import { produce } from 'immer';
const next = produce(state, draft => {
  draft.users[3].todos.push({ id, done: false }); // 見た目は mutate、実体は不変
});
// 無変更の users[0..2] は next とも参照が同じ → React の memo が効く
```

> 注意: Immer の draft をクロージャ外へ持ち出して後から触らない。`produce` のコールバック内で完結させる。

### d) 宣言的データ変換パイプライン
命令的ループより `filter → map → sort` の連鎖の方が「何を得たいか」が読める (declarative)。中間で新配列を作るので元は不変。

```ts
const top = posts
  .filter(p => !p.hidden)
  .map(p => ({ ...p, score: score(p) }))
  .sort((a, b) => b.score - a.score);
```

## よくあるアンチパターン

| アンチパターン | なぜダメ | 直し方 |
|---|---|---|
| **props / state を直接 mutate** (`item.tags.push(x)`, `state.count++`) | 再レンダーされない / 他コンポーネントに波及 / Strict Mode で二重実行時に壊れる | spread・`map`・`setState(prev => ...)`・Immer |
| **レンダー中に副作用** (`document.title=`, fetch, `console.log` で外部状態更新) | 冪等でない / 並行レンダーで複数回走る | イベントハンドラか `useEffect` へ |
| **レンダー中に `Date.now()` / `Math.random()`** | 入力が同じでも出力が変わる → メモ化・SSR・hydration 崩壊 | `useState(() => ...)` 初期化 + Effect で更新、または引数で `now` を注入 |
| **`useEffect` を第一選択にする** | データの流れが追えなくなる / 無限ループ / 過剰 fetch | まず「レンダーだけで表現できないか」を考える。Effect は**最後の手段** |
| **巨大な単一関数 (副作用 + 計算混在)** | テスト不能 / 再利用不能 | 純粋なコア (計算) と副作用の殻 (I/O) を**分離**する (functional core, imperative shell) |
| **`.forEach` で外部配列に push して "変換"** | mutation・意図が不明瞭 | `.map`/`.filter`/`.reduce` で新配列 |
| **過度な抽象 (point-free 病・深いカリー化・自作モナド)** | 可読性が逆に落ちる。JS/TS には高階種 (HKT) が無く `fp-ts` 等は学習コスト大 | 標準の `map/filter/reduce` + 小さい純関数で十分。FP は手段、目的ではない ([[可読性・保守性]]) |
| **`as any` で型を握り潰す** | 不変性・網羅性のコンパイル時保証を全部捨てる | `unknown` + 型ガード、discriminated union (geek-v4 は `no-explicit-any: error`) |

## ★Geekへの適用

geek-v4 は「**純粋なロジックは `lib/` に純関数として隔離 → `tests/unit/` で検証 → React/Supabase は薄い殻**」という functional-core / imperative-shell を実際に徹底している。`CLAUDE.md §10` に「**新しい純関数 / 純 reducer を書いたら test も追加**」と明文化済み。具体ファイルで対応を見る。

### 1. 純粋関数 — `lib/` のスコアリング/判定ロジック群
`grep "純粋関数|pure function|immutable"` で 11 ファイルがヒット。代表例:

- **`lib/personalize/profile.ts`** — 冒頭コメントが宣言:「**純粋関数として実装、Storage には触らない**」。`computeProfile(events, now = Date.now())` は **`now` を引数で受ける**(= `Date.now()` を内部で呼ばない冪等化)。イベント配列から嗜好プロファイルを算出する完全な純関数で、I/O は呼び出し側 (hook) の責務。
- **`lib/utils/searchAlgo.ts`** — `textRelevance(text, query)` / `scorePost(...)` / `scoreTag(...)` はすべて純粋スコアリング。`Set` で `seen` 管理し副作用なし。返り値で `reasons: [...new Set(reasons)].slice(0, 3)` と**新配列を返して dedup**。
- **`lib/feed/smartRank.ts`** — `smartScore(p, ctx, mode)` は純粋 (重み × 各シグナルの線形結合)。`smartSort` は `.map → .filter → .sort → .map` の宣言的パイプラインで元配列を mutate しない。
- **`lib/utils/commentCollapse.ts`** — 「**純関数 — supabase / RN を一切 import しない。unit test 容易**」と明記。閾値 (`COLLAPSE_CONCERN_THRESHOLD` 等) を**export して**テスト・UI から参照可能にする = マジックナンバー排除 ([[Clean Code 原則]])。
- 同系統: `lib/utils/voteFuzz.ts` / `qaSort.ts` / `commentBestScore.ts` / `lib/trust/score.ts`。

### 2. 不変性 — cache patch と入力の非破壊
- **`lib/cacheUpdates/feedPagePatcher.ts`** が好例。React Query の cache を patch する `patchFeedPagePost` は**新しい参照の配列**を作って書き戻す:

```ts
const next = rows.map((p) => {
  if (p.id !== postId) return p;            // 無変更 row は同一参照のまま（structural sharing）
  touched = true;
  return typeof patch === 'function' ? patch(p) : { ...p, ...patch }; // 変更 row だけ新オブジェクト
});
if (touched) qc.setQueryData(exactKey, next);
```
無変更 row を**同じ参照で返す**ことで FlashList / memo の再描画を最小化する (= Immer の structural sharing と同じ発想を手書きで実現)。

- **`lib/utils/commentCollapse.ts`** は入力を `readonly T[]` で受け、`run.slice()` でコピーし、`annotateCollapsed` は `comments.map(c => ({ ...c, collapsed }))` と**全件新オブジェクト**。コメントにも「引数の comment は mutate しない (純関数)」と明記。
- **`tests/unit/xorSelection.test.ts`** には不変性そのものを検証するテストがある:

```ts
it('baseline immutable: visiblyPicked does not mutate input', () => {
  const baseline = ['a', 'b'];
  visiblyPicked(baseline, ['c']);
  expect(baseline).toEqual(['a', 'b']);   // 入力が変わっていないことを保証
});
```

### 3. 合成 / 判別 union
- **`smartSort`** / **`searchAlgo`** の `.filter().map().sort()` 連鎖 = 宣言的合成パイプライン。
- **`commentCollapse.ts`** の戻り値 `CollapseGroupItem<T> = { kind: 'single'; comment: T } | { kind: 'group'; comments: T[]; count: number }` は**判別可能 union**。UI 側は `kind` で網羅分岐でき、不正状態 (group なのに count 無し等) を**型で表現不能**にしている。
- **custom hook 合成**: `hooks/` (40+) は小さい hook を組み合わせる FP 的構造 (`useT`, `useDebounce`, `useHaptic` 等を上位 hook が利用)。

### 4. 副作用の分離 (functional core, imperative shell)
geek-v4 のアーキテクチャそのもの:

- **コア (純粋)**: `lib/personalize/profile.ts`・`lib/feed/smartRank.ts`・`lib/utils/searchAlgo.ts`・`lib/utils/commentCollapse.ts` — I/O ゼロ。
- **殻 (副作用)**: `lib/api/*` (Supabase クエリ)・`lib/realtime.ts`・`lib/storage.ts`・hook の `useEffect`。`CLAUDE.md §14` に「**component から `supabase.from(...)` を直接叩かない → `lib/api/*` に通す**」と境界を明文化。
- **副作用の制御点を 1 箇所に集約**: `lib/swallow.ts` は `try{}catch{}` の代替で、例外を握り潰しつつ Sentry breadcrumb を残す「管理された副作用」。それ自身も breadcrumb 書き込み失敗で落ちないよう内側を `try/catch` で固める防御。`CLAUDE.md §14` に「silent な `try{}catch{}` 禁止 → `swallow('scope', e)`」。
- **時刻という副作用の注入**: `lib/utils/date.ts` の `formatRelativeFor(dateStr, lang)` は **lang を引数で受ける純関数版**を基底にし、`formatRelative(dateStr)` だけが `useLanguageStore.getState().lang` という外部状態を読む薄いラッパ。テストは純関数版を直接叩ける。
- **Zustand と FP**: `CLAUDE.md §5.4` の「**selector で subscribe** (全 destructure 禁止)」は「最小の入力だけに依存させる = 不要な再レンダーを防ぐ」FP 的最適化 ([[Zustand・React Query ベストプラクティス]])。

### geek-v4 で守るべき具体ルール (このノートの実践チェックリスト)
- [ ] 新しい計算ロジックは `lib/` に**純関数**で書き、`Date.now()`/`Math.random()` は**引数で注入**する (`computeProfile(events, now)` に倣う)。
- [ ] cache / state 更新は `.map` + spread で**新参照**、無変更要素は同一参照を返す (`feedPagePatcher` に倣う)。
- [ ] 入力は `readonly` で受け mutate しない。`as any` 禁止 (`no-explicit-any: error`)。
- [ ] 分岐が増える戻り値は**判別 union** にして網羅性をコンパイラに守らせる。
- [ ] 副作用 (Supabase / storage / DOM) は `lib/api`・`lib/realtime`・`useEffect`・イベントハンドラへ。コアに混ぜない。
- [ ] 純関数を書いたら `tests/unit/` に**必ずテスト追加** (`CLAUDE.md §10`)。「入力 mutate しない」アサートも入れる (`xorSelection.test.ts` に倣う)。
