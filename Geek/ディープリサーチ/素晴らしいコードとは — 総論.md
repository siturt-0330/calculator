## TL;DR (この 1 枚の主張)

1. **コードは「コンピュータのため」ではなく「人間が読むため」に書く**。実行できるだけのコードは fool でも書ける (Fowler)。素晴らしいコードの第一義は **読み手にとって明白 (obvious)** であること。
2. **品質 = 後から変えるコストが低いこと**。"良さ" は美学ではなく経済性。複雑さ (complexity) は「理解と変更を難しくするすべて」(Ousterhout) であり、それを下げる活動が設計。
3. **正しさ → 意図の明示 → 重複ゼロ → 最小要素**、の優先順位 (Kent Beck の simple design 4 則)。テストで担保された正しさが土台、その上で可読性・DRY・簡潔さを積む。
4. ツール (型・lint・テスト・CI) は **人間の注意力を希少資源として節約する** ための増幅器。判断を置き換えるものではない。
5. 2023-2025 の現実: **AI 生成コードが大量流入**する時代だからこそ、「人間が読んで意図が分かる/テストで守られている/小さく変更できる」という古典原則の価値が**上がっている**(AI コードは長いメソッド・重複ロジック・ハードコード秘密を生みやすい — Codacy)。

## 定義・原則

### 「素晴らしいコード」の作業定義

権威ある複数ソースを統合すると、高品質コードの核は次の属性に収束する (Codacy / TIOBE / 2024 code quality guides):

| 属性                    | 意味                                      | なぜ重要か                                      |
| --------------------- | --------------------------------------- | ------------------------------------------ |
| 可読性 (Readability)     | 命名・小さい関数・低い構文複雑度で「逆エンジニアリングなしに」意図が掴める   | コードは書かれる回数より**読まれる回数が圧倒的に多い**              |
| 保守性 (Maintainability) | 欠陥修正・性能改善・環境変化への適応が容易                   | ソフトのコストの大半は初回開発後に発生する                      |
| 単純性 (Simplicity)      | 必要以上に複雑でない。賢いトリックより素直さ                  | 複雑さは複利で増える (Ousterhout)                    |
| テスト可能性 (Testability)  | unit/integration/e2e で振る舞いを検証でき、回帰を早期検出 | リファクタ・拡張の**安全網**になる                        |
| 一貫性 (Consistency)     | 命名・フォーマット・パターンが揃っている                    | 認知負荷を下げる。CMU SEI 研究で一貫フォーマットは理解度を最大 20% 改善 |
| 堅牢性/信頼性 (Robustness)  | 異常入力・障害に耐え、予測可能に振る舞う                    | 本番の信用に直結                                   |
| セキュリティ (Security)     | OWASP Top 10 等に沿い、悪用可能な欠陥がない            | 「動くコード」と「安全なコード」は別物                        |

> 一言でいうと: **「読んで分かる × 安全に変えられる × テストで守られている」**。残りはこの 3 つに奉仕する細目。

### 4 つの古典的レンズ (出典つき)

#### 1. Fowler — 人間のためのコード / リファクタリング
> "Any fool can write code that a computer can understand. Good programmers write code that humans can understand." — Martin Fowler, *Refactoring*

- **リファクタリング** = 「外部から見た振る舞いを変えずに、内部構造を理解しやすく・変更しやすくする変更」。
- 「良いコード」は一発で書くものではなく、**動かす → 振る舞いを変えずに整える**の反復で**育てる**もの。テストがあるから安全に整えられる(→ [[テスタビリティとテスト戦略]])。

#### 2. Ousterhout — 複雑さとの戦い (*A Philosophy of Software Design*)
- **複雑さの定義**: 「システムの構造のうち、理解と変更を難しくするすべて」。式で表すと **Total Complexity = Σ(各部の複雑さ × その部に費やす時間)** — つまり**頻繁に触る所の複雑さ**ほど重い。
- **複雑さの 2 大原因**:
  - **依存 (Dependencies)**: あるコードが単独で理解/変更できず、他に連動する。
  - **不明瞭さ (Obscurity)**: 重要な情報が**自明でない**(命名・ドキュメント不足)。
- **3 つの症状** (これが出たら複雑さが溜まっているサイン):
  - **Change amplification**: 小さな変更が多数の箇所の修正を要求する。
  - **Cognitive load**: 1 つの変更を行うのに知らねばならない量が多い。
  - **Unknown unknowns**: 何を変えれば正しく動くのか、**変えるべき箇所自体が分からない**(最悪。事故の温床)。
- **深いモジュール (Deep modules)** を作れ: **「インターフェースが実装よりずっと単純」**な部品。逆に **浅いモジュール (shallow)** = 小さくても複雑なインターフェースを露出する部品は、数が増えるほど**全体の複雑さを増やす**(Java IO の薄いクラス群が反例)。
  - 系: **「単純な実装より単純なインターフェースを優先せよ」**。`分割しすぎ ≠ 良い設計`。クラスを増やすこと自体は美徳ではない。
- **情報隠蔽 (Information hiding)** で深さを作る。`temporal decomposition`(処理の**順序**でモジュールを切る)を避け、「そのタスクに必要な**知識**」で切る。
- **戦略的 vs 戦術的**: その場しのぎの "tactical tornado" は速いが設計を蝕む。**「設計を良くしていないなら、たぶん悪くしている」**。少額の投資(1 タスクあたり 10-20% を設計に回す)を継続。
- **エラーは設計で消す (Define errors out of existence)**: 例外を増やすより、**例外が要らない API** を設計する。
- **Design it twice**: 採用前に「**根本的に異なる**2 案以上」を比較してトンネルビジョンを避ける。

#### 3. Kent Beck — Simple Design 4 則 (優先順位つき)
XP 発祥。**上から優先**:
1. **Passes the tests** — まず正しく動く(テストで担保)。
2. **Reveals intention** — 読み手に意図が伝わる(命名・構造で「なぜ」を語る)。
3. **No duplication** — Once and only once。Fowler 曰く「最も**繊細に強力**」な則で、**重複除去こそ良い設計を炙り出す**駆動力。
4. **Fewest elements** — 上 3 つを満たしたら、クラス/関数/型を**最小化**する。
- ポイント: **テストが最優先**だが、可読性 (2) と DRY (3) は相互に磨き合う。「動く」だけでは不十分、「動く・読める・重複ない・最小」で初めて simple。(→ [[Clean Code 原則]])

#### 4. Google eng-practices — コードレビューの基準
- 一義的目的: **「コードベース全体の health が時間とともに改善し続けること」**。
- **完璧なコードは存在しない。"より良い" コードがあるだけ。** レビュアーは「**health を確実に改善するなら、完璧でなくても approve**」する(進捗 vs 質のバランス)。
- レビューで見る軸: **設計(まず全体設計を見る)→ 機能 → 複雑さ(もっと単純にできないか/他人が読めるか)→ テスト → 命名 → コメント → 一貫性**。
- 反 YAGNI: 「**今は要らないが将来要るかも**」で作り込むのを止める。
- 設計判断は**根拠 (underlying principles)** ベースで議論。純粋なスタイルは style guide に従う(好みで殴らない)。

### 横断する小原則 (頭字語の地図)

| 原則 | 中身 | 注意 |
|---|---|---|
| **DRY** | Don't Repeat Yourself — 知識の単一情報源 (SPOT) | 「**見た目**の重複」より「**知識**の重複」を消す。早すぎる共通化は誤った抽象を生む (AHA: Avoid Hasty Abstractions) |
| **KISS** | Keep It Simple — 素直に書く | 賢さは負債。明白さ優先 |
| **YAGNI** | You Aren't Gonna Need It — 未来の仮想要件を実装しない | Ousterhout の strategic とは矛盾しない(設計**余地**は残すが**機能**は足さない) |
| **SOLID** | SRP/OCP/LSP/ISP/DIP — OOP の保守性 5 原則 | React/関数型では「責務分離・依存方向・置換可能性」に翻訳して適用 (→ [[SOLID 原則と React 実践]]) |
| **SRP** | 1 つのモジュールは 1 つの「変更理由」 | 「1 つのことだけする」より「**1 つの理由でだけ変わる**」が正確 |
| **POLA** | Principle of Least Astonishment — 驚き最小 | API/命名は予想通りに振る舞え |
| **Boy Scout Rule** | 来た時より少しきれいにして去る | レビュー/CI と合わせて漸進改善 |

## 具体例 (コードブロック)

### 1. 命名で意図を語る (Reveals intention)

```ts
// ❌ 何を、なぜ計算しているか読み取れない。マジックナンバー。
function f(p: any, c: any) {
  let s = 0;
  for (const t of p.tags) if (c.l.has(t)) s += 1.5;
  return s * 0.5 + (Date.now() - p.t) / 3600000;
}

// ✅ 名前と定数が「なぜ」を語る。型がある。1 つの理由でだけ変わる。
const RECENCY_HALF_LIFE_HOURS = 24;
const LIKED_TAG_AFFINITY = 1.5;

function affinityFromLikedTags(post: Post, likedTags: Set<string>): number {
  let affinity = 0;
  for (const tag of post.tag_names ?? []) {
    if (likedTags.has(tag)) affinity += LIKED_TAG_AFFINITY;
  }
  return affinity;
}
```
読み手は実装を追わずに**名前だけで意図**を理解できる。これが Beck の rule 2、Ousterhout の「obscurity を下げる」。

### 2. 深いモジュール: インターフェース < 実装

```ts
// 浅い (shallow): 呼び手に timeout / race / clearTimeout / Jest open-handle を全部押し付ける
const id = setTimeout(...); const r = await Promise.race([p, ...]); clearTimeout(id); ...

// 深い (deep): 呼び手が知るのは「Promise に timeout を付ける」だけ。
//   複雑さ(race・finally で必ず clear・thenable 対応)は内側に隠れている。
const data = await withApiTimeout(supabase.from('posts').select('*'), 'posts.fetch', 8000);
```
**1 行の単純な呼び口**の裏に、競合処理・リーク防止という実装複雑さが**隠蔽**されている。深いモジュールの教科書例。(実物: `geek-v4/lib/withApiTimeout.ts`)

### 3. 「コメントは why を、コードは what を」

```ts
// ❌ コードを読めば分かることの反復(ノイズ)
i = i + 1; // i を 1 増やす

// ✅ コードからは読めない「なぜ/前提/トレードオフ」を残す
// Supabase render endpoint に width だけ + resize=cover を渡すと高さが元のまま返り
// 画像が水平に潰れる(実測 1179x866→480x866)。height 未指定は contain を既定にする。
const resize = opts.resize ?? (opts.height ? 'cover' : 'contain');
```
高品質コードのコメントは「非自明なトレードオフ・前提・エッジケース」を書き、コードの言い換えはしない (2024 code quality guides / Ousterhout: コメントは intuition/precision を足せ)。実物の罠は [[画像・メディアパイプライン]] と [[地雷・落とし穴 総覧]] 参照。

### 4. エラーを「握りつぶす」のではなく「観測可能にする」

```ts
// ❌ silent: 本当に壊れていても誰も気づけない
try { localStorage.setItem(k, v); } catch {}

// ✅ 本筋は止めない(catch と同じ)が breadcrumb を残す = 後で診断できる
try { localStorage.setItem(k, v); } catch (e) { swallow('storage.set', e); }
```
"防御的に無視" と "事故を隠す" は紙一重。**観測可能性 (observability)** を足すと、同じ「無視」でも品質が変わる。(実物: `geek-v4/lib/swallow.ts`)

### 5. 純粋関数 + テスト = 安全に変えられる

```ts
// 入力→出力が決定的な純関数。副作用なし → そのまま unit test できる。
export function smartScore(p: Post, ctx: RankingContext, mode: RankingMode = 'hot'): number { ... }

// tests/unit/* で境界(tier 2↔3 等)を固定。リファクタ時の回帰をテストが捕まえる。
```
Beck rule 1「passes the tests」を**最も安く**満たせるのは、ロジックを純関数に切り出した時。I/O と判断を分離する設計が**テスト可能性**を生む(→ [[テスタビリティとテスト戦略]] / [[関数型プログラミングパターン]])。実物: `geek-v4/lib/feed/smartRank.ts`。

## よくあるアンチパターン

| アンチパターン | 何が問題か | 対応する原則 |
|---|---|---|
| **God Object / 神クラス** | 1 つの巨大モジュールが何でもやる。複数の理由で変わる | SRP / 深いモジュールではなく**広くて浅い**怪物 |
| **早すぎる抽象 / 過剰汎化** | 「将来のため」に作った設定・フックが誰にも使われず複雑さだけ残す | YAGNI 違反。AHA(急いで抽象化するな) |
| **分割しすぎ (classitis)** | クラス/関数が小さすぎ、薄いモジュールが大量。呼び出し連鎖を追えない | Ousterhout: 浅いモジュールの増殖 = 複雑さ増 |
| **マジックナンバー/文字列** | `0.1` や `'cover'` が文脈なしに散る | 命名された定数 (SPOT) |
| **コピペ重複** | 同じ知識が複数箇所に。1 つ直して他を忘れる | DRY / Once and only once |
| **silent catch** (`try{}catch{}`) | 例外を完全に飲み、本当の障害が不可視に | observability 欠如。`swallow` 化 |
| **コメントでコードを言い換え** | `i++ // increment i`。陳腐化してコードと乖離 | コメントは why を書く |
| **賢すぎるワンライナー** | ネストした三項・ビット技で「動くが読めない」 | KISS / reveals intention |
| **temporal decomposition** | 「最初にこれ、次にこれ」と処理順でモジュールを切る | 知識で切る (情報隠蔽) |
| **boolean trap** | `fn(true, false)` 呼び出し側で意味不明 | enum/オブジェクト引数で意図明示 (POLA) |
| **広い `any` / 型の穴** | TS で `as any` して型安全を自分で壊す | → [[TypeScript 型安全性]] |
| **完璧主義レビュー** | "perfect" を求めて PR を止め続ける | Google: 完璧でなく health 改善で approve |
| **AI 生成の丸呑み** | 長いメソッド・重複ロジック・ハードコード秘密・一貫性欠如を混入 | 上記原則を**AI コードにも**同じ基準で適用 (Codacy) |

## ★Geekへの適用 (この知識ベースの入口)

ここが **Geek (React Native / Expo / TypeScript / Supabase) 文脈での「良いコード」総括**であり、各リサーチノートと Geek 知識ノートへのハブ。

### Geek における「素晴らしいコード」の定義(このリポジトリ固有の翻訳)

このプロジェクトは **「開発者 1 人 + Claude 1 機」** で、**Lean に小さく検証**しながら進める (`geek-v4/CLAUDE.md` §15, §17)。だから良いコードの基準は次のように具体化される:

1. **読み手 = 数週間後の自分 / もう 1 機の AI**。日本語コメントで「なぜ」を残す文化 (CLAUDE.md §6「design decision は日本語で書く」) は、まさに Fowler「人間のためのコード」の実践。
2. **小さく動かして即捨てられる**こと (CLAUDE.md §17-2)。= 深いモジュール + 純関数で**変更コストを下げる**。
3. **過去の地雷を 2 度踏まない** (CLAUDE.md §11 の cheat sheet)。= Ousterhout の "unknown unknowns" をドキュメントで "known" に変換する営み。
4. **green build ≠ 安全**。型/lint 0 + 精読で「安全」と断言しない(監査が 2 HIGH + 8 回帰を捕まえた実例 — [[運用 — デプロイ・プレビュー・本番反映確認]])。= テストと**レビュー**を品質ゲートに据える Google 流。

### React / RN 特有の「良いコード」(総論を Geek 技術へ翻訳)

- **State は selector で購読**。Zustand を全 destructure すると 1 フィールド変更で全 re-render → これは「不要な依存 = 複雑さ(change amplification)」の RN 版。`(s) => s.user` で必要分だけ subscribe。→ [[State管理 (Zustand・React Query)]] / [[Zustand・React Query ベストプラクティス]]
- **サーバ状態は TanStack Query に寄せる**(staleTime/gcTime/optimistic update)。手書きのキャッシュ整合ロジックを各所に**重複させない** (DRY)。cache patch は `feedPagePatcher.ts` に集約。→ [[State管理 (Zustand・React Query)]]
- **60fps を死守する設計**(Reanimated worklet / FlashList recycler)。「動きは本当に必要な時だけ」(CLAUDE.md §15-4) は YAGNI のアニメ版。→ [[React Native・Expo パフォーマンス最適化]] / [[パフォーマンス最適化]] / [[モバイル UX 品質指標]]
- **1 channel / 1 table**(Realtime)。1 つの CHANNEL_ERROR で全死する密結合を避ける = 依存の局所化。→ [[Realtime]]
- **セキュリティを後回しにしない** (CLAUDE.md §15-3): RLS 全 table、service_role をクライアントに置かない、EXIF strip、SSRF 対策。良いコード属性「Security」の実装。→ [[データ層・Supabase・RLS・マイグレーション運用]] / [[匿名性設計と de-anon ホール]]
- **i18n / テーマ / デザイントークンを単一情報源に**(`design/tokens.ts`、`lib/i18n.ts` の `DICT`)= SPOT の UI 版。→ [[i18n・テーマ・デザインシステム]]
