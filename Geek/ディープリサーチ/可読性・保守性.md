# 可読性・保守性

> コードは「**書く時間 < 読まれる時間**」。一度書いたコードは何十回も読み返され、数か月後の自分や他人(=GeekではClaude)が改修する。可読性は美的趣味ではなく、**バグ修正効率・リファクタ成功率に直結する測定可能な品質指標**。
> 関連: [[素晴らしいコードとは — 総論]] / [[Clean Code 原則]] / [[可読性・保守性]]の周辺は [[TypeScript 型安全性]] / [[テスタビリティとテスト戦略]] / [[関数型プログラミングパターン]] とも密接。Geek の落とし穴総覧は [[地雷・落とし穴 総覧]]。

---

## 1. 定義・原則

### 1.1 可読性 (Readability) と保守性 (Maintainability)

| 用語 | 定義 |
|---|---|
| **可読性** | コードを読んで「何をしているか」「なぜそうしているか」を、前提知識なしにどれだけ早く正確に把握できるか |
| **保守性** | コードをどれだけ容易に **理解・修正・適応・拡張** できるか(可読性はその最大の構成要素) |
| **認知負荷 (Cognitive Load)** | 読み手が同時に頭に保持しなければならない文脈の量。これが人間の作業記憶の限界(Miller の 7±2)を超えると理解が破綻する |

実証研究では、**コード可読性がソフトウェア保守効率の分散の約 58% を説明**する(デバッグ効率とリファクタ成功率を介して)。貧弱な語彙(命名)はプログラム理解を阻害し、保守工数を直接押し上げる。

### 1.2 認知負荷を下げる = 可読性の本質

- 「分かりやすい名前は認知負荷を下げ、開発者が **目の前の問題そのもの** に集中できるようにする」
- 巨大なロジックを小さく分解する → 一度に頭に入れる量が減る → 可読性と保守性が上がる
- **Cognitive Complexity(SonarSource, 2017)** は「テスト容易性 ≠ 理解容易性」という洞察から生まれた、**理解しやすさ**専用のメトリクス:
  - **0 から開始**(Cyclomatic は 1 から)
  - **線形フローの中断**(`if`/`for`/`catch`/`&&` 連鎖等)で +1
  - **ネスト**は累進的にペナルティ(深い `if` の中の `if` ほど重い)→ 「複数の文脈を頭に抱える」コストを反映
  - `switch` の 5 ケースは Cyclomatic では +5 だが Cognitive では **+1**(人間は一括で読める)
  - 推奨しきい値: **関数あたり 15** を超えたら分割を検討
- 対して **Cyclomatic Complexity** は実行パス数 = **テスト容易性**の指標としては優秀だが、保守性の代理指標としては弱い(巨大なドメインクラスと小さく濃密なクラスが同スコアになりうる)。

> Geek 適用: ESLint(`.eslintrc.js`)で `no-explicit-any: error` 等は強制しているが、Cognitive Complexity ルールは未導入。`lib/feed/smartRank.ts` や `lib/personalize/score.ts` のようなロジック密度の高いファイルは、関数を「1 つの責務 = 1 関数」に割り、ネストを early-return で潰すと読みやすくなる。

### 1.3 普遍原則(2023–2025 の合意)

1. **意図を表す命名 (Intent-revealing naming)** — 名前そのものが説明になる
2. **小さい関数・小さいモジュール** — 1 関数 1 責務、巨大関数を分解
3. **モジュール性** — 独立してテスト・修正できる単位に分割
4. **一貫性 (Consistency)** — コードベース全体で同じ書き方。既存コードに「馴染ませる」
5. **静的解析の自動化** — ESLint / Prettier / 型チェッカで「人間がレビューで指摘する前に」機械が弾く
6. **コメントは "why" を補う** — "what" は名前とコードで語らせ、コメントは設計判断・理由・落とし穴に使う

## 2. 具体例(コードブロック)

### 2.1 意図を表す命名

```ts
// ❌ 何の数値か読み手が逆算しないと分からない
const d = 86400;
// ✅ 名前が単位と意味を語る
const SECONDS_IN_A_DAY = 86400;
```

Google eng-practices の指針:識別子は「**それが何であるか/何をするかを完全に伝えるのに十分な長さ**」かつ「**読みにくくなるほど長すぎない**」のバランス。`x` でも `theObjectThatRepresentsTheCurrentlyLoggedInUserSession` でもなく `currentSession`。

### 2.2 複雑な条件式を名前付き変数に抽出

```ts
// ❌ 条件の "意味" が読めない
if (user.age > 18 && user.hasSubscription && !user.isBanned) { /* ... */ }

// ✅ 条件に名前を付けると「何を判定しているか」が一目で分かる
const canAccessPremiumContent =
  user.isAdult && user.hasActivePlan && !user.isAccountFlagged;
if (canAccessPremiumContent) { /* ... */ }
```

### 2.3 型で語らせる(マジック文字列を消す)

```ts
// ❌ 何を渡せるか呼び出し側が知らない
function setScope(scope: string) { /* ... */ }

// ✅ 取りうる値が型から自明。typo は型エラーで弾かれる
type FeedScope = 'all' | 'forMe' | 'community' | 'open';
function setScope(scope: FeedScope) { /* ... */ }
```

詳細は [[TypeScript 型安全性]]。Geek は `tsconfig` strict + `noUncheckedIndexedAccess` で、型を「読み手への最良のドキュメント」として機能させている。

### 2.4 ネストを潰して認知負荷を下げる(early return)

```ts
// ❌ 深いネスト = 各行で複数の真偽文脈を頭に保持
function getName(user) {
  if (user) {
    if (user.profile) {
      if (user.profile.name) {
        return user.profile.name;
      }
    }
  }
  return 'guest';
}

// ✅ ガード節で「例外ケースを先に弾く」と本筋が平坦になる
function getName(user) {
  if (!user?.profile?.name) return 'guest';
  return user.profile.name;
}
```

## 3. よくあるアンチパターン

| アンチパターン | なぜ悪いか | 対処 |
|---|---|---|
| **"what" コメント** (`// ループを回す`) | コードと重複。**コメントは嘘をつく**(リファクタで本体だけ変わり腐る) | 削除し、必要なら名前で語らせる |
| **マジックナンバー/文字列** | 意味も単位も不明。複数箇所に散ると変更漏れ | 名前付き定数 / enum / union 型 |
| **省略名 (`d`, `tmp`, `mgr`, `data2`)** | 逆算コストが毎回かかる | 意図を表す名前 |
| **巨大関数 / God オブジェクト** | 一度に保持する文脈が限界超え。テスト不能 | 1 責務に分割([[SOLID 原則と React 実践]]) |
| **深いネスト** | Cognitive Complexity 累進ペナルティ。読み手が真偽の山を抱える | early return / 条件抽出 |
| **silent な `try{}catch{}`** | 本当に壊れたケースが闇に消える(デバッグ不能) | 後述の Geek `swallow()` |
| **巨大 PR** | レビュアの認知が破綻 → レビューが形骸化 | 小さく分割(Google: 1 CL 1 目的) |
| **過剰コメント** | ノイズでコードが読めなくなる | 冗長コメントを消し、価値あるコメントだけ残す |
| **早すぎる最適化 / 過剰設計** | 「将来必要かも」で複雑性を先払い。読み手が惑う | Google: 「**今解くべき問題を解け**」 |
| **コメントの腐敗 (rotting)** | 本体だけ更新しコメント据え置き → 誤った情報源 | 「Code never lies, comments often do」前提でコメント最小化 + why に限定 |

## 4. コメント論 — "what" でなく "why"(自己文書化とのバランス)

「**自己文書化コード vs コメント**」は二者択一ではなく **バランス**:

- **第一原則**: 名前・型・構造で語れることはコメントにしない(コメントは腐る・嘘をつく)
- **コメントが真に価値を持つのは "why"**:
  - **回避策の理由**: `// IE11 で .map() polyfill が落ちるため旧式 for を使う`
  - **業務ロジックの文脈**: `// 外部 API のハンドシェイク確定のため同期を 2 回叩く`
  - **危険警告**: `// 呼び出し順を変えるな。DB デッドロックを起こす`
  - **設計判断 (なぜこの形にしたか)**

ドキュメントの優先順位(上から先に使う):
1. クリーンでモジュラなコード(大きい関数を割る)
2. 型定義(データの形を語らせる)
3. doc-string(JSDoc — 公開 API / IDE ツールチップ用)
4. why-コメント(クリーンコードでも残る疑問にだけ)

> 目標は「コメントを 0 にする」ではなく「**冗長なコメントを 0 にする**」。1 行 1 行が、コード自体が語る以上の価値を持つこと。

## 5. コードレビュー文化(Google eng-practices)

レビューの**唯一の目的**:「**コードベース全体の健全性 (code health) が時間とともに改善し続けること**」。

- **承認基準**: 完璧ではなく「**確実に全体の code health を改善する状態**になったら承認」(完璧なコードは存在しない、より良いコードがあるだけ)。逆に「**確実に code health を悪化させる**変更は(緊急時を除き)承認するな」。
- **レビュー観点**: 設計 / 機能 / **複雑性** / テスト / **命名** / **コメント** / スタイル / 一貫性 / ドキュメント
- **命名の指針**: 「意図を完全に伝えるのに十分な長さ、かつ読みにくくない長さ」
- **コメントの指針**: 「コメントは **なぜそのコードが存在するか** を説明する時に有用で、**何をしているか** を説明すべきではない」
- **過剰設計に警戒**: 「今解く必要があると分かっている問題を解け、将来必要かもしれないと推測する問題ではなく」
- **テストは同 CL で**: 緊急時を除き、プロダクションコードと同じ変更にテストを入れる([[テスタビリティとテスト戦略]])
- **意思決定の優先順位**: ①技術的事実 > 意見 ②スタイルガイド > 個人の好み ③設計はエンジニアリング原則に基づく
- **Nit:** プレフィックス = 「直さなくてもよい些細な指摘」。教育的コメントは歓迎されるが任意であることを明示
- **レスポンス**: レビュー依頼への返答は **最大 1 営業日**。レビュアが厳しすぎると改善の意欲を削ぐので、進捗と健全性のトレードオフを取る

> 2023 以降は Google 社内で **ML 提案付きレビュー**(コメントに対し 1 クリックで修正案を適用)も導入され、開発者満足度 97% を実現。レビューは「人を責める場」ではなく「コードベースを育てる協働」。

## 6. ★Geek への適用

Geek は **「開発者 1 人 + Claude ペアプロ」**(`CLAUDE.md` §17)という特殊体制。レビュアも将来の改修者も**多くが AI**。だからこそ可読性 = **AI が文脈を素早く正しく掴めるか**が死活的。Geek には独自の優れた「可読性文化」が既に根付いている。

### 6.1 日本語コメント文化 — 設計判断は日本語で書く

`CLAUDE.md` §6 のコード規約に明記:

> 「日本語コメント可。むしろ **design decision は日本語で書く文化**。」
> 「user とのやりとりは日本語。コード comment も日本語が default。ただし **コード本体(識別子/関数名/型名)は英語**。」

これは前述の原則と完全に一致する:**識別子(=what)は英語で語らせ、設計判断(=why)は最も精緻に書ける日本語で残す**。`console.log`/Sentry は英語 prefix + 日本語説明(`[realtime] ...`)。

**実物 — `lib/cropMath.ts` 冒頭**(why を厚く書く模範):

```ts
// ============================================================
// lib/cropMath.ts — クロッパーの座標計算 (pure / テスト可能)
// ============================================================
// `app/image-cropper.tsx` の handleNext に inline していた crop 矩形計算を
// 純関数として切り出したもの。(...)
//
// なぜ切り出すか:
//   - この座標計算は過去に何度も事故っている (非正方形化 → 引き伸ばし等)。
//     component に埋まっていると Jest で検証できず回帰が捕まらない。
//   - icon (正方形/円) と 投稿写真 (矩形) で同じ数学を共有したい。
//
// 重要な性質 (cover-fit の前提): (...)
// ============================================================
```

- 「**なぜ切り出すか**」「**重要な性質**」が日本語で書かれ、コードを読んだだけでは分からない**設計の意図と前提条件**を補っている。これは Google 言う「コメントは why を語れ」の理想形。
- さらに **純関数 (pure) に切り出す → Jest でテスト可能**にして、可読性だけでなく [[テスタビリティとテスト戦略]] と [[関数型プログラミングパターン]] を同時に満たしている。

**実物 — `lib/swallow.ts`**(silent catch アンチパターンの撲滅):

```ts
// swallow — try { ... } catch {} の代替ヘルパー
// 旧コードは `try { something(); } catch {}` で error を完全に握りつぶしていた。
// 99% は意図的な defensive code (...) だが、**本当に何かが壊れている** ケースも紛れ込む。
```

- §3 で挙げた「silent な `try{}catch{}`」を、breadcrumb を残す `swallow('scope', e)` に置換する規約(`CLAUDE.md` §5.6/§14)。**アンチパターンを命名されたヘルパに昇格させて一掃**した好例。

**実物 — `lib/resilient.ts`** も `// 透明なリトライ:` で「何を再試行し何を即 fail させるか(401/403/RLS は再試行しない)」を日本語で明文化。`ResilientOptions` の各フィールドに行末日本語コメント(`name: string; // ログ/Sentry 用ラベル`)。

### 6.2 自己文書化 — 型・命名・中央集約

- **design トークンの中央管理**: `design/tokens.ts` に `C`(color)/`GRAD`/`SP`(spacing)/`R`(radius)/`SIZE` を集約。マジックナンバーを散らさず、名前で意味を語らせる([[i18n・テーマ・デザインシステム]])。
- **lib/api/* への集約**: 「component から `supabase.from(...)` を直接叩かない → `lib/api/*` に通す」(§14)。クエリの所在が予測可能 = 探す認知負荷が下がる。`lib/api/` は `posts.ts`/`comments.ts`/`reactions.ts` … と**ドメイン名 = ファイル名**で自己文書化。
- **型で取りうる状態を語る**: `Rotation90 = 0 | 90 | 180 | 270`(cropMath.ts)、trust tier、CW 種別など union 型で「ありうる値」を型から自明に。
- **grep しやすい固定キー**: `swallow('storage.set', e)` のように「短く grep しやすい固定 scope」を推奨(swallow.ts コメント)。**後で探せる = 保守性**。

### 6.3 認知負荷を下げる構造的工夫

- **Zustand は selector で subscribe**(§5.4):`const user = useAuthStore((s) => s.user)`。全 destructure 禁止。これは re-render 連鎖(=パフォーマンス)対策だが、「この component が何を購読しているか」が 1 行で読める **可読性向上**でもある。詳細 [[Zustand・React Query ベストプラクティス]] / [[State管理 (Zustand・React Query)]]。
- **エラー処理 3 種を用途で命名分離**(§5.6):`resilient`(retry+timeout)/ `withApiTimeout`(timeout のみ)/ `swallow`(握りつぶし+breadcrumb)。**名前を見れば副作用が分かる**。
- **1 channel / 1 table**(§5.3, [[Realtime]]):複雑な chain を禁じ、`attachChannel` 1 経路に統一 → 読み手が realtime の挙動を 1 箇所で理解できる。

### 6.4 レビュー文化 = 「地雷の文書化」+ AI 並列監査

Geek 流の code health 維持装置:

1. **`CLAUDE.md` §11「過去に踏んだ地雷」表** — 症状 / 真因 / 対策の 3 列。同じ罠を 2 度踏まない**組織的記憶**。これはレビュー指摘を恒久ドキュメント化したもの = Google の「code health を時間とともに改善」の Geek 実装。
2. **`CLAUDE.md` §5.10「★事故多発」`thumbedUrl` の resize=cover 罠** — 「🔴 鉄則」「🚫 EXIF を疑う前にこれを疑え」と、**読み手の typical な誤解を先回りして潰す why コメント**。実測値(`width=480&resize=cover → 480x866 潰れ`)まで載せ、「推測でなく実測する」検証コマンドも同梱。これは「コメントは why を語れ」の極致。
3. **6–8 agent 並列 audit**(§15, `HYPOTHESIS_LOG.md`)— AI を並列レビュアとして使う。緑ビルド ≠ 安全という学び([[運用 — デプロイ・プレビュー・本番反映確認]] 参照)。
4. **重い設計判断のあるファイルは枠線コメント** `// ====...` でヘッダを切る(§6, 例 `realtime.ts` / `feedPagePatcher.ts` / `cropMath.ts` / `swallow.ts`)。**「ここには読むべき設計説明がある」という視覚的シグナル**。
5. **PR は提案・merge は user 判断**(§17.6)、**1 PR 1 目的**(§17.8)— Google の「小さい CL」原則に対応。

### 6.5 機械で強制(レビュー前に弾く)

- `.eslintrc.js`:`@typescript-eslint/no-explicit-any: 'error'`(`as any` 禁止 → `unknown` + 型ガード)、`react-hooks` ルール有効。
- `.prettierrc`:singleQuote / trailingComma=all / printWidth=100 / tabWidth=2 — **スタイル論争を機械が終わらせる**(Google「スタイルガイド > 個人の好み」)。
- CI(`.github/workflows/ci.yml`):毎 PR で `type-check + test`。commit 前に `npm run type-check && npm test && npm run lint`(`--max-warnings 0`)。

### 6.6 Geek の伸びしろ(可読性・保守性の観点)

- **Cognitive Complexity の ESLint ルール未導入** — `lib/feed/smartRank.ts` / `lib/personalize/` / `lib/search/`(BM25・variants)等のロジック密度が高い領域に閾値(例 15)を入れると、肥大化を早期検知できる。
- **日本語コメントの一貫性** — 一部 lib は手厚いが、`hooks/`(40+)や `components/` の why コメント密度にばらつき。設計判断のあるカスタム hook には cropMath.ts 流のヘッダを足すと、AI 改修時の取り違えが減る。
- **`as any` ゼロは型安全だが、`@ts-expect-error` の濫用に注意** — 黙らせるより根本の型を直す([[TypeScript 型安全性]])。

## 7. チェックリスト(Geek 開発で使える即用版)

- [ ] 名前だけで「何をする/何である」が分かるか(`d`/`tmp`/`mgr` を排除)
- [ ] マジックナンバー/文字列を名前付き定数・union 型にしたか
- [ ] 複雑な条件を名前付き変数に抽出したか
- [ ] ネストを early-return で平坦化したか(Cognitive Complexity を意識)
- [ ] コメントは "why"(設計判断・回避策・危険警告)を**日本語で**書いたか。"what" の冗長コメントを消したか
- [ ] 設計判断の重いファイルは `// ====` ヘッダ + 「なぜ◯◯か:」を書いたか
- [ ] silent `try{}catch{}` を `swallow('scope', e)` にしたか
- [ ] Zustand は selector 購読か(全 destructure 禁止)
- [ ] supabase 直叩きでなく `lib/api/*` 経由か
- [ ] 純ロジックは切り出してテストを足したか
- [ ] PR は 1 目的・小さく分割したか / §11 地雷表を確認したか
- [ ] `type-check` / `test` / `lint --max-warnings 0` が緑か
