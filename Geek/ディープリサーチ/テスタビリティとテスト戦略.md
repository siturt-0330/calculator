# テスタビリティとテスト戦略

> 「どれだけテストを書くか」ではなく「**どこに投資すれば一番 confidence が得られるか**」を決めるノート。
> 起点となる一次情報は Kent C. Dodds の Testing Trophy、Testing Library の guiding principle、Martin Fowler の Test Doubles。
> 関連: [[テスタビリティとテスト戦略]] は [[素晴らしいコードとは — 総論]] / [[関数型プログラミングパターン]] / [[SOLID 原則と React 実践]] / [[可読性・保守性]] と密接に絡む(テスト容易性は設計品質の*結果*であって、後付けの作業ではない)。

---

## 定義・原則

### テスタビリティ (testability) とは何か

テスタビリティ = 「そのコードが、**少ない労力で・速く・決定的に (deterministic) 検証できる度合い**」。テストが書きにくいコードは、ほぼ例外なく**設計が悪い**サイン。具体的には:

- **副作用が分離されていない** — I/O・時刻・乱数・グローバル状態がロジックに混ざっている → モックだらけになる。
- **依存が暗黙** — モジュールが内部で `import` した依存を直接叩く → 差し替え (seam) がない。
- **関数が大きすぎる / 分岐が多すぎる** — 1 関数で 10 個の責務 → テストケースが組合せ爆発。
- **出力が観測できない** — 戻り値ではなく内部状態を変えるだけ → state verification できない。

逆に、**純粋関数 (pure function)** = 「同じ入力なら同じ出力・副作用なし」はテスタビリティの理想形。入力を渡して戻り値を assert するだけで済む(モック不要、setup 不要、並列実行可能、フレーキーにならない)。詳細は [[関数型プログラミングパターン]] と [[TypeScript 型安全性]]。

### 中核原則: "The more your tests resemble the way your software is used, the more confidence they can give you."

Testing Library 全体を貫く一文。これは「**実装詳細 (implementation details) をテストするな**」と同義:

- ユーザーは「state が変わったか」「ある関数が呼ばれたか」を気にしない。気にするのは「正しい UI が出るか」「データが反映されるか」。
- だから内部 state / private メソッド / 特定の prop 呼び出しではなく、**ユーザーが観測できる振る舞い**(画面の文言・role・遷移)を assert する。
- 効能: **リファクタしてもテストが壊れない**(挙動が同じなら緑のまま)。実装に密結合したテストは、リネーム 1 つで赤くなり「テストのためのテスト」を生む。

### Testing Trophy(テスティング・トロフィー)

Kent C. Dodds が Testing Pyramid を更新した投資配分モデル。Guillermo Rauch のツイート **"Write tests. Not too many. Mostly integration."** が出発点。下から上へ:

| 層 | 何を捕まえるか | confidence | コスト/速度 | Geek での担い手 |
|---|---|---|---|---|
| **Static**(土台) | typo・型不一致・undefined アクセス。実行不要で最速 | 低〜中 | ほぼ 0(書きながら) | **TypeScript strict + `noUncheckedIndexedAccess` + ESLint `no-explicit-any:error`** |
| **Unit** | 依存なし or モック済みの単一ユニット(純関数・reducer) | 中 | 速い・安い | **Jest `tests/unit/` 704 it ブロック**(中心) |
| **Integration**(最も厚く) | 複数ユニットが**一緒に**正しく動くか | 高 | 中 | (現状の伸びしろ。RNTL で hook+store+RQ を結合) |
| **E2E**(頂点) | ユーザー操作のフロー全体・最小モック | 最高 | 遅い・高い・壊れやすい | **Maestro(`npm run test:e2e`、tests/e2e/)** |

**なぜ "mostly integration"** か:

> "When you mock something you're removing all confidence in the integration between what you're testing and what's being mocked." — Kent C. Dodds

ユニットテストは速いが、個々が緑でも**統合すると壊れる**ことを保証しない。E2E は最高の confidence だが遅く・脆い。Integration は「confidence と speed/cost のトレードオフのスイートスポット」。

**Static を最下層に置くのが Trophy の肝**: 型システムは「実行する前に・コストほぼ 0 で」1 クラスのバグ(null 参照・型ミス)を消す。Geek の strict TS + `noUncheckedIndexedAccess`(`array[0]` が `T | undefined`)はまさにこの層で、CI の `npm run type-check` が PR ごとに走る。

### 70% カバレッジの逓減

Dodds はおおむね 70% 前後で ROI が逓減すると指摘。それ以上は「ロジックの無い trivial code」を測りに行きがちで、カバレッジ数字のためのテストになる。**カバレッジは目標ではなく診断**。

## 具体例(コードブロック)

### 純粋関数の理想テスト(モック・setup ゼロ)

Geek 実例 `tests/unit/hotScore.test.ts`。Reddit 風 hot score を計算する純関数を、入力→戻り値だけで検証:

```ts
import { computeHotScore, HOT_TIME_DIVISOR } from '../../lib/utils/hotScore';

// 仕様: score = log10(max(|s|,1)) + sign(s) * (epoch - LAUNCH) / 28800
it('s = 10, at launch+1day → log10(10) + 86400/28800 = 1 + 3 = 4', () => {
  const score = computeHotScore({
    likesCount: 10, concernCount: 0, createdAt: '2024-05-17T00:00:00.000Z',
  });
  expect(score).toBeCloseTo(4, 6);
});

// 防御的: 不正入力でも壊れず 0 を返す(契約のテスト)
it('invalid date string → 0', () => {
  expect(computeHotScore({ likesCount: 10, concernCount: 0, createdAt: 'not-a-date' })).toBe(0);
});
```

ポイント: SQL マイグレーション (`0058_hot_score.sql`) とクライアント実装が**同じ式**であることをテストが固定する = "仕様の二重実装の同期" を保証。

### プロパティベース寄りの網羅(境界バグの回帰防止)

`lib/utils/voteFuzz.test.ts`。決定性・レンジ・境界を**多数の入力でループ**して assert(乱数を使わず deterministic):

```ts
it('toggling self-like changes display by exactly +1 across all tiers', () => {
  const otherCounts = [0, 1, 2, 3, 4, 9, 10, 11, 12, 50, 100, 999]; // tier 境界を網羅
  for (let i = 0; i < 300; i++) {
    const id = `viewer-${i}`;
    for (const others of otherCounts) {
      const notLiked = getDisplayLikesForViewer(id, others, false);
      const liked = getDisplayLikesForViewer(id, others + 1, true);
      expect(liked - notLiked).toBe(1); // 自票は必ず ±1 反映
    }
  }
});
```

これは [[地雷・落とし穴 総覧]] の「いいね押しても数字が変わらない」バグ(fuzz tier 境界 2↔3 / 10↔11 で自票が飲み込まれた)の**回帰テスト**。テストが「旧挙動にバグが存在したこと」自体も assert している(`expect(boundaryBugExisted).toBe(true)`)= テストの健全性チェック。

### 境界をモックする(I/O 直前で seam を切る)

`tests/unit/spamDetection.test.ts`。Supabase RPC だけ `jest.mock` で差し替え、**ロジックは本物**を通す:

```ts
const mockRpc = jest.fn();
jest.mock('../../lib/supabase', () => ({
  supabase: { rpc: (...a: unknown[]) => mockRpc(...a) },
}));

it('空文字列はクライアントチェックで即 isSpam、RPC 呼び出しなし', async () => {
  const result = await runAllChecks('');
  expect(result.isSpam).toBe(true);
  expect(mockRpc).not.toHaveBeenCalled(); // 早期 return を behavior verification
});

it('checkDuplicateContent で例外 → fail-open (isSpam:false)', async () => {
  mockRpc.mockResolvedValueOnce({ data: null, error: null });
  mockRpc.mockRejectedValueOnce(new Error('network timeout'));
  expect((await runAllChecks('正常な投稿')).isSpam).toBe(false); // 障害時の安全側挙動を固定
});
```

ここで使っている検証スタイル(Martin Fowler の分類):
- `expect(result.isSpam).toBe(true)` → **state verification**(戻り値の状態を見る)
- `expect(mockRpc).not.toHaveBeenCalled()` → **behavior verification**(協力オブジェクトへの呼び出しを見る。mock/spy だけが持つ)

### Cache helper の integration 寄りテスト(本物の QueryClient)

`tests/unit/feedPagePatcher.test.ts` は **本物の `@tanstack/react-query` の `QueryClient`** を使い、モックしない:

```ts
const qc = new QueryClient();
qc.setQueryData([FEED_PAGE_KEY, 'user-1', 'p1,p2,p3'], [mkPost('p1'), mkPost('p2'), mkPost('p3')]);
patchFeedPagePost(qc, 'p2', { my_like: true, likes_count: 5 });
expect(qc.getQueryData<FeedPagePost[]>([FEED_PAGE_KEY,'user-1','p1,p2,p3'])![1]!.my_like).toBe(true);

// React.memo が反応するよう reference が変わることまで保証
it('patch 後は配列 reference が変わる', () => { /* expect(after).not.toBe(initial) */ });
```

これは「unit と integration の中間」: 外部ライブラリの実挙動(structural sharing・reference 同一性)に依存する部分を**本物で**検証している。`mkPost()` は object mother(テストデータ生成ヘルパ)で、必要なフィールドだけ override させて DRY に保つ。

### E2E(Maestro): YAML で「ユーザーが操作する通り」

```yaml
# tests/e2e/like-post.yaml(将来像)
appId: app.geek.v4
---
- launchApp
- tapOn: "ログイン"
- tapOn:
    id: "feed-post-0-like"
- assertVisible:
    id: "feed-post-0-like-active"
```

Maestro はプロジェクトに依存を入れず CLI が端末経由で操作。flow を「ログイン」「投稿作成」など**再利用可能な小片**に分割できる。

## よくあるアンチパターン

| アンチパターン | なぜ悪い | 正しくは |
|---|---|---|
| **実装詳細をテスト**(internal state / private method / `instance()`) | リネーム/リファクタで赤くなる。confidence を生まない | ユーザー観測可能な振る舞い(role / text / 遷移)を assert |
| **モックし過ぎ** | "what's being mocked との統合の confidence をすべて捨てる"。本物の DB スキーマ変更を検知できない | I/O 境界(Supabase/network/時刻/乱数)**だけ**モック。ロジックは本物 |
| **`getByTestId` を第一選択** | アクセシビリティと乖離。ユーザーは testid を見ない | `getByRole` → `getByLabelText` → … の順。testid は最後の逃げ道 |
| **`fireEvent` でクリック** | 単一イベントしか飛ばさず実挙動と乖離 | `userEvent`(`pressIn/press/pressOut` を順に発火、async) |
| **スナップショット乱用** | 巨大 snapshot は誰も読まず `-u` で機械的に更新 → ガード機能ゼロ | 小さく狙った assert。snapshot は安定した小出力のみ |
| **カバレッジ 100% を目標化** | trivial code を測りに行く / テストのためのテスト | 70% 目安、ロジックの濃い所に集中 |
| **フレーキーなテスト**(時刻 `Date.now()` / 乱数 / 実ネットワーク直叩き) | 落ちたり通ったり → CI の信頼が崩壊し全員が無視し始める | 時刻・乱数を引数注入(Geek: `createdAt` を渡す、fuzz は `post_id` ハッシュで決定的) |
| **テストピラミッドを E2E 倒立**(E2E ばかり) | 遅い・脆い・原因特定が困難 | Trophy 配分。E2E は critical path だけ |
| **AAA が崩れた読めないテスト** | 何を検証しているか不明 | Arrange→Act→Assert。`it('...が...のとき...')` で意図を文章化 |
| **共有可変 fixture** | テスト間で状態が漏れて順序依存 | `beforeEach` で再構築 / object mother で都度生成(Geek `mkPost`) |

## ★Geek への適用

### 現状の地形(実測)

```
geek-v4/
  tests/unit/        … 42 ファイル(純関数 unit test の中心)
  lib/utils/*.test.ts… 3 ファイル(co-located: voteFuzz / commentDisplay / lastViewed)
  tests/e2e/         … Maestro 用(scripts は配線済 "maestro test tests/e2e"、flow は今後追加)
  → it/test ブロック合計 704
```

- **Static 層**: `tsconfig` strict + `noUncheckedIndexedAccess`、`.eslintrc.js` の `no-explicit-any:error` + react-hooks rules。CI(`.github/workflows/ci.yml`)が **PR ごとに `type-check` → `npm test -- --ci --runInBand`** を実行、master push のみ web build smoke。Trophy の土台がきちんと効いている。
- **Unit 層が圧倒的に厚い**: feedRanking / hotScore / rising / trending / searchRanking / tagClustering / commentTree / trustTier / automodMatcher / voteFuzz / spamDetection / quotePosts / sharePost … いずれも**純関数 or 純ロジック**。これは設計上 `lib/` にロジックを純関数として切り出している([[アーキテクチャ概要]]・[[関数型プログラミングパターン]])結果で、テスタビリティが高い。
- **意図的にフレームワーク・レス**: リポジトリに `jest.config.*` も package.json の `jest` フィールドも**無い** = Jest デフォルト設定(Node 環境)で動く。`jest-expo` と `@testing-library/react-native` は devDependency に**入っているが、まだどの commit のテストからも使われていない**。つまり現行スイートは 100% 純ロジックで、**コンポーネント/フックの render テストは伸びしろ**(下記)。

### Trophy を Geek に当てはめた処方箋

1. **Static(済・維持)**: strict TS を緩めない。`as any` を入れない([[TypeScript 型安全性]])。型で消せるバグはテストを書く前に消す。
2. **Unit(継続)**: CLAUDE.md §10 のルール「**新しい純関数 / 純 reducer を書いたら test も追加**」を死守。ロジックは `lib/` の純関数に寄せてから書くと、モックゼロで書ける(spamDetection の同期チェック部、hotScore、voteFuzz が手本)。
3. **Integration(最優先で増やすべき層)**: `jest-expo` preset + `@testing-library/react-native`(both 既にインストール済)を実際に配線し、**hook + Zustand store + React Query** を結合してテストする。狙いどころ:
   - `useLike` / `useConcern` の **optimistic update → 失敗時 revert**(`feedPagePatcher` を本物の QueryClient で通す。snapshot→apply→error→revert の契約を [[State管理 (Zustand・React Query)]] の規約通りに検証)。
   - i18n: `protectBrandNames`→API→`restoreBrandNames` の往復(既に `i18nBrand.test.ts` が「Geek を翻訳しない」を固定)。
   - **モック方針**: モックは `lib/supabase` / network / 時刻 の**境界だけ**(spamDetection.test の `jest.mock('../../lib/supabase')` が手本)。store・RQ・純ロジックは**本物**を通して "統合の confidence" を取りに行く。
4. **E2E(critical path に絞る)**: Maestro flow を「**ログイン → フィード表示 → いいね → 投稿作成**」など壊れたら致命的な導線だけに。flow を小片(login / create-post)に分けて使い回す。脆さとコストを考え数を絞る = Trophy の頂点は小さい。

### テスタビリティを上げる設計判断(Geek 固有)

- **時刻・乱数を引数化**: `computeHotScore({createdAt})` は `Date.now()` を内部で呼ばず引数で受ける → deterministic。`voteFuzz` は乱数ではなく `fnv1a32(post_id)` ハッシュで決定的 → 同じ post は常に同じ表示。**これがフレーキー撲滅の核心**。
- **副作用を境界に押し出す**: コンポーネントから `supabase.from()` を直接叩かず `lib/api/*` に集約(CLAUDE.md §14)→ テスト時に差し替える seam が 1 箇所に集まる。[[SOLID 原則と React 実践]] の依存性逆転。
- **回帰テストを「バグの墓標」にする**: 踏んだ地雷([[地雷・落とし穴 総覧]])は必ず unit test 化する文化。例 = `introSplashLock.test.ts`(splash↔intro の固定値一致を assert、寸法/色/タイミングを変えると落ちる)、`voteFuzz.test.ts`(自票飲み込み)。テスト名に「何のバグの再発防止か」を日本語で書く。
- **`--passWithNoTests`** が `npm test` に付いている = テストの無いパッケージでも CI を止めない実用判断。ただし純関数を足したらテストも足す前提。

### モック方針の一行まとめ(Geek)

> **境界(Supabase / network / 時刻 / 乱数)だけモック。ロジック・store・React Query は本物を通す。** モックを足すたびに "そことの統合の confidence" を 1 つ捨てている、と意識する。
