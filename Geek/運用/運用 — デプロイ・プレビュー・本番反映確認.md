# 運用 — デプロイ・プレビュー・本番反映確認

GEEK (geek-v4) を本番に出す／検証する一連のフロー。配信先は **Web (Netlify)** / **iOS・Android (EAS)** / **Supabase (DB・Edge Functions)** の 3 系統。
このノートは「いつ・どうやって反映するか」「反映されたか確認する方法」「踏みやすい地雷」を実コードと運用ルールから具体的にまとめる。仕様の全体像は [[機能一覧・仕様サマリー]]、データ層の適用手順は [[データ層・Supabase・RLS・マイグレーション運用]] を参照。

---

## 概要

- **本番 Web**: <https://geekboard.netlify.app/> — Netlify が **master / main 更新で自動ビルド**。Netlify project = `geekboard`、リポジトリ = `siturt-0330/calculator`、**base directory = `geek-v4`**。
- **Supabase project ref**: `migpiwdlpwpvehzvdjyh`（`netlify.toml` / `web-postbuild.mjs` にハードコード）。
- **🛑 最重要ルール**: Netlify への反映 = **master/main への merge・push・`gh pr merge`** は **ユーザーが明示指示したときだけ**。ローカル commit / branch push / PR 作成は別扱い（詳細は下記「注意点・地雷」）。
- **green build ≠ 反映して安全**: type-check 0 / lint 0 / 精読だけでは「安全に出せる」と断言しない。挙動回帰は `/code-review` (audit) で別途確認する。
- **本番反映の確認**: production(master) deploy は **GitHub commit status を返さない**。確認は live bundle の content-hash 突合 + ASCII マーカー grep で行う（ポーリングは無駄）。
- **dev preview の地雷**: full reload で dev セッションが落ちて `/login` に飛ぶ → **soft-nav でセッション保持**。HMR は効く時と効かない時がある。

> 運用ガイド本体は `geek-v4/CLAUDE.md` §0（standing policy）/ §9（デプロイ）、Runbook は `geek-v4/docs/DEPLOY.md`（7 章構成）。

## 仕組み・設計

### 1. Web (Netlify) のビルドパイプライン

設定ファイルは **`geek-v4/netlify.toml`**（★ repo root の `netlify.toml` は使われない。Netlify は base=geek-v4 で **このファイル**を読む）。

```toml
[build]
  publish = "dist"
  command = "npm ci && npm run build:web"
```

- **build コマンドの実体**（`package.json`）:
  ```
  "build:web": "expo export --platform web --output-dir dist && node scripts/web-postbuild.mjs"
  ```
  → `expo export` で `dist/` に Metro バンドルを出力 → `web-postbuild.mjs` で `dist/index.html` を後処理。
- `[build.environment]` に **`EXPO_PUBLIC_*` を直書き**している（`EXPO_PUBLIC_SUPABASE_URL` / `ANON_KEY` / `VAPID_PUBLIC_KEY` / `POSTHOG_HOST` / `HOME_FEED_RPC=1`）。これは Netlify dashboard 設定の drift / 削除で「build は通るが env が無くて画面真っ白」事故（2026-05-24）を防ぐための **意図的な repo 固定**。public-safe な値のみ（anon key は RLS 前提で晒される設計）。
- `NODE_VERSION=20` / `NODE_OPTIONS=--max-old-space-size=4096`（大規模 Expo Web build はメモリ headroom が要る）/ `CI=1`。
- `.npmrc` の `legacy-peer-deps=true` が必須（react-native-worklets 等の peer dep 厳格化を緩める。無いと ERESOLVE で落ちる）。

`netlify.toml` のその他の設計:
- **SPA fallback**: `from="/*" to="/index.html" status=200`（Expo Router の client-side routing にすべての path を渡す。deep link / リロードで 404 にしない）。
- **キャッシュ戦略**: `/_expo/static/*`・`/assets/*`・`/static/*` は `immutable, max-age=31536000`（ハッシュ付きなので 1 年固定）。`/`・`/index.html` は `max-age=0, must-revalidate`（古い shell をユーザーに残さない）。
- **セキュリティヘッダ**: RN Web は inline style 多用のため CSP は外し、`X-Frame-Options=SAMEORIGIN` / `X-Content-Type-Options=nosniff` / HSTS / `Permissions-Policy` で対応。

### 2. `web-postbuild.mjs` が `dist/index.html` に注入するもの

`expo export` が吐く素の SPA shell は `<div id="root"></div>` が空＝JS 到着まで真っ白。これを後処理で強化する:

1. body 背景を `#0a0a0a`（白フラッシュ消し）
2. ブランド付き **instant splash**（Geek ワードマーク + 進捗バー）を素 HTML/CSS で描画（JS 到着前 ~100-300ms から「読み込み中」が見える）
3. React が `#root` に mount したら splash を fade-out して除去（MutationObserver + 12s safety timer）
4. Supabase への `preconnect` / `dns-prefetch`（最初の feed RTT の DNS/TLS を前倒し）

idempotent（marker `geek-splash-style` が既にあれば skip、marker 不在なら warn して no-op で build を壊さない）。
この splash は起動演出（イントロ）と寸法・色・タイミングを完全一致させる【確定版・変更禁止】(CLAUDE.md §0)。詳細は [[i18n・テーマ・デザインシステム]] / [[パフォーマンス最適化]]。

> ⚠️ **`web-postbuild.mjs` に Service Worker を足さない**（古い shell 残存事故源）。詳細は地雷セクション。

### 3. iOS / Android (EAS) — `eas.json`

profile は 3 つ:

| profile | distribution / channel | 用途 |
|---|---|---|
| `development` | internal / `development`、`developmentClient:true`、android=apk | dev client ビルド。`env` に SUPABASE_URL/ANON_KEY 直書き |
| `preview` | internal / `preview`、android=apk | 内部配布（QA） |
| `production` | `production`、`autoIncrement:true` | ストア提出用。build number 自動採番 |

```bash
# ストア提出
eas build  --platform ios --profile production
eas submit --platform ios --latest
# Android: eas submit --platform android --latest --track internal → Closed→Open→Production

# OTA (JS のみの hotfix・ストア審査不要)
eas update --branch production --message "fix: <概要>"
```

- **OTA で出せるのは JS-only な修正だけ**。Native module を触ったら必ず通常 `eas build` & `eas submit`。
- `eas.json` の `submit.production` は現状 `{}`（空）。Apple/Google の実値（appleId / ascAppId / appleTeamId / serviceAccountKeyPath）は提出時に埋める（`docs/DEPLOY.md` §3-4 にテンプレあり、実値は EAS Secret 推奨）。
- env 配置: iOS/Android は `eas.json` の `build.env` か `eas secret:create`。

### 4. Supabase (DB / Edge Functions) のデプロイ

```bash
supabase db diff --linked        # ローカル↔リモート diff
supabase db push --linked        # migration 適用 (適用後 supabase/migrations/ に commit)
supabase functions deploy <name> # Edge Function (check-content は --no-verify-jwt)
supabase secrets set KEY=...     # 秘密鍵 (VAPID_PRIVATE_KEY 等)
```

- migration は **既存 file 編集禁止**（idempotency 崩壊）。revert は新 file `00XX_revert_*.sql` を追加して再 push。
- ⚠️ **migration は Supabase が自動適用しない**。Netlify は Web bundle を配るだけで DB スキーマには触らない。RPC やテーブルを使う新機能は **手動で `db push` or SQL エディタ適用が必須**。未適用だと client が PGRST202 で fallback して「壊れはしないが遅い／silent degrade」になる（例: `HOME_FEED_RPC=1` は 0114 適用が前提）。手順は [[データ層・Supabase・RLS・マイグレーション運用]]。

### 5. 環境変数の配置場所（まとめ）

| 配信先 | どこに置く |
|---|---|
| ローカル | `geek-v4/.env`（gitignore 済） |
| Web (Netlify) | **`geek-v4/netlify.toml` の `[build.environment]`**（repo root のは無視） |
| iOS/Android | `eas.json` の `build.env` or `eas secret:create` |
| Supabase Edge | `supabase secrets set KEY=...` |

- `EXPO_PUBLIC_*` は **バンドル同梱＝ブラウザ DevTools で読まれる前提**。RLS 前提で安全な anon key / DSN / VAPID 公開鍵のみ。
- `SUPABASE_SERVICE_ROLE_KEY` / `VAPID_PRIVATE_KEY` / `ANTHROPIC_API_KEY` は **絶対にクライアントへ置かない** → Supabase secrets / Edge のみ。`EXPO_PUBLIC_*` に秘密鍵を置いたら即時 rotate。
- `process.env.EXPO_PUBLIC_*` は **static 評価**。`process.env[key]` のような動的アクセスは inline されない。`.env` 変更後は Expo dev server 再起動が必要。

### 6. 本番反映の確認方法（content-hash 突合）★ 重要

**production(master) deploy は GitHub の commit status を返さない。** deploy-preview は PR commit に status を出すが master deploy は出さないので、`gh api repos/<o>/<r>/commits/<sha>/status` は永遠に `{"state":"pending","checks":[]}`（= status 0 件＝GitHub 既定の pending）。これは「ビルド中」ではなく「status 無し」。**ポーリングしても無駄。**

確認はサイト実体で行う:

1. `curl -s https://geekboard.netlify.app/` → HTTP 200 + HTML から `/_expo/static/js/web/entry-<hash>.js` を抽出。
2. その `entry-<hash>.js` の **content-hash が `deploy-preview-<PR>--geekboard.netlify.app` の bundle hash と一致**すれば、本番 = その PR ビルドとバイト一致で確定（Expo/Metro の bundle hash は内容由来）。
3. 追加で **ASCII の「新規限定」マーカー**を grep（例: 新機能由来の `blur(24px) saturate(180%)` のような旧 prod に無い文字列）。minified は 1 行なので `grep -c` は不可 → `grep -oF ... | wc -l`。
   - ⚠️ **日本語/非ASCII マーカーは使うな** — prod minify が `\uXXXX` にエスケープするので raw UTF-8 grep が必ず 0 になり「旧デプロイのまま」と誤検知する。

### 7. ローカル PR preview のための switcher

`scripts/preview-pr.ps1` — 特定 PR の branch にローカル切替して preview 検証する safe switcher（**絶対に git push しない / Netlify を trigger しない** と明記）。working tree clean check 付き、`git switch` のみ。切替後は Metro 再起動:
- 通常: `npx expo start --web --port 8081`（`.metro-cache/` の FileStore キャッシュ再利用 = 2nd start 30〜60s で速い）
- キャッシュ汚染疑い時のみ: `--clear`（cold start ~185s で遅い）。毎回付けない。

## 注意点・地雷

### 🛑 Netlify 反映は「ユーザー明示指示時のみ」(standing rule)

「**netlify には俺が指示したときだけ反映させて**」「**勝手に PR/merge はしないで**」がユーザーの明確な要請（CLAUDE.md §0 にも明記）。

- **明示指示が要る操作**: master/main への merge（PR merge 含む）/ master/main への push / `gh pr merge` / `gh pr create`（PR 作成だけでも）。
- **デフォルトの停止点 = ローカル commit まで**（push すら指示があれば）。迷ったら branch push までで止めて確認を取る。
- **バグ報告 ≠ deploy 指示**。「〜が表示されない」等の報告は修正依頼であって PR/merge/deploy 依頼ではない。コード修正と PR/merge は **別の指示**として扱う。
- **deploy トリガー明示語**: 「反映」「push」「deploy」「netlify」「PR 作って」「PR 出して」「merge して」「リリースして」。これが user message に出たときに限り commit→push→PR→merge を通しで実行してよい。
- ローカルの fix / type-check / test / preview verify / `git status` / `git diff` は自由に進めて OK。

### ⚠️ green build を根拠に「安全に出せる」と言い切らない

type-check 0 / lint 0 + コード精読は **構文・型の網であって挙動回帰の網ではない**。
実例: PR #154 で green build + 精読で「安全に出せる」と判断したが、ユーザーの監査コミットが **2 HIGH + 8 回帰**（search.tsx / CommunityIcon.tsx / mypage.tsx / VideoPlayer.tsx / posts.ts / checkContent.ts）を捕まえた。
→ 「反映してほしい」系の変更は **merge 前に `/code-review` (audit) を一度通す** か通っているか確認する。報告は「型/lint は通った。挙動回帰は別途要確認」と分けて言う。詳細は [[テスタビリティとテスト戦略]] / [[地雷・落とし穴 総覧]]。

### ⚠️ dev preview: full reload で dev セッションが落ちる → soft-nav せよ

Claude Preview（Metro Expo web @ 8081/8082）で authed 画面を検証するときの環境固有挙動:

- **full reload（`location.assign` / `location.reload`）は dev のログインセッションを落とす。** トークン（`geek-v4-auth`）は localStorage に残るが、ヘッドレスの preview ブラウザでは Supabase セッションが復元されず、`_layout.tsx` の auth ガードがコールドブートのレースで `/login` へ飛ばす。一度 authed ビューを失うと資格情報なしでは戻れない（ユーザーの実ブラウザは影響なし＝preview 文脈だけ）。
- **session を保ったまま画面遷移する soft-nav 手順**（full reload 回避）:
  1. `history.pushState({}, '', '/feed'); dispatchEvent(new PopStateEvent('popstate'))` で URL を変える
  2. 底タブ DOM 要素に合成 `pointerdown/up + click` を撃つ（RN Web Pressable は click に反応）
  これで `/login` に飛ばされず feed 等へ遷移して検証でき、検証後は同手順で元画面に戻せる。
- 詳細・FlashList 重複 key 警告での eval 詰まり（screenshot で代替）は [[地雷・落とし穴 総覧]]。

### ⚠️ HMR（Fast Refresh）は効く時と効かない時がある

この preview は **編集をホットリロードしない時がある**（HMR 実質オフ）。Metro は fetch 要求時に最新ファイルから再ビルドするが、起動済みセッションは編集前バンドルを実行し続ける（bundle URL は `hot=false`）。
→ ただし「常時オフ」ではない（2026-06-05 に edit の live 反映を観測）。**まず HMR が効いているか試す**。効かない場合、ソース修正の live 検証は「型チェック + Metro 配信バンドル本文に編集が入っているか fetch 確認」が現実的な上限。実挙動確認は dev server 再起動 + 再ログイン or 本番ビルドが要る。

### 🔴 `fix-html.mjs` の Service Worker は本番未配線（絶対に繋ぐな）

- `build:web` が実際に呼ぶのは **`web-postbuild.mjs`**（SW を入れない）。`scripts/fix-html.mjs` は **cache-first の `service-worker.js` を書き出して register する別スクリプトだが、build パイプラインに繋がっていない（dead）**。
- 同様に `scripts/netlify-build.sh`（debug 出力 + `npm ci --legacy-peer-deps && npx expo export`）も **未配線** — `netlify.toml` は `npm ci && npm run build:web` を直接呼ぶので使われない。
- ⚠️ **`fix-html.mjs` を build に繋いだり、SW を `web-postbuild.mjs` に足したりしない**。cache-first SW は「古い shell が残る」事故源（Web 起動時に古い画面が出る既知の真因系）。`web-postbuild.mjs` 冒頭コメントにも「Service Worker は入れない」と明記。本番の freshness は `netlify.toml` の `/index.html: max-age=0, must-revalidate` で担保する。

### ⚠️ 設定ファイルの二重化に注意

- **`netlify.toml` は repo root と `geek-v4/` の 2 つあり、使われるのは `geek-v4/netlify.toml` だけ**（base=geek-v4 のため）。env を repo root に書いても「白画面」になる。
- `fix-html.mjs` は `dist/_redirects` と `dist/netlify.toml`（SPA fallback）も書くが、これも未配線。実際の SPA fallback / cache / header は `geek-v4/netlify.toml` が担う。
- `fix-html.mjs` 内の `SITE_URL = 'https://geek-app.netlify.app'` は OG/canonical 用の古いドメイン。実 live は `geekboard.netlify.app`（このスクリプト自体が未配線なので実害なしだが、繋ぐと OG が誤ドメインになる）。

### ⚠️ Rollback / インシデント

- Web: Netlify Dashboard → Deploys → 過去 deploy の "Publish deploy"（SPA fallback で URL は壊れない）。
- Native: JS のみ → EAS Update で hotfix / Native も触る → 緊急 EAS Build + Expedited Review / 即停止 → TestFlight・Play で旧版に戻す。
- DB 起因: revert SQL を新 file で作り即 `db push`。
- post-mortem は `docs/HYPOTHESIS_LOG.md` に記録。
