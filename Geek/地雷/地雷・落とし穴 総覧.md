# 地雷・落とし穴 総覧

Geek (geek-v4) で実際に踏んだ / 現存する地雷を「症状 → 真因 → 対策 → 関連」で横断する総覧ノート。
出典は `geek-v4/CLAUDE.md` §11(再発防止 cheat sheet)・§5.10(画像表示)・§14(設計 NG リスト)と、
セッションをまたいで蓄積した memory 17 ファイルの「真因 → 対策」。各地雷は深掘りノートへリンクするので、
ここは「索引 + 一目で分かる表」として使う。

> 鉄則: **同じ罠を 2 度踏まない**。新機能着手前にこのノート(と CLAUDE.md §11)を必ず一読する。
> 「green build(type-check 0 / lint 0)= 安全に反映」ではない。挙動回帰は別の網(後述)で捕る。

---

## 概要 — 地雷の「型」5 分類

Geek の地雷はだいたい以下の型に収斂する。新しい不具合を見たら、まずどの型かを当てると真因に早く着く。

| 型 | 何が起きるか | 代表例 |
|---|---|---|
| **A. RLS だけでは塞げない** | 行は守れても「列」「相関」「派生ビュー」から情報が漏れる/改竄できる | 匿名 de-anon(author_id 漏洩)/ posts 列 UPDATE 改竄 |
| **B. キャッシュ層と表示層のズレ** | データは正しいのに画面が古い/変わらない | RQ persist に buster 無 / vote fuzz 自票飲み込み / FlashList `extraData` |
| **C. 環境固有(web / native / preview)** | 「自分の環境だけ」で起き、本番ユーザーや別 OS では出ない(逆も) | preview full reload で dev ログアウト / web blob: revoke / native SecureStore 非アトミック |
| **D. 外部 endpoint の暗黙仕様** | こちらの想定と endpoint の挙動が食い違う | Supabase render `resize=cover` の高さ据え置き / WebP transform が EXIF 無視 |
| **E. デプロイ・適用の silent degrade** | 適用し忘れ/未配線でもフォールバックで「動いて見える」ため気付けない | migration 手動適用漏れ / fix-html.mjs 未配線 / 0129 列 revoke が no-op |

## 仕組み・設計 — 地雷別ディープダイブ(具体ファイルパス付き)

### D系: 画像表示 — `resize=cover` の高さ据え置き + EXIF/WebP(★事故多発, CLAUDE.md §5.10)

**この領域は Geek で最も事故が多い。** 「横長写真が縦長/細く表示される・画像が水平に潰れる」が繰り返し再発した。
真因は **2 つあり、別物・別時期** に確定した。混同しないこと。

1. **真因その1 = `thumbedUrl` の `resize=cover` 高さ据え置き(commit `6a6c064`)**
   `lib/utils/imageUrl.ts` の `thumbedUrl()` が Supabase render endpoint
   (`/storage/v1/render/image/public/`)に **`width` だけ + `resize=cover`** を渡すと、
   endpoint は **幅だけ要求値に縮小し高さは元のまま** 返す。
   - 実測(元 1179x866 横長): `width=480&resize=cover` → **480x866(aspect 0.55 潰れ)** / `width=240` → **240x866(0.28 激細)**。
   - 影響は二重: ① `RNImage.getSize()` の測定値が 0.28 に化け「縦長/細いセル」で出る、② 表示 bitmap 自体も水平に潰れる。
   - 修正: `resize` 既定を **height 未指定なら `contain`** に。
     ```ts
     // lib/utils/imageUrl.ts:63
     const resize = opts.resize ?? (opts.height ? 'cover' : 'contain');
     ```
     `contain` は width だけでも比率維持(480x866 → 480x353)。crop/fit は **expo-image の `contentFit`** に任せる(Supabase 側で cover crop しない)。
   - 正方形 crop が要る avatar/icon は `squareThumbedUrl`(=`width=height`+cover)/ `iconThumbedUrl`(=`width=height`+contain)を使う。
     **生 `thumbedUrl(url, w, {resize:'cover'})` を height 無しで書かない。**

2. **真因その2 = EXIF orientation + WebP transform(commit `b138c51`, 2026-06-10)**
   resize 修正後も横長が縦長表示される個体が残り、**今度は本当に EXIF が絡んでいた**。
   - `lib/image.ts` の `stripExifAndResize` が `manipulateAsync(uri, [], ...)` と **空 actions** を渡すと、
     iOS が EXIF rotation タグを保持したまま JPEG を返す。その JPEG を Supabase transform でサムネ化すると
     transform が EXIF を無視し raw ピクセル向きで縮小 → サムネが縦横逆 → aspect 測定が逆になる。
   - 修正: action に **`[{ rotate: 0 }]`** を入れると `manipulateAsync` が EXIF 向きを **物理的に適用してタグをリセット**する
     (保存後 = display 向きピクセル + EXIF=1)。
     ```ts
     // lib/image.ts:54
     const first = await ImageManipulator.manipulateAsync(uri, [{ rotate: 0 }], { ... });
     ```
   - 加えて `AnonPostCard` / `FeedMediaGrid` の **測定サムネを `format=webp` → `format=origin`** に変更。
     **WebP 変換は Supabase が EXIF rotation を適用しないケースがあり**縦横逆寸法を返していた。`measureAspect` の error callback でオリジナル URL 再試行も追加。
   - aspect クランプも緩和: `Math.max(0.5, Math.min(2.0, …))` → `Math.max(0.3, Math.min(3.0, …))`(超横長パノラマの歪み解消)。

**★重要な解釈の更新**: CLAUDE.md §5.10 / memory には「**EXIF を疑う前に resize=cover を疑え**」とある。
これは「resize=cover が判明した時点(2026-06)では EXIF は無罪だった」という意味で **正しい**。
だが **その後 `b138c51` で resize 以外に本物の EXIF/WebP 起因が見つかった**。
→ 現在の正しい順序は **「① まず `resize=cover` を `image-size` で実測切り分け → ② それでも残るなら EXIF(空 actions)+ WebP transform を疑う」**。
EXIF を最初から思考停止で疑うのは依然 NG だが、「EXIF は常に無関係」も誤り。

**検証は推測でなく実測**(CLAUDE.md §5.10 のスニペット): 本番の実画像 URL を `image-size` で object(生)と render(変換)で比較。
preview なら `img.naturalWidth/Height` と `getBoundingClientRect()` を eval し `cellAspect ≈ natAspect` を確認。

### B系: いいねが反映されない — vote fuzz の自票飲み込み(修正済 `24b0ada`)

**症状**: いいねを押すとハートは点くのに **数字が変わらない**(home / community 両方)。
**意外な真因**: キャッシュ層は完全に正常(`window.__qc` で全 cache が ±1 更新済を確認)。
犯人は `lib/utils/voteFuzz.ts` の **vote fuzz**(表示用の決定的ノイズ。`≤2`=なし / `≤10`=±1 / `>10`=±5)。
**自分のいいねで実数が段階境界(2↔3・10↔11)を跨ぐとノイズ幅自体が変わり、自分の +1 を打ち消す**(実数 10→11 なのに fuzz が +1→0 で表示 11→11)。

**修正(3点)**:
1. `getDisplayLikesForViewer(postId, realLikes, viewerLiked)` 新設 — fuzz は「**自分を除いた票数** `others = realLikes-(viewerLiked?1:0)`」に対して計算し、自分の +1 は素で上乗せ。toggle しても他人分の fuzz は不変 = 自分の操作は必ず ±1 反映。`voteFuzz.test.ts` に全 tier の self=±1 回帰テスト。
2. `useLike` が **community feed cache を patch していなかった** → `onMutate` に `['community', id, 'feed', sort]` の ±1 を追加(非配列メタは `!Array.isArray` で skip、`onError` revert)。
3. home feed が stale な base post から count を読んでいた → `feed.tsx` の enrichedPost で **`full`(feed-page cache・patch 済)の count を優先**。理由: FlashList の `data=feedItems` は **意図的に不変**(`extraData={fullPosts}` で再 render させる設計)なので base post の更新が表示に伝播しない。**count は必ず full 経由で読む**。

**再利用できる debug 技法**: preview_eval で fiber を辿って QueryClient に到達し `window.__qc` でキャッシュ実値を覗く(DOM 表示と cache のズレ診断の必須技。スニペットは [[State管理 (Zustand・React Query)]] 参照)。DOM の like 数点読みは crossfade アニメ + `staleTime:0` churn で不安定 → (a) unit test + (b) cache 直読みの二段で確実に。

### B系: Web 起動時に古い画面 — RQ persist に buster 無 + refetchOnMount false(修正済)

**症状**: アプリを開くと昔の ver / 古いデータが出て、**1 回リロードすると直る**。deploy 非依存で「しょっちゅう」起きるのが特徴。
**真因**: `app/_layout.tsx` の `PersistQueryClientProvider` の persistOptions に **buster が無く**、QueryClient 既定が **`refetchOnMount: false`** + staleTime 30s。→ 永続化された dehydrated cache(feed/posts/notifications)が maxAge 2h 内で cold open ごとに復元され、stale でも mount 時に再 fetch されず固定。
**対策(適用済)**:
```ts
// app/_layout.tsx
refetchOnMount: true,                 // L175: stale だけ裏で SWR 再取得(fresh は触らず連打にならない)
const PERSIST_BUSTER = 'geek-rqcache-v1'; // L205: cache shape を変えたら必ず bump
persistOptions={{ persister, maxAge: 1000*60*60*2, buster: PERSIST_BUSTER }} // L664
```
runtime コードなので効くのは **次回 Web deploy 後**。既存ユーザーは buster で 1 度だけ clean slate + 以降 refetchOnMount で常時最新化。

### A系: 匿名 de-anon ホール — author_id 漏洩(★既存・一部現存, exploit 可能)

**ホール**: フィード RPC は `is_anonymous=true` の投稿でも **実 `author_id`(auth user UUID)を payload に返す**。一方 `profiles_public` ビュー(0081, `security_invoker=off` で RLS bypass)が全 id の `nickname/avatar` を anon+authenticated に素通し公開。→ client が `author_id` を集めて `select nickname from profiles_public where id in (...)` するだけで「匿名で書いた人」を一斉に実名特定でき、同一 author_id 相関で名寄せも可能。**匿名 SNS のコア約束に直結する重大ホール。**

**7 漏洩経路(2026-06-04 完全監査)**: ① `get_feed_page`(0107) ② `get_home_feed`(0114) ③ `get_community_feed`(0112)= 3 RPC が author_id 非マスク(0113 `get_discovery_payload` のみマスク済=リファレンス) / ④ **本丸** `lib/api/posts.ts` の `POSTS_SELECT_COLS` が author_id を直 select(RPC マスクを完全 bypass) / ⑤ `comments.ts` `bbs.ts` も同様 / ⑥ realtime `useFeed.ts` が posts UPDATE payload(全列)を spread + posts は full-row publication / ⑦ posts.author_id に列 revoke 無し + `v_search_signals`(0087) が第2オラクル。

**段階修正計画(★ Phase1 までローカル commit・未 deploy、Phase2 未着手)**:
- **Phase1**: feed 3 RPC を 0113 と逐語一致の CASE マスクに統一(`is_own` boolean も server 供給)。**mod は「削除のみ」に決定**(マスクで他人 author_id が null → kick/ban が壊れる + 「匿名投稿の author を ban=de-anon」なので)。⚠️ Phase1 単独では閉じない(posts 直 select / realtime / v_search_signals が残る)。
- **Phase2(破壊的・最後)**: 擬似名は **案B = profiles に `pseudonym_id`(random・per-user 安定)列追加**、RPC が `author_token=pseudonym_id` を返し client は author_id を二度と受け取らない。`pseudonymFor()` の入力を author_id→author_token に。★**pseudonym_id を profiles_public に絶対追加しない**(de-anon 鍵を与えない)。順序厳守 = client を author_id-free に deploy → **後で `REVOKE SELECT(author_id)` を適用**(逆だと旧 client が PostgREST 400 で全滅)。author_id 消費は監査で **124 箇所**(GDPR export / admin views / realtime / qaSort の is_op 判定 等)。

### A系: posts 列 UPDATE 改竄ホール(★ 0134/0138 未適用)

**ホール**: RLS `posts_update` は行(`auth.uid()=author_id`)しか制限せず **列を制限しない** → 認証ユーザが自分の投稿に `update posts set likes_count=99999, score=999, visibility='public' ...` を直 REST で撃てる(いいね数/ランキング/公開範囲の捏造)。
**修正 = `0134_post_column_update_hardening.sql`(★未適用・SQL エディタ手動適用待ち)**:
- counter トリガ関数(`update_likes_count` 等)を `ALTER FUNCTION ... SECURITY DEFINER` で **DEFINER 化**(本体は触らない)。**`CREATE OR REPLACE` を使わない理由** = 0001 と complete_schema.sql で本体が食い違うため、書き直すと live を巻き戻すリスク。ALTER は属性のみで最も安全。
- そのうえで `revoke update on posts from authenticated,anon` → `grant update(content,title,tag_names,...11列) to authenticated`(updatePost + togglePostQAMode が触る列のみ)。PostgreSQL 仕様で **「table 権限を先に revoke してから列 GRANT」の順序が必須**。
- **DEFINER 化が必須な理由**: counter 列を revoke すると INVOKER のままの counter トリガが like 押下時に 42501 で壊れる。DEFINER(owner=postgres)なら RLS bypass + 全列権限で通る。
- ★**適用前に必ず like→count 反映を実機確認**(DEFINER 化で counter が壊れないこと)。

**派生: 0129 は no-op 確定 → 0138 で是正(★未適用・ブロッカーあり)**: `0129_revoke_author_id` の `revoke select(author_id)` は **no-op**。理由 = posts/comments は Supabase 既定の `grant all ... to anon,authenticated` で **table 全体 SELECT** を持ち、PostgreSQL では実効列権限 = table権限 OR 列権限 なので列 revoke は table 権限を覆せない。修正 `0138` は「table 全体 SELECT revoke → author_id 以外の全列を列 GRANT」。**★ブロッカー**: `lib/api/account.ts` の `exportUserData`(GDPR export)が `select('*').eq('author_id',uid)` で直読みしており、REVOKE 後は catch で空配列 = 自分の投稿/コメントが **エクスポートから黙って欠落**(要 auth.uid() ベース RPC 化)。

関連: [[匿名性設計と de-anon ホール]] / [[データ層・Supabase・RLS・マイグレーション運用]]

### A系: realtime — 1 channel に複数 table を chain(過去 hot bug 頻発)

**症状**: リアルタイムのスタンプ/反応が届かない。
**真因**: 1 channel に複数 table を `.on()` chain すると、**publication 未登録 table の binding が CHANNEL_ERROR を起こした瞬間に channel 全体が死ぬ**(event 一切配信されない)。
**対策**: 必ず `lib/realtime.ts` の `attachChannel(name, build, onStatus?)` 経由(channel 名で refCount 共有)。**1 channel / 1 table**。subscribe ライフサイクル(SUBSCRIBED/CHANNEL_ERROR/TIMED_OUT/CLOSED)を `onStatus` で観測すると debug が桁違いに楽(本番 console.warn は babel strip 除外設定済)。上限 `MAX_CONCURRENT_CHANNELS = 20`。
- ✅ 登録済 publication: `post_reactions`, `likes`, `bbs_replies`, `comments`, `notifications`
- ❌ 未登録(subscribe するだけで CHANNEL_ERROR): `concerns`, `saves`, `community_stamp_reactions` 等

### A系: 認証セッション — 急に login/onboarding に戻る(修正済 PR#139)

**症状**: 投稿しようとした時などに急に login/onboarding へ。**真因は複数経路**で、access token(~1h)切れ時に refreshSession を試さず即 signOut していた。
- **F1(最重要)**: Supabase auth-js が getUser/refresh/復帰失敗時に **内部で自律的に SIGNED_OUT を発火**し、その前に storage を消す。authStore の SIGNED_OUT 分岐が無条件 `set({user:null})` していた。**対策**: TOKEN_REFRESHED/SIGNED_IN 時に refresh_token をメモリ保持し、SIGNED_OUT 時に **`setSession({access_token, refresh_token})` で復活**を試す。★引数なし `refreshSession` は storage 削除済みで効かない → token を明示的に渡すのが要点(二重復活ガード必須)。
- **F3**: profile fetch timeout で `onboarded=undefined` → `_layout` が undefined を falsy 扱いし /onboarding へ誤バウンス。**対策**: `user.onboarded === false` の厳密比較。
- **F4**: redirect ガードが `ready`(forceReady 500ms 含む)基準だと hydrate 完了前に誤バウンス。**対策**: redirect 判定を `authHydrated` 基準に。
- **F6**: Native SecureStore の setItem が delete-before-write で **非アトミック** → rotation 中断で session 永久消失。**対策**: 新 chunk 書込 → meta(コミット点)→ 旧掃除 の非破壊順 + `keychainAccessible: AFTER_FIRST_UNLOCK`。

### C系: web 画像 upload の blob: revoke(連続再発した地雷)

**症状**: web で画像を選んでも反映されない(無音でアップロード失敗)。
**真因**: web では `ImagePicker.launchImageLibraryAsync` の `asset.uri` が `blob:` URL。**iPhone Safari がこの blob: を勝手に revoke** するため、生のまま `prepareImageUpload`(`lib/image.ts`)に渡すと web fetch が空 Blob / "Load failed" になる。
**対策パターン**: ピック時に **web のみ `makeWebPreviewDataUrl(asset.uri, maxEdge, 0.85)` で `data:` URL 化**してから渡す(`data:` は revoke 耐性ショートパス `uri.startsWith('data:')` を通る)。`maxEdge` は uploader の `maxWidth/maxHeight` と一致(album/cover/post いずれも **1600**)。変換失敗時は生 URI フォールバック。native は `asset.uri` のまま。
**適用済**: `app/post/create.tsx` / `app/post/comment.tsx` / `app/settings/profile-edit.tsx`(pickCover)/ `app/mypage/photo/add.tsx`(PR#157)。
**Why 連続再発**: 同じ blob: revoke がカバー画像 → アルバム写真と連続で再発。既存 upload 経路を踏襲せず新規に書くと必ず踏む。

### E系: デプロイ・適用の silent degrade(「動いて見える」罠)

| 罠 | 何が危険か | 対策 |
|---|---|---|
| **migration 手動適用漏れ** | ユーザーは migration を **Supabase SQL エディタで手動適用**(Netlify は migration を流さない)。0118-0123 / 0134 / 0138 / 0139-0141 等は **未適用でも client が fallback で動く** = silent degrade。「動いてる=適用済」ではない | 適用必須の migration を明示追跡。Admin Console / レコメンド / de-anon は適用 + pg_cron 登録 + フラグ設定まで揃って初めて有効 |
| **nested dollar-quote が SQL エディタで誤分割** | `do $$ ... create function ... $fn$ ... $fn$; ... end $$;` を SQL エディタの statement splitter が誤分割 → 関数 body が plain SQL 化し `ERROR 42601 syntax error at or near "uuid"`(`declare v_viewer uuid;` で発生) | **SECURITY DEFINER 関数を含む migration は do ブロックで包まず top-level の `create or replace function`** で書く(dollar-quote を `$fn$` 1レベルに)。CLI `supabase db push` なら nested でも通るが、手動適用前提なら top-level 必須 |
| **`scripts/fix-html.mjs` が build 未配線** | OG/PWA/lang=ja + **cache-first Service Worker** を生成するが build に一切配線されていない(本番 index.html は素の Expo 出力 = SW 無し、curl 確認済)。**繋ぐと cache-first SW が入り stale の温床を新規に作る** | **fix-html.mjs / web-postbuild.mjs に SW を足さない**(「SW を直す」系の修正案は無意味かつ有害)。push 用 `public/push-sw.js` は別物(fetch handler 無し) |
| **本番 deploy は commit status を返さない** | master deploy は GitHub commit status を出さない → `gh api .../status` は永遠に `pending`(=status 0 件)。ポーリング無駄 | live 実体で確認: ① `curl` で `entry-<hash>.js` 抽出 → deploy-preview の bundle hash と **content-hash 一致**で確定、② **ASCII の新規限定マーカー** grep。⚠️ **日本語/非 ASCII マーカーは使うな**(prod minify が `\uXXXX` エスケープ → raw UTF-8 grep が必ず 0 = 誤検知) |
| **green build ≠ 反映して安全** | type-check 0 / lint 0 + 精読で「safe to ship」と断言したが、監査(code-review)が **2 HIGH + 8 回帰**を捕まえた(PR#154)。lint/型は構文・型の網で **挙動回帰の網ではない** | 「反映して」系は merge 前に **`/code-review`(audit)を一度通す**。報告は「型/lint は通った。挙動回帰は別途要確認」と分ける。**deploy は [[運用 — デプロイ・プレビュー・本番反映確認]] のルール = ユーザー明示指示時のみ** |

## 注意点・地雷(このノートを使うときのメタ注意)

- **情報の鮮度**: memory は point-in-time。file:line 引用は古びる(本ノートは 2026-06-10 時点の実コードで主要点を再確認済 = `imageUrl.ts` の `resize` 既定 / `_layout.tsx` の buster・refetchOnMount / `b138c51` の EXIF 修正)。断定前に現コードを確認する。
- **★ = 未解決/未適用**: 0134 / 0138(posts 列改竄)、de-anon Phase2、0118-0123 / 0139-0141(手動適用待ち)は **未適用でも fallback で動く**ため見落としやすい。
- **「resize=cover vs EXIF」の両論併記**を必ず守る: 「EXIF を疑う前に resize=cover」(時系列の前半)と「resize 修正後に本物の EXIF/WebP が見つかった」(`b138c51`)は **どちらも真**。片方だけ覚えると次の再発で迷う。
- **deploy 可否**: 修正・型チェック・preview verify は自由だが、**git push / PR 作成 / merge はユーザー明示指示時のみ**(「反映」「push」「deploy」「merge して」等の語があるときだけ)。バグ報告 ≠ deploy 指示。
- **検証の二段構え**: DOM 点読み(crossfade / staleTime churn で不安定)に頼らず、**unit test(決定的)+ cache/実測 直読み**で確証を取る。
